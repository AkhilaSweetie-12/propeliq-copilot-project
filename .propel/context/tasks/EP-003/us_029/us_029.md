# User Story - US_029

## Story ID

- ID Format: US_029

## Story Title

- General Waitlist Queue with FIFO Position & Slot-Open Notification

## Description

- As a patient for whom no available appointment slots exist this week, I want to join a FIFO waitlist for any slot and receive a notification when I reach the front of the queue and a slot opens, so that I am guaranteed the next available slot in request-time order without needing to check the portal repeatedly.

## Acceptance Criteria

1. **Given** no available slots exist for the queried date range and the patient is on the SCR-008 empty state, **When** the patient clicks the "Join waitlist" call-to-action, **Then** the SPA navigates to a minimal waitlist-join form (can be a modal or inline panel on SCR-008 / SCR-009); the patient may optionally specify a preferred time window (morning / afternoon / any); on submission the SPA calls `POST /api/waitlist` with `{ patient_id, preferred_time_window? }`; the API inserts a `waitlist_entries` row with `status = Waiting`, `requested_at = NOW()` (UTC), `queue_position` computed as `SELECT COUNT(*) WHERE status=Waiting` + 1 at insert time; the API returns HTTP 201 with `{ waitlist_position }`.

2. **Given** a patient submits the waitlist join request, **When** the API commits the insert, **Then** the patient's current `queue_position` is returned in the HTTP 201 response body; the SPA displays a confirmation message: "You are #N in the waitlist. We'll notify you when a slot opens."; this queue position is informational only and is not guaranteed to reflect real-time position (no live polling required); `INSERT audit_logs (action = JoinedWaitlist, patient_id, requested_at)` is appended within the same transaction.

3. **Given** a `appointment_slots.is_available` transitions to `true` for any slot (the same `SlotReleasedEvent` used in us_028), **When** the Hangfire `WaitlistNotificationJob` evaluates whether a general-waitlist patient should be notified, **Then** the job executes `SELECT waitlist_entries WHERE slot_id IS NULL AND status = Waiting ORDER BY requested_at ASC LIMIT 1 FOR UPDATE` to identify the FIFO-first general waitlist patient (rows with no `slot_id` are general-queue entries; rows with a specific `slot_id` are preferred-slot entries handled by us_028); the job sends an email notification to the patient with the newly available slot's date, time, and provider; `UPDATE waitlist_entries SET status = Notified, notified_at = NOW()` is applied in the same transaction; the slot is **not** pre-reserved — the patient must book it independently within the notification email's CTA link before another patient claims it.

4. **Given** the notified patient does not book the slot within 2 hours, **When** the notification TTL expires, **Then** the `waitlist_entries` row status transitions from `Notified` to `Expired` via a scheduled Hangfire cleanup job (`WaitlistExpiryJob`, runs every 15 minutes); the slot remains `is_available = true` and the next-in-queue patient is notified by re-running the `WaitlistNotificationJob`; `INSERT audit_logs (action = WaitlistEntryExpired)` is written.

5. **Given** a patient who is on the general waitlist later books an available slot through the standard booking flow, **When** the booking confirms (us_027 AC-4), **Then** the API checks for any `waitlist_entries` rows with `status = Waiting` or `status = Notified` for this patient and updates them to `status = Cancelled`; this prevents the patient receiving out-of-date waitlist notifications after they already have an appointment; `INSERT audit_logs (action = WaitlistCancelledOnBooking)` is written.

## Edge Cases

- What happens if a patient tries to join the waitlist twice for the same date range? → The API enforces a uniqueness guard: `SELECT waitlist_entries WHERE patient_id=X AND status IN ('Waiting','Notified')` — if an active entry exists, the API returns HTTP 409 with message "You are already on the waitlist"; the SPA surfaces this as an inline message instead of allowing a duplicate insert.
- What happens if the notification email fails to deliver to the first-in-queue patient? → The `WaitlistNotificationJob` retries email delivery up to 3 times with exponential backoff; if all retries fail, the `waitlist_entries` status transitions to `NotificationFailed`; the slot remains `is_available = true`; the next scheduled run of the job skips `NotificationFailed` entries and notifies the next eligible patient in FIFO order; failures are logged to `audit_logs`.
- What happens when multiple slots become available simultaneously (e.g., a provider cancels multiple appointments)? → Each released slot fires a separate `SlotReleasedEvent`; each event triggers a separate `WaitlistNotificationJob` scoped to that slot; the `FOR UPDATE` row lock on `waitlist_entries` ensures that only one job claims the first-in-queue entry; each subsequent job finds the next `Waiting` entry in FIFO order, so multiple newly-open slots can notify multiple distinct patients.
