# User Story - US_027

## Story ID

- ID Format: US_027

## Story Title

- Booking Form, Insurance Soft Pre-Check & Preferred Slot Registration

## Description

- As an authenticated patient who has selected an available slot, I want to enter my insurance details, optionally designate one preferred unavailable slot, and confirm my booking so that the system records my appointment, executes a non-blocking insurance pre-check, and stores my preferred slot preference for automatic swap consideration.

## Acceptance Criteria

1. **Given** the patient arrives on the Booking & Insurance Form (SCR-009) with a `slot_id` in React Router state, **When** the page renders, **Then** the booking summary sidebar displays the selected appointment date, time, provider name, and appointment type sourced from the passed state; two required fields are displayed: "Insurance provider" (`#insurance-provider`) and "Member ID / Policy number" (`#insurance-id`); both fields are `aria-required="true"` and validated client-side on blur — blank values display inline validation error messages before form submission is attempted.

2. **Given** the patient has entered insurance provider and member ID, **When** the patient submits the form by clicking "Confirm appointment →" (`#btn-confirm`), **Then** the SPA calls `POST /api/appointments/book` with payload `{ slot_id, preferred_slot_id?, insurance_name, insurance_id }`; the API queries `insurance_records` table for a match on `(name, member_id)`; the match result (`Matched`, `Unmatched`, or `Not Found`) is stored against the appointment record and displayed inline on SCR-009 (`role="status"`, `aria-live="polite"`) with icon and colour: Matched = green ✓, Unmatched = amber ✗, Not Found = grey ?; the insurance result **never** blocks booking completion (FR-024, FR-025).

3. **Given** the patient is on SCR-009, **When** the patient clicks the "Set a preferred slot (optional)" section, **Then** a dropdown (`#preferred-slot`) lists all unavailable slot IDs retrieved from the slot grid; the patient can select at most one preferred slot; if a preferred slot is selected, it is included as `preferred_slot_id` in the booking payload; if the patient selects "No preference" (default), `preferred_slot_id` is omitted from the payload (FR-011).

4. **Given** the booking request `POST /api/appointments/book` is received by the API, **When** the transaction executes, **Then** the API runs `BEGIN TRANSACTION`; executes `SELECT appointment_slots WHERE slot_id=X FOR UPDATE`; if the slot is still available it inserts into `appointments` (`status=Confirmed`, `preferred_slot_id=Y?`) and updates `appointment_slots SET is_available=false`; inserts `INSERT audit_logs (action=AppointmentBooked)`; commits; returns HTTP 201 with `{ appointment_id }`; the SPA redirects to the Appointment Confirmation screen (SCR-010).

5. **Given** the slot is taken by another patient between calendar selection and the booking submission, **When** the API executes `SELECT FOR UPDATE` and finds `is_available=false`, **Then** the API rolls back, returns HTTP 409 Conflict; the SPA displays inline error "That slot was just taken — please choose another" and navigates back to SCR-008 with no partially saved state.

## Edge Cases

- What happens if the patient navigates directly to SCR-009 without a `slot_id` in React Router state (e.g., browser refresh or deep link)? → The SPA checks for `slot_id` on component mount; if absent, it immediately redirects to SCR-008 with no error message — the patient must start the slot selection flow normally; no API call is made.
- What happens if the preferred slot selected by the patient becomes available before the booking transaction completes? → The `preferred_slot_id` is simply stored against the appointment as a preference; the auto-swap Hangfire job (FR-012, us_028) will detect the slot state at execution time and act accordingly; no special handling is needed in the booking transaction for this race.
- What happens if the insurance_records table is empty or the service call times out? → The insurance pre-check is advisory only; if the query returns no rows or times out within 500 ms, the result is recorded as `Not Found`; the API does not delay or block the booking transaction on behalf of the insurance check — the transaction proceeds regardless.
- What happens if `preferred_slot_id` equals `slot_id` (patient accidentally selects the same slot being booked)? → The API validates that `preferred_slot_id != slot_id` and that the preferred slot's `is_available = false`; if either validation fails, the API returns HTTP 400 "Invalid preferred slot selection"; the SPA displays the error inline.
