# User Story - US_026

## Story ID

- ID Format: US_026

## Story Title

- Booking Calendar View & Available Slot Selection

## Description

- As an authenticated patient, I want to view a weekly calendar of appointment slots that clearly distinguishes available, unavailable, and waitlisted slots using colour and icon, so that I can pick one available slot per session to book.

## Acceptance Criteria

1. **Given** an authenticated patient navigates to `/book`, **When** the Booking Calendar screen (SCR-008) loads, **Then** the SPA calls `GET /api/slots?date_range=X` where `X` is the current week; the API returns all `appointment_slots` rows (both `is_available = true` and `false`); the SPA renders a weekly grid where each cell is one of three visual states: Available (green fill, ✓ icon, `aria-label="…— Available"`), Unavailable (grey fill, — icon), or Waitlist/Preferred (amber fill, ★ icon); a persistent legend panel maps colour and icon to state label — satisfying UXR-101's colour-blind-safe requirement (icon + colour, never colour alone).

2. **Given** the slot calendar is rendered, **When** the patient clicks an Available slot cell, **Then** the cell transitions to the Selected state (teal fill, `aria-selected="true"`); the booking sidebar panel updates in real time to display the selected date, time, provider name, and appointment duration; the "Continue →" button (`#btn-continue`) becomes active; only one slot can be selected at a time — clicking a second Available slot deselects the first and selects the new one.

3. **Given** an authenticated Staff user navigates to `/book` (e.g., via direct URL), **When** the booking API validates the JWT role claim, **Then** the API returns HTTP 403 Forbidden; the patient-facing booking calendar is not accessible to Staff or Admin roles — walk-in bookings are exclusively handled through the Staff walk-in flow (EP-004); this RBAC guard is enforced at the API route level independently of the SPA navigation.

4. **Given** no available slots exist for the queried date range, **When** the calendar grid renders, **Then** all cells are in the Unavailable state; an empty-state component is displayed with message "No slots available this week" and a "Join waitlist" call-to-action that links to the SCR-009 booking form in waitlist mode; the "Continue →" button remains disabled.

5. **Given** the patient selects an Available slot and clicks "Continue →", **When** the SPA navigates to the Booking & Insurance Form (SCR-009), **Then** the selected `slot_id` is passed in React Router state so that SCR-009 can display it in the appointment summary sidebar without a second API call; if the patient navigates back from SCR-009 using "← Change slot", the previously selected slot cell re-renders as Selected.

## Edge Cases

- What happens if a slot that was Available when the calendar loaded is taken by another patient before the current patient clicks it? → The SPA does not re-check availability on click; the race condition is resolved at booking time (`POST /api/appointments/book`) using `SELECT FOR UPDATE` — if the slot is taken, the API returns HTTP 409 and the SPA navigates back to SCR-008 with message "That slot was just taken — please choose another" (handled in us_027).
- What happens when the patient uses keyboard navigation to select a slot (tab + Enter)? → All slot cells have `tabindex="0"` for Available and Waitlist cells; keyboard Enter key fires the same selection handler as a click; `aria-selected` is updated; the booking sidebar panel updates as with a click — WCAG 2.2 AA keyboard operability is maintained (UXR-201).
- What happens when the API returns more than 200 slots (e.g., a very long date range)? → The `date_range` query parameter is validated server-side to a maximum span of 28 days; requests exceeding this return HTTP 400 "Date range cannot exceed 28 days"; the SPA defaults to a 7-day range and does not expose the raw parameter to the patient.
