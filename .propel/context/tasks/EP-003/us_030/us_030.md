# User Story - US_030

## Story ID

- ID Format: US_030

## Story Title

- PDF Appointment Confirmation via QuestPDF & Self-Check-In Block

## Description

- As a patient who has just confirmed an appointment, I want to receive a PDF confirmation by email containing my appointment details and cancellation policy, and I want to be prevented from self-checking in via any web or QR interface, so that my booking record is formally documented and arrival is always handled exclusively by Staff.

## Acceptance Criteria

1. **Given** the booking transaction commits with HTTP 201 and Hangfire enqueues `send_confirmation_job(appointment_id)`, **When** the Hangfire worker picks up the job, **Then** the worker calls the QuestPDF Community (MIT) fluent document API to generate a PDF document containing: appointment date, time, location / clinic name, provider full name and specialty, appointment type (e.g., "General Consultation"), patient full name, insurance provider, and a cancellation policy section; the generated PDF bytes are attached to an email sent to the patient's `email` address with subject "Your PropelIQ Health Appointment Confirmation" (TR-010, FR-016).

2. **Given** the PDF confirmation email is sent, **When** the patient views the Appointment Confirmation screen (SCR-010), **Then** the SPA navigates to SCR-010 immediately after receiving the HTTP 201 from `POST /api/appointments/book`; SCR-010 displays: appointment date and time, provider name and specialty, appointment type, insurance provider, a "Back to dashboard" link (`href` to SCR-004), and an "Add to calendar" button; a status message confirms "A PDF confirmation has been emailed to your address"; SCR-010 does not contain any "Check in" or "I have arrived" button or control.

3. **Given** the email delivery for the PDF confirmation fails on the first attempt, **When** the `send_confirmation_job` exception handler catches the failure, **Then** Hangfire retries the job up to 3 times with exponential backoff (15 s, 30 s, 60 s); if all retries fail, the job transitions to the `Failed` state; `INSERT audit_logs (action = ConfirmationEmailFailed, appointment_id, error_message, failed_at)` is written; the appointment record remains `status = Confirmed` — a failed email does NOT cancel or roll back the booking.

4. **Given** any user (Patient, Staff, or Admin) attempts to call `PATCH /api/appointments/{id}/checkin` or accesses any web route containing `/checkin`, **When** the API receives the request, **Then** the endpoint does not exist: the API returns HTTP 404 Not Found for any route matching `/checkin`; no QR-code-to-checkin handler exists in the routing table; patient-facing navigation in SCR-010 and any other patient portal screen contains no check-in button, link, or form element — arrival confirmation is exclusively handled by Staff via the Staff queue management screen (EP-004); this restriction is enforced at both API routing level and SPA navigation level (FR-017).

5. **Given** a Staff user views the staff-side appointment record for a patient who was notified of a preferred slot swap (us_028), **When** the swap notification email was successfully sent, **Then** the patient receives an updated PDF-format appointment details email generated by the same QuestPDF template, with updated date/time and a note "Your appointment has been moved to your preferred slot"; the PDF contains the same fields as the original confirmation; `INSERT audit_logs (action = SwapConfirmationSent)` is written.

## Edge Cases

- What happens if QuestPDF throws an exception during PDF generation (e.g., missing data field)? → The `send_confirmation_job` wraps QuestPDF generation in a try-catch; if the generation step fails, the job logs `ConfirmationPDFGenerationFailed` to `audit_logs`, transitions the Hangfire job to `Failed`, and retries per AC-3; the appointment booking remains confirmed — a PDF generation failure is not a critical booking path error; an alert is surfaced in the admin dashboard (EP-001 audit viewer, us_021).
- What happens if a patient attempts to reach a self-check-in URL from a third-party QR code printed before this feature was explicitly removed? → The SPA router has no route matching `/checkin`, `/self-checkin`, or similar patterns; all unmatched routes render the SPA's 404 "Page not found" view; the API also returns HTTP 404 for any such path; no residual check-in capability exists in the codebase (FR-017).
- What happens if the patient uses "Add to calendar" on SCR-010 and the Google / Outlook calendar sync (UC-003 Postcondition) fails? → The calendar sync failure is already handled per the UC-003 sequence diagram: failures are logged and a toast notification is shown ("Calendar sync unavailable. Your appointment will still be confirmed."); the toast is non-blocking — the PDF confirmation and booking record are unaffected; this is an existing NFR-014 concern, not new work in this story.
- What happens if SCR-010 is accessed via direct URL before a booking is confirmed (e.g., a patient bookmarks the confirmation page)? → The SPA checks for the `appointment_id` in React Router state on SCR-010 mount; if absent, it redirects to SCR-004 (Patient Dashboard); no booking data is displayed without a valid `appointment_id` in state.
