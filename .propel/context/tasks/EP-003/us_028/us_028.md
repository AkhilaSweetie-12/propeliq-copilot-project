# User Story - US_028

## Story ID

- ID Format: US_028

## Story Title

- Preferred Slot Auto-Swap via Hangfire (SELECT FOR UPDATE)

## Description

- As a patient who registered a preferred slot at booking time, I want the system to automatically move my appointment to that preferred slot the moment it opens up, release my original slot back to the pool, and notify me by email and SMS, so that I always get the earliest possible appointment without manual intervention.

## Acceptance Criteria

1. **Given** a patient has an appointment with a non-null `preferred_slot_id` and a `waitlist_entries` record exists with `status = Waiting` for that preferred slot, **When** a `SlotReleasedEvent` is detected by Hangfire (triggered whenever `appointment_slots.is_available` transitions to `true` for any slot), **Then** Hangfire enqueues a `PreferredSlotSwapJob` scoped to the released `slot_id`; the job executes within the Hangfire worker process and is guaranteed exactly-once by Hangfire's internal job state machine.

2. **Given** the `PreferredSlotSwapJob` runs for a released slot, **When** the job acquires row-level locks, **Then** the job executes `BEGIN TRANSACTION`; runs `SELECT waitlist_entries WHERE slot_id=preferred_slot AND status=Waiting ORDER BY requested_at ASC LIMIT 1 FOR UPDATE` (NFR-013); if exactly one row is locked it proceeds with the swap; if no rows match (e.g., patient already cancelled), the transaction is committed with no changes and the job completes with a no-op log entry; if multiple Hangfire workers race for the same slot, the `FOR UPDATE` lock ensures only one worker proceeds.

3. **Given** the `PreferredSlotSwapJob` has locked the FIFO-first waitlist entry, **When** the swap transaction executes, **Then** in a single database transaction: `appointments` is updated with `slot_id = preferred_slot_id, preferred_slot_id = NULL`; `appointment_slots` for the original slot is updated `SET is_available = true`; `appointment_slots` for the preferred slot is updated `SET is_available = false`; `waitlist_entries` is updated `SET status = Fulfilled`; `INSERT audit_logs (action = PreferredSlotSwapped, appointment_id, patient_id, original_slot_id, new_slot_id, swapped_at)`; COMMIT.

4. **Given** the swap transaction commits successfully, **When** the notification step runs, **Then** Hangfire enqueues a `SendSwapNotificationJob` for the patient; the notification job sends an email to the patient's `email` with the new appointment details and a non-blocking SMS via the configured SMS gateway; if email delivery fails, the job retries up to 3 times with exponential backoff and logs each failure to `audit_logs (action = NotificationFailed)`; a failed notification does NOT roll back the swap transaction.

5. **Given** the swap transaction fails mid-execution (e.g., database error), **When** the exception is caught by Hangfire, **Then** the Hangfire job transitions to the `Failed` state and is retried up to 3 times with exponential backoff; the transaction is rolled back to its pre-swap state; neither the appointment record nor the waitlist entry is left in a partially updated state; the `appointment_slots.is_available` field retains its pre-swap value until the transaction succeeds; an error entry is written to `audit_logs (action = PreferredSlotSwapFailed)`.

## Edge Cases

- What happens if the patient cancels their appointment while the `PreferredSlotSwapJob` is queued but not yet run? → The swap job reads the `waitlist_entries` row with `FOR UPDATE`; if the entry's `status` is no longer `Waiting` (e.g., it was set to `Cancelled`), the job detects a no-op condition and commits with no changes; the released slot remains `is_available = true` for other patients.
- What happens if two patients each have the same slot as their preferred slot and it becomes available simultaneously? → Both patients have separate `waitlist_entries` rows with `status = Waiting`; the `ORDER BY requested_at ASC LIMIT 1` clause guarantees FIFO selection; the `FOR UPDATE` row lock ensures only one job proceeds to swap for the released slot; the second patient remains on the waitlist with `status = Waiting` until another slot they prefer is released.
- What happens if the patient has their preferred slot become available after their original appointment has already passed (i.e., the appointment is in the past)? → The swap job validates that `appointments.slot_date >= CURRENT_DATE` before proceeding; if the appointment is historical, the job logs `PreferredSlotSwapSkipped` and marks the `waitlist_entries` row `status = Expired` to prevent future re-processing.
