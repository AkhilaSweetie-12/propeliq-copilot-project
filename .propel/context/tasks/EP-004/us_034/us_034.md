# User Story - US_034

## Story ID

- ID Format: US_034

## Story Title

- Same-Day Queue View, Drag-to-Reorder & Mark Patient Arrived

## Description

- As an authenticated Staff user, I want to view today's full appointment queue ordered by time, reorder it by drag-and-drop as operational needs change, and mark individual patients as Arrived so that the system maintains an accurate real-time record of patient status throughout the day.

## Acceptance Criteria

1. **Given** the Staff navigates to `/staff/queue` (SCR-014 Same-Day Queue View) from SCR-011 or from a post-walk-in redirect (SCR-012 → SCR-014), **When** `GET /api/queue/today` loads, **Then** the queue table renders all appointments for today ordered by `queue_position ASC` (or `slot_time ASC` as tie-breaker); each table row shows: position number with drag handle (⠿), patient name (linked to SCR-016), appointment time, provider name, appointment type, status badge (Scheduled / In Progress / Arrived / Walk-in), and a no-show risk badge; filter chips at the top allow filtering by: All, Arrived, Scheduled, In Progress, Walk-ins, High Risk; only Staff role users can access this screen (FR-018).

2. **Given** the queue table is rendered on a desktop viewport (≥ 768 px wide), **When** the Staff drags a queue row's drag handle (⠿) and drops it into a new position, **Then** the optimistic UI reorders the rows immediately without waiting for the API response; the SPA then calls `PATCH /api/queue/reorder { ordered_appointment_ids: [uuid, ...] }`; the API updates each `appointments.queue_position` value in order and inserts `audit_logs (action=QueueReordered, actor=staff_id, change_summary="positions updated for N appointments")`; if the API call fails, the SPA reverts to the pre-drag order and shows a non-blocking toast error; drag-to-reorder is a desktop-only interaction and is hidden on mobile viewports (FR-018).

3. **Given** a patient row is visible in the queue table, **When** the Staff clicks the "Arrived" button on that row (`aria-label="Mark [patient name] as arrived"`), **Then** the SPA calls `PATCH /api/appointments/{id}/arrive`; the API validates that `appointments.slot_date = CURRENT_DATE`; if valid, it updates `appointments SET status=Arrived, arrived_at=NOW()` and inserts `audit_logs (action=PatientArrived, actor=staff_id, entity=appointment_id, occurred_at=NOW())`; returns HTTP 200; the row status badge updates from "Scheduled" to "Arrived" in the UI; the "Arrived" button is replaced by a "View" button (FR-019).

4. **Given** the Staff attempts to mark a patient arrived whose `appointments.slot_date ≠ CURRENT_DATE` (e.g., a future appointment accidentally visible in the queue), **When** `PATCH /api/appointments/{id}/arrive` validates the date constraint, **Then** the API returns HTTP 400 with message "Appointment not scheduled for today"; the SPA displays a confirmation dialog: "Patient not scheduled for today. Confirm override?"; if Staff confirms, the SPA re-calls `PATCH /api/appointments/{id}/arrive?override=true`; the API proceeds with the update and logs `audit_logs (action=PatientArrivedOverride, actor=staff_id, override_reason=NotScheduledToday)`; if Staff dismisses, no state change occurs.

5. **Given** a Patient or Admin user attempts to access `/staff/queue` or call `PATCH /api/appointments/{id}/arrive`, **When** the API validates the JWT role claim, **Then** HTTP 403 Forbidden is returned; the SPA route guard redirects non-Staff users to their role-appropriate home screen before any API call is made; this dual-layer RBAC enforcement satisfies FR-018 (Staff-only queue management) and FR-019 (Staff-only arrival marking).

## Edge Cases

- What happens when a Staff user reorders the queue and another Staff user is viewing the same queue concurrently? → The queue does not implement real-time WebSocket sync; the second Staff user's view reflects stale data until they click "↻ Refresh"; the PATCH /api/queue/reorder endpoint processes the `ordered_appointment_ids` array atomically — the last write wins; no conflict resolution UI is shown; concurrent edits are operationally acceptable given the low-frequency nature of queue reordering.
- What happens when the Staff filter selects "Walk-ins" and there are no walk-in appointments today? → The filter returns an empty table with message "No walk-in appointments today"; all other filter chips remain accessible; the "+ Add walk-in" button in the page header remains visible regardless of filter selection.
- What happens if the "Arrived" click triggers but the patient was already marked Arrived by another Staff user concurrently? → The API checks `appointments.status` before updating; if `status = Arrived` already, the API returns HTTP 409 with message "Patient has already been marked arrived"; the SPA reloads the queue row for that appointment and updates the button to "View" — no double-arrival log is written.
- What happens when a walk-in added via SCR-012 appears at the bottom of the queue (as per UC-004 postcondition) but the Staff has applied a filter that hides walk-ins? → A transient notification toast informs the Staff "Walk-in added — currently hidden by your active filter"; the toast includes a "Show all" CTA that clears the active filter; the walk-in row becomes visible after the filter is cleared.
