# User Story - US_032

## Story ID

- ID Format: US_032

## Story Title

- Patient Search & Optional Account Creation for Walk-In

## Description

- As an authenticated Staff user initiating a walk-in booking, I want to search for an existing patient by name or date of birth and — if none is found — optionally create a new patient account or proceed with a guest profile, so that every walk-in appointment is correctly linked to a patient identity before slot assignment.

## Acceptance Criteria

1. **Given** the Staff navigates to the Patient Search screen (SCR-013) by clicking "+ Add walk-in" on SCR-011, **When** the page renders, **Then** a single search input field is shown with placeholder "Search by name or date of birth"; the field uses a 300 ms debounce before triggering `GET /api/patients/search?query=X`; each keystroke after the debounce fires the API call; a loading spinner is shown inline while results load; results are displayed as a list of patient cards showing name, DOB, MRN, and last visit date — satisfying the FL-004 real-time search interaction requirement.

2. **Given** the Staff enters a query that matches one or more existing patients, **When** the API returns results, **Then** each result card has a "Select" button; clicking "Select" stores the `patient_id` in component state and navigates to the Walk-In Booking Panel (SCR-012) with the patient summary (name, DOB, MRN, insurance) pre-loaded in the patient-found card; no additional API call is required to populate SCR-012 — the data comes from the search result object.

3. **Given** the Staff search returns no matching patients (empty result set), **When** the empty state renders, **Then** a "Create new patient account" button and a "Proceed without account (guest)" link are shown below the search field; clicking "Create new patient account" expands an inline patient creation form on SCR-013 with required fields: first name, last name, date of birth, phone number, and optional email address; all required fields are validated client-side on blur before form submission.

4. **Given** the Staff submits the new patient creation form, **When** the form passes client-side validation and the Staff clicks "Create & Continue", **Then** the SPA calls `POST /api/patients { first_name, last_name, dob, phone, email?, role=Patient, status=Active }`; the API inserts into `users`, inserts `audit_logs (action=PatientCreatedByStaff, actor=staff_id)`; returns HTTP 201 with `{ patient_id, name, dob, mrn }`; the SPA navigates to SCR-012 with the new `patient_id` pre-loaded; the new account is immediately active — no email verification required for Staff-created accounts (FR-015).

5. **Given** the Staff chooses "Proceed without account (guest)" instead of creating an account, **When** the Staff clicks the guest link, **Then** the SPA navigates to SCR-012 with `patient_id = null` and a `guest_profile = true` flag; the Walk-In Booking Panel renders in guest mode showing a guest badge; the appointment will be inserted with `patient_id = null, guest_profile = true`; the API accepts a null `patient_id` only when `booking_type = WalkIn` and `guest_profile = true` — all other booking types require a non-null `patient_id`.

## Edge Cases

- What happens if the search query is fewer than 2 characters? → The API call is suppressed client-side until the query reaches 2 characters; this prevents unnecessary API calls for single-keystroke inputs; the search results area shows "Enter at least 2 characters to search" as a hint below the input field.
- What happens if two concurrent Staff users try to create a patient account for the same person at exactly the same time (e.g., same DOB + name)? → The API does not enforce uniqueness on `(name, dob)` since duplicate names are clinically valid; each submission creates a separate record with a distinct UUID; deduplication of patient records is handled as an out-of-scope administrative function; no error is returned for near-duplicate inserts.
- What happens if the `POST /api/patients` call fails (e.g., network error)? → The form displays an inline error message "Unable to create account — please try again"; the form data is preserved so the Staff does not need to re-enter; the "Create & Continue" button re-enables immediately; no partial record is left in the database (the transaction is atomic).
- What happens when a Patient or Admin role user navigates directly to `/staff/walkin/search`? → The SPA route guard checks the JWT role claim; `role ≠ Staff` results in an immediate redirect to the role-appropriate home screen (`/portal` for Patient, `/admin` for Admin); the API also returns HTTP 403 for any `GET /api/patients/search` call from non-Staff roles, providing defence in depth (FR-014).
