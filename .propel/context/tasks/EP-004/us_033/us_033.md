# User Story - US_033

## Story ID

- ID Format: US_033

## Story Title

- Walk-In Booking Panel with Same-Day Slot Selection & Override

## Description

- As an authenticated Staff user who has identified a walk-in patient, I want to complete an appointment booking form with chief complaint, provider, urgency level, and a same-day slot, so that the walk-in is confirmed in the system, added to the same-day queue, and — if the patient has an email — receives a PDF confirmation.

## Acceptance Criteria

1. **Given** the Staff arrives on the Walk-In Booking Panel (SCR-012) with a patient identity (patient_id or guest flag) from SCR-013, **When** the page renders, **Then** the patient summary card at the top shows: patient name, DOB, MRN, and insurance badge sourced from the React Router state passed from SCR-013; a "Change patient" link (`#link-new-pt`) navigates back to SCR-013; three required form fields are shown: Chief complaint (text input), Assign provider (select dropdown populated from `GET /api/providers/available-today`), and Urgency level (select: Standard / Urgent / Emergency); one optional field: Staff notes (textarea).

2. **Given** the form is visible, **When** the same-day slot mini-grid renders, **Then** the grid shows today's `appointment_slots` retrieved from `GET /api/slots?date=TODAY` and displays each slot as Available (green) or Unavailable (grey) using the same CSS variables as the patient-facing calendar — ensuring visual consistency; slots use both colour and ARIA labels for colour-blind safety; the Staff can click exactly one Available slot to select it (`aria-selected="true"`); the confirmation action button remains disabled until a slot is selected and all required fields are filled.

3. **Given** the Staff selects an Available slot, fills all required fields, and clicks "Add to queue →" (`#btn-confirm`), **When** the SPA submits `POST /api/appointments/walkin { patient_id?, guest_profile?, slot_id, chief_complaint, provider_id, urgency_level, notes?, booking_type=WalkIn }`, **Then** the API begins a `BEGIN TRANSACTION`; executes `SELECT appointment_slots WHERE slot_id=X FOR UPDATE`; if the slot is available inserts into `appointments (booking_type=WalkIn, status=Confirmed, created_by=staff_id)`, updates `appointment_slots SET is_available=false`, inserts `audit_logs (action=WalkInBooked, actor=staff_id)`; COMMITs; returns HTTP 201; the SPA navigates to SCR-014 (Same-Day Queue View) with the new appointment visible at the bottom of the queue.

4. **Given** all slots for today are Unavailable when the form loads, **When** the Staff selects an Unavailable slot and clicks the slot, **Then** an "Override booking" confirmation dialog appears with message: "No available slot selected. Walk-in bookings can override scheduling. A justification note will be auto-logged. Confirm override?"; if the Staff confirms, the booking proceeds with `slot_override = true` and `audit_logs (action=WalkInSlotOverride, actor=staff_id, note=auto-logged_override_justification)`; the audit log entry captures the override explicitly; if the Staff dismisses the dialog, no action is taken and the slot grid remains editable (extension 4a in UC-004).

5. **Given** the walk-in booking transaction commits successfully and the patient has a non-null `email` address, **When** Hangfire picks up the `send_walkin_confirmation_job(appointment_id)` enqueued by the API, **Then** the job sends an email with the same QuestPDF confirmation document template used in EP-003 (us_030), containing appointment date, time, provider, and cancellation policy; if the patient is a guest profile (null `patient_id`) or has no email address, the job is skipped and `audit_logs (action=ConfirmationSkipped_NoEmail)` is written; the walk-in booking is NOT cancelled or rolled back if the email job is skipped.

## Edge Cases

- What happens if a Patient or Admin role user calls `POST /api/appointments/walkin`? → The API enforces RBAC at the route level; non-Staff JWT role claims receive HTTP 403; the SPA route guard also prevents non-Staff users from accessing `/staff/walkin/panel`; this is the dual-layer enforcement for FR-014.
- What happens if the provider selected in the dropdown has no slots available today? → The slot mini-grid is filtered by `provider_id` when the Staff selects a provider from the dropdown; `GET /api/slots?date=TODAY&provider_id=X` re-fetches slots filtered by provider; if no slots exist for that provider, the mini-grid shows all cells as Unavailable and the walk-in override dialog becomes the only path; the provider filter is applied dynamically without a full page reload.
- What happens if the Staff's session expires between opening SCR-012 and clicking "Add to queue →"? → The API returns HTTP 401; the SPA catches the error and routes to the session expiry modal; the form state is preserved in component memory so the Staff can re-authenticate and continue without re-entering data; if re-authentication fails, the form data is lost and the Staff must restart the walk-in flow from SCR-013.
- What happens if the slot selected in the mini-grid is taken by another booking (e.g., a concurrent online booking) between click and submission? → The `SELECT FOR UPDATE` in the transaction detects `is_available = false`; the API returns HTTP 409 Conflict; the SPA re-fetches the slot grid and shows an inline error "That slot was just taken — please select another"; the slot grid updates to reflect the new state without page reload.
