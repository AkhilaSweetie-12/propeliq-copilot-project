# User Story - US_035

## Story ID

- ID Format: US_035

## Story Title

- No-Show Risk Scoring Engine & Queue Risk Badges

## Description

- As a Staff user reviewing the same-day appointment queue, I want each appointment row to display a no-show risk badge computed by a rule-based engine so that I can proactively prioritise outreach for High-risk patients and reduce the clinic's no-show rate.

## Acceptance Criteria

1. **Given** a new appointment is confirmed (online booking via us_027 or walk-in via us_033), **When** the booking transaction commits and the API returns HTTP 201, **Then** Hangfire enqueues an `AppointmentRiskScoringJob(appointment_id)` immediately after the commit; the same job is also re-triggered nightly at 00:00 UTC by a recurring Hangfire CRON for all upcoming `status=Confirmed` appointments with `slot_date >= TODAY`; this ensures risk scores are refreshed as patient history changes overnight (UC-009).

2. **Given** the `AppointmentRiskScoringJob` runs for a given `appointment_id`, **When** the rule engine evaluates the appointment, **Then** the .NET `NoShowRiskEngine` service executes the following deterministic scoring formula:
   - `prior_no_shows = COUNT appointments WHERE patient_id=X AND status=NoShow`
   - `lead_time_days = (slot_date - created_at).TotalDays`
   - `insurance_unmatched = (appointments.insurance_result IN [Unmatched, NotFound]) ? 1 : 0`
   - `intake_incomplete = (patient_intakes WHERE patient_id=X EXISTS AND status=Complete) ? 0 : 1`
   - `score = (prior_no_shows × 30) + (lead_time_days < 2 ? 30 : 0) + (insurance_unmatched × 20) + (intake_incomplete × 20)`
   - Risk level assignment: Low (score 0–30), Medium (score 31–60), High (score 61–100)
   - New patients with no prior history default to Medium (score = 50) per UC-009 extension 1a.

3. **Given** the risk engine completes its evaluation, **When** the score and level are determined, **Then** the engine executes `UPDATE appointments SET no_show_risk = 'Low|Medium|High', risk_score = N` and inserts `audit_logs (action=RiskScored, entity_id=appointment_id, change_summary="risk=High, score=80")`; the update is committed in a separate transaction from the booking transaction so that a scoring failure never rolls back a completed booking; if the job fails, Hangfire retries up to 3 times; after 3 failures, the appointment retains its previous `no_show_risk` value (or NULL for new appointments).

4. **Given** `GET /api/queue/today` returns the appointment list, **When** the Staff views SCR-014 (Same-Day Queue View) or SCR-011 (Staff Dashboard queue preview), **Then** each appointment row renders a `RiskBadge` component displaying: icon (↓ for Low, — for Medium, ↑ for High), text label ("Low", "Medium", "High"), and semantic colour (green for Low, amber for Medium, red for High); the badge always includes BOTH icon + text (never colour-alone); `aria-label="[level] no-show risk"` is set on each badge element satisfying UXR-102; appointments with `no_show_risk = NULL` (score not yet computed) render a neutral grey "—" badge with `aria-label="Risk not yet assessed"`.

5. **Given** the Staff reviews a High-risk appointment row on SCR-014, **When** the Staff hovers over the `↑ High` risk badge (desktop only), **Then** an optional tooltip displays the individual contributing risk factors: e.g., "2 prior no-shows · Booked same-day · Insurance unmatched"; the tooltip is supplementary — the badge text label is always visible without hover; the tooltip does not replace the text label (FR-034, UXR-102 — text label not tooltip-only).

## Edge Cases

- What happens if `patient_id` is NULL (guest profile walk-in)? → The scoring engine checks for a null `patient_id` before querying history tables; a null `patient_id` results in a default score of 50 (Medium risk) — the same default as new patients with no history; the audit log records `change_summary="risk=Medium (guest profile — no history)"`.
- What happens if the nightly CRON risk scoring job runs while a Staff user is actively viewing SCR-014? → The queue API response does not push updates to the SPA; the risk badge on the Staff's screen reflects the score at the time of their last `GET /api/queue/today` call; a manual "↻ Refresh" will fetch the latest scores; no WebSocket or SSE is required for this story.
- What happens if `prior_no_shows = 3` (score contribution = 90 from that factor alone, already above High threshold)? → The score formula caps at 100; all individual factor contributions are summed without a cap per factor; the final `score` is clamped to the range [0, 100] before assignment: `risk_score = MIN(100, MAX(0, score))`; the High level is assigned for any score ≥ 61.
- What happens if the `AppointmentRiskScoringJob` fails all 3 retries for a new booking? → The appointment retains `no_show_risk = NULL`; the SCR-014 badge renders "—" with `aria-label="Risk not yet assessed"`; the Hangfire job transitions to the Failed state; an alert is surfaced in the Hangfire dashboard (accessible to Admin role users via the `/hangfire` route); no patient-visible error is shown.
