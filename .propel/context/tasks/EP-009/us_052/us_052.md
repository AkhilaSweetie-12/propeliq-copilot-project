# User Story - US_052

## Story ID

- ID Format: US_052

## Story Title

- Responsive Layout: 3-Breakpoint Grid, Hamburger Navigation & Mobile Form Stacking

## Description

- As a patient or staff member accessing the platform from a mobile phone or tablet, I want all screens to adapt fluidly to my device's screen width — using a single-column mobile grid, collapsing the sidebar into a hamburger menu, and stacking form fields into a single column — so that I can complete bookings, review patient data, and manage queues from any device without horizontal scrolling or UI elements overflowing the viewport.

## Acceptance Criteria

1. **Given** any screen (SCR-001 through SCR-021) is rendered at the three supported breakpoints, **When** the CSS grid layout is applied, **Then** the layout uses a 4-column grid at Mobile width (320–767px), an 8-column grid at Tablet width (768–1023px), and a 12-column grid at Desktop width (1024px+); the grid is implemented as a TailwindCSS responsive grid (`grid-cols-4 md:grid-cols-8 lg:grid-cols-12`) configured from the design system tokens in `tailwind.config.ts`; no horizontal scrollbar appears on any screen at any supported breakpoint (measured with `document.documentElement.scrollWidth === document.body.scrollWidth` assertion in a Playwright viewport test at 320px, 768px, and 1024px widths); content reflows within the grid columns at each breakpoint without content overlap, truncation of actionable elements, or loss of information; the Playwright responsive audit script visits all 21 routes at each of the three breakpoints and asserts zero horizontal overflow (UXR-301).

2. **Given** any authenticated screen (SCR-003 through SCR-021) is rendered at a viewport width narrower than 768px, **When** the layout renders the navigation component, **Then** the persistent sidebar navigation (rendered at ≥ 768px) is replaced with a hamburger menu button in the top-left of the topbar (`aria-label="Open navigation menu"`, `aria-expanded="false"` when closed); tapping or clicking the hamburger button renders a slide-over drawer from the left edge of the viewport containing all navigation items available to the user's role (Patient, Staff, or Admin); the drawer overlays the page content and includes a close button (`aria-label="Close navigation menu"`) and a backdrop overlay that closes the drawer on click; the drawer traps focus while open (per UXR-202 accessibility requirement from us_051); at viewport width ≥ 768px, the hamburger button is hidden (`hidden md:block` pattern) and the sidebar reverts to its persistent rendered state; the hamburger-to-sidebar transition is smooth and does not produce a flash of unstyled content (UXR-302).

3. **Given** any form screen with a multi-column field layout (SCR-002 Registration, SCR-006 AI Intake Chat supplementary fields, SCR-009 Booking & Insurance, SCR-013 Patient Search) is rendered at a viewport width narrower than 768px, **When** the CSS responsive classes are applied, **Then** all two-column field groups (e.g., First Name / Last Name, Insurance Provider / Member ID, Street Address / City/State) collapse to single-column stacking — each field occupying the full 4-column mobile grid width; the field stacking order is logical (top-to-bottom in reading order matching the desktop left-to-right layout); the Submit / Continue CTA button is full-width on mobile (`w-full` class applied below the `md:` breakpoint); touch targets for all interactive form elements (inputs, selects, buttons) meet the 44px minimum height (`min-height: 44px`); no form field is hidden, clipped, or obscured at the 320px minimum viewport width; a Playwright snapshot test at 375px viewport width (iPhone SE equivalent) asserts that the single-column layout produces zero horizontal overflow and all fields are visible in the DOM (UXR-303).

4. **Given** a mobile user (viewport < 768px) is on the Staff Queue View (SCR-014) or Patient 360° View (SCR-016) — screens with complex multi-column data tables or summary grids, **When** these data-heavy screens render at mobile breakpoint, **Then** data tables reflow to a card-based layout (each row becomes a stacked card with field-label: value pairs) or use horizontal scroll scoped to the table container only (not full-page horizontal scroll) — the full-page viewport has no horizontal overflow; the no-show risk badge and action buttons in SCR-014 remain visible and actionable at mobile (44px minimum touch target); the 360° Patient View summary sections in SCR-016 stack vertically with section headings remaining visible; this behaviour is consistent with the TailwindCSS responsive table pattern defined in the component inventory (UXR-301).

5. **Given** the responsive layout implementation is complete across all 21 screens, **When** the Playwright responsive audit runs in CI across all three breakpoints (320px, 768px, 1024px) for all 21 routes, **Then** the following assertions all pass: (a) zero horizontal overflow on any screen at any breakpoint (scrollWidth equals clientWidth); (b) hamburger menu button is visible at 320px and hidden at 1024px on all authenticated screens; (c) form field stacking is confirmed on all 4 form-containing screens at 375px (fields rendered as full-width single column); (d) no interactive element is smaller than 44px height at mobile breakpoint; (e) drawer open/close interaction works correctly at 375px viewport (UXR-301, UXR-302, UXR-303).

## Edge Cases

- What happens if a Staff user opens the hamburger drawer on a narrow screen and then rotates the device to landscape (viewport crossing the 768px breakpoint mid-session)? → The drawer state is managed in React component state; a `useEffect` hook listens to the `resize` event via `window.matchMedia('(min-width: 768px)')` — if the viewport crosses the 768px threshold while the drawer is open, the drawer is automatically closed and the sidebar renders in its persistent state; this prevents a state mismatch where both the drawer overlay and the sidebar are simultaneously visible.
- What happens if the platform is accessed on a device with an unusual viewport width (e.g., 540px — between mobile and tablet breakpoints)? → The responsive grid uses TailwindCSS's standard breakpoint system where the `md:` prefix activates at 768px; a 540px viewport falls in the mobile (< 768px) range and receives the 4-column grid, hamburger navigation, and single-column form stacking; this is the intended behaviour — Tailwind's mobile-first approach ensures all styles below the next breakpoint threshold apply the mobile layout; no edge cases exist between 320px and 767px since all styles are applied continuously.
- What happens if a custom component renders its own hard-coded grid (e.g., the CalendarGrid in SCR-008 uses a 7-column weekly grid that cannot be reduced)? → The booking calendar's 7-column week grid (SCR-008) is a special-case component with its own responsive behaviour: at mobile breakpoint it renders a single-day view (replacing the 7-day week view) with left/right arrow navigation to scroll through days; this is defined in the component inventory and is the exception to the 4-column mobile grid rule; the full-page grid still uses 4 columns at mobile, with the CalendarGrid occupying all 4 columns in its single-day mode.
- What happens if the slide-over drawer animation causes a performance issue on low-end mobile devices (janky scroll or delayed open)? → The drawer animation uses `transform: translateX()` rather than `left` positioning — GPU-accelerated CSS transforms are used for all slide-over animations to prevent layout thrashing; animation duration is 200ms (`transition-transform duration-200`) consistent with the 200ms visual feedback budget (UXR-501 from EP-010); the drawer element uses `will-change: transform` only during the animation (applied via JavaScript class toggle) and removed after the transition ends to prevent unnecessary GPU memory allocation.
- What happens if the Playwright responsive audit fails at a specific breakpoint for a newly added screen — does it block all other breakpoint tests? → The Playwright responsive audit is structured as a parameterized test using `test.each` over the full 21-route × 3-breakpoint matrix (63 test cases total); each combination is an independent test case — a failure on one route/breakpoint combination does not block execution of other combinations; the CI report surfaces the specific route and breakpoint that failed, enabling targeted debugging without re-running the full audit.
