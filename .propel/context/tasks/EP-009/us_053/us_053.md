# User Story - US_053

## Story ID

- ID Format: US_053

## Story Title

- Design Token System, Consistent Page Header & SPA Time-to-Interactive < 3s

## Description

- As a patient or staff member opening the platform for the first time, I want the application to load and become interactive within 3 seconds on a standard broadband connection, present a visually consistent branded header across every authenticated screen, and maintain a uniform visual language (colours, spacing, and typography) that never changes without a system-wide design decision — so that the platform feels fast, professional, and coherent across every page and device.

## Acceptance Criteria

1. **Given** the React SPA production build is served from the ASP.NET Core 9 API as static files, **When** a user navigates to any public entry point (`/login`, `/register`) on a standard broadband connection (≥ 10 Mbps download, per Lighthouse definition), **Then** the Time-to-Interactive (TTI) as measured by Lighthouse CI (Lighthouse v12 configured in the GitHub Actions workflow) is less than 3 seconds; this is achieved through: (a) the SPA shell `index.html` inlining critical CSS (above-the-fold typography, topbar, login card styles) via the Vite `vite-plugin-critical` or manual `<style>` injection at build time; (b) route-based code splitting using `React.lazy` and `Suspense` — every route beyond the login shell is a lazily loaded chunk; (c) the SPA static files (`index.html`, JS chunks, CSS) are served with `Cache-Control: public, max-age=31536000, immutable` headers for all content-hashed assets; (d) the ASP.NET Core static files middleware is configured with `ServeStaticFiles` pointing to the Vite `dist/` output folder and serves pre-compressed `.gz` and `.br` variants when available; the Lighthouse CI check runs on every PR targeting `main` and fails the PR if TTI ≥ 3 seconds (NFR-002).

2. **Given** the design token system is configured in `tailwind.config.ts` sourcing all values from `designsystem.md`, **When** any UI component across all 21 screens is inspected for inline styles or hard-coded CSS values, **Then** zero hard-coded colour hex values (e.g., `#162D4A`, `#0D9488`), spacing pixel values (e.g., `padding: 16px`), or typography values (e.g., `font-size: 14px`) are present in any React component `.tsx` file or CSS module — all values are expressed exclusively through TailwindCSS utility classes mapped to design tokens (e.g., `bg-navy-700`, `text-teal-500`, `p-4`, `text-sm`); the design token palette, spacing scale, and type scale in `tailwind.config.ts` match the `designsystem.md` token specification exactly (verified by a TypeScript token-validation script that compares `tailwind.config.ts` `theme.extend` values against the parsed `designsystem.md` token table and fails CI if any discrepancy is found); a `no-restricted-syntax` ESLint rule configured in `.eslintrc.ts` flags any `style={{ }}` prop containing hex colour values, numeric pixel padding, or numeric pixel font sizes — the ESLint check runs in CI and blocks PR merge on any violation (UXR-401).

3. **Given** any authenticated user (Patient, Staff, or Admin) navigates to any authenticated screen (SCR-003 through SCR-021), **When** the page renders, **Then** the consistent page header is present in the DOM as the first child of the `<body>` (`role="banner"`) containing: (a) the PropelIQ Health logo (`<div class="topbar-logo">` with the "P" logo icon and "PropelIQ Health" text); (b) a role label badge showing the current user's role (Patient / Staff / Admin) with appropriate background colour sourced from the design token (navy-600 for Staff, teal-500 for Patient, neutral-600 for Admin); (c) the user's avatar/initials circle (sourced from `users.first_name + users.last_name` initials); (d) a "Sign out" logout button that calls `DELETE /api/auth/session` and clears the JWT from `localStorage`, redirecting to SCR-001 (Login); the page header component is a shared `<TopBar>` React component used across all 19 authenticated screens — it is not re-implemented per screen; the `<TopBar>` unit test asserts that all four elements (logo, role badge, avatar, logout button) render correctly for each of the three role types (Patient, Staff, Admin) (UXR-402).

4. **Given** the design token ESLint rule from AC-2 is active, **When** a developer attempts to merge a PR that introduces a hard-coded colour in a new component (e.g., `style={{ color: '#FF0000' }}`), **Then** the ESLint CI check reports a `no-restricted-syntax` violation with the message "Hard-coded style value detected — use TailwindCSS design token classes instead (see designsystem.md)" and the PR is blocked from merging; the ESLint rule also flags `className` values containing arbitrary Tailwind values (e.g., `text-[#FF0000]` or `p-[12px]`) — arbitrary value syntax is prohibited in the codebase; the only exception is `wireframe-shared.css` (which is a design reference document, not production code) and is explicitly excluded from the ESLint rule's file scope (UXR-401).

5. **Given** the SPA performance, design token, and page header implementations are complete, **When** a full platform integration smoke test runs in CI, **Then** the following assertions all pass: (a) Lighthouse CI reports TTI < 3 seconds for SCR-001 (Login) and SCR-004 (Patient Dashboard) on simulated broadband (10 Mbps); (b) the ESLint token check reports zero violations across all `.tsx` files in the `/client/src` directory; (c) the TypeScript token-validation script confirms `tailwind.config.ts` tokens match `designsystem.md` with zero discrepancies; (d) the `<TopBar>` renders on all 19 authenticated screens (SCR-003 through SCR-021) confirmed by Playwright asserting the topbar `role="banner"` element is present with logo, role badge, avatar, and logout button visible on each authenticated route (NFR-002, UXR-401, UXR-402).

## Edge Cases

- What happens if the SPA's lazy-loaded chunk for a deep route (e.g., SCR-021 Audit Log Viewer) fails to load due to a network interruption mid-session? → React `Suspense` error boundaries catch chunk-loading failures and display the full-page error state with a "Retry" button (UXR-603 from EP-010); the error boundary calls `window.location.reload()` on retry, which re-fetches the chunk; the `Cache-Control: immutable` headers ensure the chunk is served from the browser cache on retry if the network has recovered; chunk-loading failures do not affect routes that are already loaded and cached.
- What happens if the Lighthouse CI TTI measurement is flaky (e.g., CI runner is slower than standard and produces TTI ≥ 3s even for a performant build)? → The Lighthouse CI configuration uses `--preset=desktop` with a simulated 10 Mbps connection (not the default slow 4G throttling); the CI check allows a 200ms tolerance buffer (`assert.ttiFastEnough: [warn, {minScore: 0.9}]` — score equivalent to TTI < 3.2s tolerated, < 3.0s passes); if flakiness persists, the TTI assertion is compared against 3 sequential Lighthouse runs and the median value is used; the CI configuration documents the measurement methodology in the `README.md` to ensure reproducibility.
- What happens if the `designsystem.md` token table is updated (e.g., the brand teal-500 hex value changes) without updating `tailwind.config.ts`? → The TypeScript token-validation CI script detects the discrepancy between `designsystem.md`'s token table and `tailwind.config.ts`'s `theme.extend` values and fails the CI check with a diff showing which tokens are out of sync; the PR author is required to update `tailwind.config.ts` to match before merge; the script is the authoritative synchronisation mechanism ensuring the design system remains the single source of truth for all token values.
- What happens if the `<TopBar>` component renders on an unauthenticated screen (SCR-001 Login or SCR-002 Register) where no user role or avatar is available? → The `<TopBar>` component receives an `isAuthenticated` prop; when `false` (on SCR-001 and SCR-002), the component renders only the logo — the role badge, avatar, and logout button are conditionally hidden via `{isAuthenticated && ...}` guards; the un-authenticated header variant is visually simpler (logo-only) and is consistent with the wireframes for SCR-001 (Login) and SCR-002 (Register) which show no avatar or role badge; this is the correct behaviour per UXR-402 which specifies "all authenticated screens (SCR-003 onwards)".
- What happens if a future sprint introduces a new page that a developer forgets to wrap in the authenticated layout component (and therefore misses the `<TopBar>`)? → All authenticated routes are defined within a React Router `<AuthenticatedLayout>` wrapper component that renders `<TopBar>` as a required child before the `<Outlet>` — individual page components do not import `<TopBar>` directly; adding a new route under the `<AuthenticatedLayout>` automatically includes the topbar; the Playwright CI smoke test in AC-5 crawls all authenticated routes and asserts `role="banner"` presence, which would catch a missing topbar on any new authenticated page before merge.
