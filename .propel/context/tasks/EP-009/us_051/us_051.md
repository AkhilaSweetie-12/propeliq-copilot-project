# User Story - US_051

## Story ID

- ID Format: US_051

## Story Title

- WCAG 2.2 AA Accessibility: Colour Contrast, Keyboard Navigation, ARIA Labels & Form Associations

## Description

- As a patient or staff member with a visual impairment, motor disability, or who relies on a screen reader, I want the platform to meet WCAG 2.2 AA accessibility standards across every screen — with sufficient colour contrast, a logical keyboard Tab order, descriptive ARIA labels on interactive elements, and correctly labelled form fields with associated error messages — so that I can use the platform independently and effectively without relying on a mouse or on colour alone.

## Acceptance Criteria

1. **Given** any screen in the platform (SCR-001 through SCR-021) is rendered at any breakpoint, **When** the colour contrast of text and UI components is measured, **Then** all body text (≥ 18px regular or ≥ 14px bold) achieves a contrast ratio of at least 4.5:1 against its background; all UI component elements (borders of form fields, icon-only buttons, status badges, progress indicators, navigation links) achieve a contrast ratio of at least 3:1 against adjacent colours; the WAVE or axe-core automated accessibility audit run in CI (via `@axe-core/playwright` in the Playwright test suite) reports zero WCAG 2.2 AA critical or serious contrast violations across all 21 screens; colour is never the sole means of conveying information — every colour-coded indicator (availability slots, no-show risk badges, extraction status, code confidence) also carries a text label or icon (UXR-201).

2. **Given** a user navigates any screen (SCR-001 through SCR-021) using only the keyboard (Tab, Shift+Tab, Enter, Space, Arrow keys), **When** focus moves between interactive elements, **Then** the Tab order follows the visual reading order (top-to-bottom, left-to-right in LTR layout) with no focus traps outside of modal dialogs; every focused element displays a visible focus ring (`outline: 2px solid var(--color-teal-500); outline-offset: 2px` consistent with `wireframe-shared.css`); the focus ring remains visible at browser zoom levels 100%, 150%, and 200%; modal dialogs (session expiry warning, Code Finalise Confirm, walk-in override confirm, conflict acknowledgement) trap focus within their bounds while open and return focus to the triggering element on close; skip-to-main-content link is present and functional on all authenticated screens (SCR-003 through SCR-021), positioned as the first focusable element in the DOM (UXR-202).

3. **Given** any icon-only button, status badge, or progress indicator is rendered on any screen, **When** a screen reader (NVDA on Windows / VoiceOver on macOS) announces the element, **Then** the screen reader reads a descriptive label that communicates the element's purpose — not its visual appearance; specifically: icon-only buttons carry `aria-label` describing the action (e.g., `aria-label="Mark Jane Smith as arrived"`, `aria-label="View source evidence for 99213"`, `aria-label="Upload clinical document"`); status badges carry `aria-label` describing the full state (e.g., `aria-label="No-show risk: High"`, `aria-label="Status: Accepted"`, `aria-label="Extraction status: Processing"`); progress indicators carry `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax` and a descriptive `aria-label` (e.g., `aria-label="Extraction progress"`); no interactive element presents solely as a generic element (e.g., `<div onClick>`) without a proper ARIA role (UXR-203).

4. **Given** any form is rendered on a screen that contains form fields (SCR-002 Registration, SCR-006 AI Intake Chat, SCR-007 Manual Intake, SCR-009 Booking & Insurance, SCR-013 Patient Search, SCR-020 User Detail), **When** the form is inspected via axe-core form field association audit, **Then** every `<input>`, `<select>`, and `<textarea>` element has an associated visible `<label>` element (via `for`/`id` pairing or wrapping `<label>`) — no placeholder-only labelling; when a field has a validation error, the error message element is referenced by the field via `aria-describedby` matching the error element's `id`; when multiple error messages appear for a single field (e.g., required + format), all are contained within the single referenced error element; the axe-core `form-field-multiple-labels`, `label`, and `aria-describedby-id-exists` rules all pass with zero violations across all 6 form-containing screens; password fields include `autocomplete="current-password"` or `autocomplete="new-password"` as appropriate (UXR-204).

5. **Given** the accessibility implementation is complete across all 21 screens, **When** the Playwright E2E accessibility test suite runs in CI (against the running application in GitHub Codespaces), **Then** the `@axe-core/playwright` scan across all 21 routes reports zero WCAG 2.2 AA critical violations; a manual keyboard-navigation smoke test (automated via Playwright `page.keyboard.press('Tab')` chain) visits SCR-001 (Login), SCR-004 (Patient Dashboard), SCR-008 (Booking Calendar), SCR-014 (Queue View), and SCR-018 (Coding Panel) and confirms: (a) all interactive elements receive focus in visual order; (b) no focus is lost on dynamic content insertion; (c) modals trap and restore focus correctly; (d) the skip-to-main-content link is the first Tab stop on authenticated screens (UXR-201, UXR-202, UXR-203, UXR-204).

## Edge Cases

- What happens when a status badge must convey state through both colour and icon (e.g., the no-show risk badge uses red/amber/green colouring plus ↑/—/↓ arrow icons), and a user has a colour vision deficiency? → Per UXR-102 and UXR-201, the badge always shows a text label ("High", "Medium", "Low") alongside the arrow icon — colour is supplementary, never sole conveyance; users with full colour vision deficiency (achromatopsia) still read the text label; this satisfies WCAG 1.4.1 (Use of Colour) at Level A.
- What happens if a third-party component library introduces elements without proper ARIA roles (e.g., a date-picker or calendar grid without `role="grid"`)? → The platform uses React ARIA (`@react-aria/calendar`) for the booking calendar grid (SCR-008), which provides WAI-ARIA Grid keyboard navigation and role management built-in; all other interactive components are custom-built and explicitly include ARIA roles per the component inventory; the `@axe-core/playwright` CI scan will catch any new ARIA role omission before merge; third-party libraries are vetted for WCAG compliance before adoption.
- What happens if the TailwindCSS purge removes an ARIA-related utility class from the production bundle (e.g., `sr-only` used for screen-reader-only text)? → `sr-only` and its `focus:not-sr-only` variant are explicitly listed in the TailwindCSS `safelist` array in `tailwind.config.ts`, preventing purge removal; a post-build CI step checks for the presence of the `.sr-only` CSS rule in the production bundle and fails the build if missing; this ensures screen-reader-only labels are always available in production.
- What happens when a dynamic content update (e.g., the extraction status badge changing from "Processing" to "Extracted" after a Hangfire job completes) occurs — does the screen reader announce the change? → Status badge elements that can update dynamically are wrapped in an `aria-live="polite"` region so screen readers announce the change without interrupting the user's current focus position; badge elements with critical status changes (e.g., "Extraction Failed") use `aria-live="assertive"` to immediately interrupt and announce the failure; the live region approach is tested with VoiceOver and NVDA in the accessibility smoke test suite.
- What happens if a developer adds a new form screen in a future sprint without following the label association pattern? → The axe-core CI scan configured in Step 4 covers all pages returned by the Playwright site-crawl; any new form page that fails `label` or `aria-describedby` rules will fail the CI check and block the PR merge; the Playwright accessibility test is part of the required CI quality gate, not an optional check — this ensures ongoing compliance as the codebase grows.
