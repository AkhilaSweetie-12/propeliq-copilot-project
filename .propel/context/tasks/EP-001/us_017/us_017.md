# User Story - US_017

## Story ID

- ID Format: US_017

## Story Title

- Patient Self-Registration with Email Verification

## Description

- As a new patient, I want to create an account using my email and a valid password so that I can access the platform with a verified, uniquely-owned identity.

## Acceptance Criteria

1. **Given** an unauthenticated visitor on the Registration screen (SCR-002), **When** they submit name, email, and password where the password contains at least 8 characters including one uppercase letter, one lowercase letter, one digit, and one symbol, **Then** the API calls `POST /api/auth/register`, inserts a `users` row with `status = Inactive`, `role = Patient`, writes an `audit_logs` entry with `action_type = UserRegistered`, and responds with HTTP 201; the React SPA displays "Check your email to verify your account."

2. **Given** the registration form is displayed, **When** the patient enters a password that does NOT meet complexity rules (e.g., all lowercase, no digit, no symbol), **Then** the SPA shows an inline validation error per failing rule on field blur — without calling the API — so that the user understands exactly which rule was violated before submission.

3. **Given** a patient attempts to register with an email address already stored in `users`, **When** the API receives the `POST /api/auth/register` request, **Then** the API returns HTTP 409 Conflict with a generic message "An account with this email already exists"; no new `users` row is created; no verification email is sent.

4. **Given** the patient receives the verification email and clicks the signed verification link (`GET /api/auth/verify?token=XYZ`), **When** the token is valid and unexpired, **Then** the API updates `users.status` to `Active`, inserts an `audit_logs` entry with `action_type = EmailVerified`, and redirects the patient to the Login screen (SCR-001); the patient can now log in.

5. **Given** the patient clicks a verification link, **When** the token has expired (configurable TTL, default 24 hours), **Then** the API returns HTTP 400 with message "Verification link expired"; the Email Verification screen (SCR-003) displays a "Resend verification email" button; clicking it triggers `POST /api/auth/resend-verification`; a new signed token is issued and a new email is sent; the previous token is invalidated.

6. **Given** the Registration screen (SCR-002), **When** rendered on a viewport ≤ 768 px, **Then** the form displays as a single-column layout (matching the SCR-002 wireframe mobile breakpoint) with all fields, labels, and the submit button fully visible without horizontal scrolling.

## Edge Cases

- What happens when the patient submits the registration form twice in rapid succession (double-click)? → The submit button is disabled immediately on first click (disabled + spinner state) to prevent duplicate `POST /api/auth/register` calls; if a duplicate reaches the API, the second call returns HTTP 409 due to the unique email index on `users`.
- What happens when the verification email link is clicked after the `users` row has already been activated (e.g., patient clicks link twice)? → The API checks `users.status`; if already `Active`, it returns HTTP 200 with message "Your account is already verified" and redirects to the Login screen — idempotent and safe.
- What happens when the SMTP relay is unavailable when the verification email is sent? → The API still returns HTTP 201 to the patient (account row is created, Inactive); the email-send failure is logged to Serilog at `ERROR` level; a background Hangfire retry job re-attempts the SMTP send up to 3 times within 10 minutes; if all retries fail, an `audit_logs` entry with `action_type = EmailVerificationFailed` is written.
