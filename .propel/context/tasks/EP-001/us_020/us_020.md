# User Story - US_020

## Story ID

- ID Format: US_020

## Story Title

- Admin User Management — Create, Update & Deactivate

## Description

- As an Admin user, I want to create new Staff and Patient accounts, update existing user profiles and roles, and deactivate accounts without permanently deleting them so that I can control platform access while preserving the complete historical record for every user.

## Acceptance Criteria

1. **Given** an Admin on the User Management List screen (SCR-019), **When** they search by name, email, or role filter and the API processes `GET /api/admin/users?query=X&role=Staff`, **Then** the response returns a paginated list of `[{user_id, name, email, role, status, last_active}]`; the SPA renders rows matching the search with an "Edit" link per row and a "+ Create user" button; an empty-state message is shown when no users match.

2. **Given** an Admin on SCR-019 clicks "+ Create user" and fills in name, email, and role on SCR-020 (User Detail/Edit), **When** the Admin submits the form and the API processes `POST /api/admin/users {name, email, role}`, **Then** if the email is new, the API inserts a `users` row with `status = Active`, inserts an `audit_logs` entry with `action_type = UserCreated`, and returns HTTP 201 with `{user_id}`; the SPA displays toast "Staff account created"; if the email already exists, the API returns HTTP 409 and the SPA displays inline error "Email already registered."

3. **Given** an Admin on SCR-020 (User Detail/Edit) views an existing user's profile, **When** they change the role or status field and submit `PATCH /api/admin/users/{id} {role?, status?}`, **Then** the API validates the request (Admin cannot change their own role or status), applies the update via `UPDATE users SET role=X, status=Y WHERE user_id=Z`, inserts an `audit_logs` entry with `action_type = UserUpdated`, and returns HTTP 200; the SPA displays toast "User updated" and refreshes the displayed fields.

4. **Given** an Admin on SCR-020 (User Detail/Edit) clicks "Deactivate Account" in the Danger Zone section, **When** the confirmation prompt is accepted and the API processes `DELETE /api/admin/users/{id}`, **Then** the API executes a **soft-delete only**: `UPDATE users SET status = Inactive WHERE user_id = X` — no row is physically removed; an `audit_logs` entry with `action_type = UserDeactivated` is written; the API returns HTTP 200; the SPA navigates back to SCR-019 and the user row shows status badge "Inactive."

5. **Given** a deactivated user (status = Inactive) attempts to log in, **When** the login API processes their credentials, **Then** the API returns HTTP 401 with the generic message "Invalid credentials" (same response as wrong password — no disclosure of account deactivation status); the deactivated user cannot access any platform resources.

6. **Given** an Admin attempts to hard-delete a user record (e.g., via direct API call `DELETE /api/admin/users/{id}` with a `permanent=true` query parameter), **When** the API receives the request, **Then** the API returns HTTP 400 with message "Permanent deletion of user records is prohibited — use deactivation"; no `users` row is removed; this constraint is enforced at the application controller layer, independent of database-level permissions.

## Edge Cases

- What happens when an Admin tries to deactivate their own account? → The API checks if `actor_id == target_user_id`; if so, it returns HTTP 400 with message "Administrators cannot deactivate their own account"; no status update is made; this prevents accidental admin lock-out.
- What happens when an Admin updates a Staff user's role to Patient for a user with active upcoming appointments (in `Confirmed` status)? → The role change is permitted (no blocking check on appointments); an `audit_logs` entry records the role change; a warning banner is displayed on SCR-020: "This user has upcoming appointments — role change will take effect immediately on next login."
- What happens when the Admin search query contains SQL injection characters (e.g., `'; DROP TABLE users; --`)? → The API uses parameterised LINQ queries via EF Core; the input is passed as a bound parameter to `ILIKE` — no raw SQL is constructed; the query returns zero results matching the literal string; no SQL execution risk is introduced.
