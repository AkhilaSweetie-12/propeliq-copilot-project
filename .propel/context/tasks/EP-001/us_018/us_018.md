# User Story - US_018

## Story ID

- ID Format: US_018

## Story Title

- Shared Login Page with Role-Based Routing

## Description

- As an authenticated user (Patient, Staff, or Admin), I want to log in using a single shared login page so that the system routes me directly to my role-specific dashboard and rejects any API calls I make outside my permitted role boundary.

## Acceptance Criteria

1. **Given** an active, verified user on the Login screen (SCR-001), **When** they submit valid credentials via `POST /api/auth/login`, **Then** the API verifies the BCrypt password hash, issues a 15-minute JWT access token and a rotating 7-day refresh token stored in Upstash Redis (`SETEX refresh_token:{user_id}`), inserts an `audit_logs` entry with `action_type = UserLoggedIn`, and returns HTTP 200 with `{access_token, refresh_token}`; the React SPA inspects the JWT role claim and navigates to SCR-004 (Patient), SCR-011 (Staff), or SCR-019 (Admin) accordingly.

2. **Given** a user submits invalid credentials on the Login screen, **When** the API processes `POST /api/auth/login`, **Then** the API returns HTTP 401 with the generic message "Invalid credentials" — without disclosing which field (email or password) is incorrect — and the SPA displays that message as an error banner; no JWT is issued; no audit log entry is created for failed attempts.

3. **Given** an authenticated Patient whose JWT contains `role = Patient`, **When** the React SPA or HTTP client sends a request to any Staff-only or Admin-only endpoint (e.g., `GET /api/admin/users` or `POST /api/walk-ins`), **Then** the ASP.NET Core RBAC middleware returns HTTP 403 Forbidden; the SPA displays a generic "Access denied" message; no data is returned; the rejection is logged to Serilog at `WARN` level.

4. **Given** an authenticated Staff user whose JWT contains `role = Staff`, **When** the client sends a request to any Admin-only endpoint (e.g., `DELETE /api/admin/users/{id}`), **Then** the RBAC middleware returns HTTP 403 Forbidden; the same access-denial flow applies as in AC-3.

5. **Given** a user's 15-minute JWT access token has expired, **When** the client sends a request to any protected endpoint with the expired token, **Then** the API returns HTTP 401 with `WWW-Authenticate: Bearer error="token_expired"`; the client automatically uses the stored refresh token (`POST /api/auth/refresh`) to obtain a new access token and retries the original request transparently — the user does not see a login prompt unless the refresh token is also expired.

6. **Given** a user explicitly signs out via the "Sign out" button, **When** the client sends `POST /api/auth/logout`, **Then** the API adds the current JWT to the Redis blocklist (`SETEX blocklist:{jti}` with TTL matching remaining token lifetime) and deletes the refresh token key; all subsequent requests using the old JWT return HTTP 401; the SPA navigates to the Login screen (SCR-001).

## Edge Cases

- What happens when a user attempts to log in with an account that has `status = Inactive` (deactivated by Admin)? → The API returns HTTP 401 with message "Invalid credentials" (same generic message — no disclosure of account status); no JWT is issued; the login attempt is logged to Serilog at `WARN` level with reason `AccountInactive`.
- What happens when two browser tabs are open and the user signs out in one tab? → The Redis blocklist invalidates the JWT globally; the second tab's next API request returns HTTP 401; the SPA intercepts the 401 in its Axios interceptor and navigates both tabs to the Login screen (SCR-001) on the next request.
- What happens when the Upstash Redis instance is temporarily unavailable during logout? → The API returns HTTP 503 to the client; the sign-out fails gracefully with an error toast "Could not sign out — try again"; the current JWT remains valid until its natural 15-minute expiry; no silent security bypass is introduced.
