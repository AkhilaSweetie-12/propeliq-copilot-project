# User Story - US_021

## Story ID

- ID Format: US_021

## Story Title

- Audit Log Middleware & Admin Audit Log Viewer

## Description

- As an Admin user, I want every create, update, and delete action on patient and appointment records to be automatically captured in an immutable audit log, and I want to view that log in a read-only, filterable viewer so that I can investigate any data mutation for compliance, security review, or dispute resolution.

## Acceptance Criteria

1. **Given** an ASP.NET Core request pipeline with the Audit Middleware registered, **When** any controller action tagged with `[AuditAction]` successfully completes a create, update, or delete mutation on a PHI entity (patient records, appointment records), **Then** the middleware inserts one row into `audit_logs` with all of: `actor_id` (from JWT claim), `actor_role` (from JWT claim), `action_type` (e.g., `PatientCreated`, `AppointmentUpdated`, `UserDeactivated`), `entity_type` (e.g., `Patient`, `Appointment`, `User`), `entity_id` (the affected record UUID), `change_summary` (human-readable description, max 4,000 characters), `ip_address` (resolved from `HttpContext.Connection.RemoteIpAddress`), and `occurred_at` (defaulted by PostgreSQL `NOW()`); the INSERT uses a fire-and-forget approach (no await on the write) so that audit log latency does not add to the API response time.

2. **Given** the Audit Middleware is active, **When** a request results in a 4xx or 5xx response (i.e., the mutation was not actually applied), **Then** NO audit log entry is written; audit entries are only written for successfully completed mutations (2xx responses); this prevents phantom entries from appearing in the audit log.

3. **Given** an Admin on the Audit Log Viewer screen (SCR-021), **When** the page loads and the API processes `GET /api/admin/audit-logs?page=1&pageSize=50` (default), **Then** the API returns a paginated list of audit log entries ordered by `occurred_at DESC`; the SPA renders a read-only table with columns: Timestamp, Actor, Action, Resource (entity_type badge), Details (change_summary), IP Address; no edit or delete controls are shown anywhere on the screen.

4. **Given** an Admin on SCR-021, **When** they apply any combination of filters (date range, actor name, action_type, entity_type) and the API processes `GET /api/admin/audit-logs?from=X&to=Y&actor=Z&action=A&entity=E`, **Then** the API executes the corresponding WHERE clauses using parameterised queries; the filtered results are returned paginated with `X-Total-Count` header; the SPA updates the displayed rows without full-page reload; the existing composite index on `(entity_type, occurred_at DESC)` (from us_015) is used.

5. **Given** the Audit Middleware is processing a request where `change_summary` content exceeds 4,000 characters, **When** the middleware constructs the audit entry, **Then** the summary is truncated at 4,000 characters with `"...[truncated]"` appended; the full payload is logged to Serilog at `DEBUG` level for diagnostic purposes; the audit row is inserted normally.

6. **Given** an Admin attempts to call any API endpoint that would mutate the `audit_logs` table (e.g., `DELETE /api/admin/audit-logs/{id}` or `PUT /api/admin/audit-logs/{id}`), **Then** the API returns HTTP 405 Method Not Allowed; no such routes are registered in the Admin router; the application-layer guard defined in us_015 (`InvalidOperationException` on Remove/Update) provides a secondary defence; the immutability of the audit log is preserved.

## Edge Cases

- What happens when the audit log INSERT fails (e.g., database timeout) after the primary mutation has already succeeded? → The Audit Middleware catches the INSERT exception, logs it to Serilog at `ERROR` level with the full payload, and does NOT roll back the primary mutation (audit write uses a separate DbContext transaction); a Hangfire retry job is scheduled to re-attempt the audit INSERT up to 3 times within 5 minutes; if all retries fail, an alert is raised via Seq dashboard.
- What happens when a high volume of mutations fires simultaneously and the audit log INSERT queue is congested? → The fire-and-forget approach uses `Task.Run(() => _auditContext.AuditLogs.Add(entry).SaveChangesAsync())` with a bounded channel; if the channel is full, excess inserts are held in-memory briefly and flushed in the next processing cycle; the primary API response is never blocked; channel overflow is logged to Serilog at `WARN` level.
- What happens when the `ip_address` field cannot be resolved (e.g., reverse proxy misconfiguration)? → The middleware records `ip_address = "unknown"` with a `WARN` log entry; this does not block the mutation or the audit write; the entry is still inserted with all other fields fully populated.
