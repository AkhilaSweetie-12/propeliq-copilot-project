# User Story - US_038

## Story ID

- ID Format: US_038

## Story Title

- Microsoft Outlook Calendar Sync for Confirmed Appointments with Graceful Degradation

## Description

- As a patient who uses Microsoft Outlook, I want the system to automatically create an Outlook Calendar event when I confirm an appointment, so that it appears in my calendar without manual entry — and if the sync fails, I want to be notified by a non-blocking toast so that my confirmed booking is never put at risk.

## Acceptance Criteria

1. **Given** a patient's appointment booking transaction commits (HTTP 201 from `POST /api/appointments/book`), **When** the API enqueues jobs, **Then** a `OutlookCalendarSyncJob(appointment_id)` is enqueued via Hangfire immediately after the commit, independently of the `GoogleCalendarSyncJob` from us_037; the same job is enqueued for walk-in bookings (via us_033) when the patient has an email address that can be associated with a Microsoft account; the booking transaction does NOT wait for the calendar sync job to complete before returning 201 to the SPA (NFR-014, TR-014).

2. **Given** the `OutlookCalendarSyncJob` runs for a confirmed appointment, **When** the job calls the Microsoft Graph API, **Then** the job uses the configured OAuth 2.0 application permissions or delegated token (stored in environment secrets — never in source code); constructs an `Event` object matching the Microsoft Graph API `Event` resource schema with: `subject = "Appointment at PropelIQ Health"`, `start.dateTime = slot_datetime (UTC ISO 8601)`, `end.dateTime = slot_datetime + duration_minutes (UTC ISO 8601)`, `start.timeZone = "UTC"`, `end.timeZone = "UTC"`, `body.content` containing provider name and appointment type (HTML format), `location.displayName = clinic_address`; calls `POST /me/events` or `POST /users/{id}/events`; on a successful HTTP 201 from Graph the job updates `appointments SET calendar_event_id_outlook = event_id` and inserts `audit_logs (action=OutlookCalendarSynced, appointment_id)` (FR-023, TR-013).

3. **Given** the Microsoft Graph API returns a transient error (HTTP 429, 500, 503), **When** the `OutlookCalendarSyncJob` catches the exception, **Then** Hangfire retries the job with exponential backoff: first retry after 15 s, second after 30 s, third after 60 s (maximum 3 retries per TR-013); after the third failed attempt the job transitions to the Hangfire `Failed` state; `audit_logs (action=CalendarSyncFailed, provider=Microsoft, appointment_id, error_message)` is written; the appointment record remains `status=Confirmed` — the calendar sync failure does NOT cancel or alter the booking (NFR-014).

4. **Given** all 3 retry attempts for the `OutlookCalendarSyncJob` have failed, **When** the SPA is displaying SCR-010 (Appointment Confirmation), SCR-008 (Booking Calendar), or SCR-014 (Same-Day Queue View), **Then** a non-blocking warning toast is displayed at bottom-right: "Outlook Calendar sync unavailable. Your appointment is still confirmed."; the toast auto-dismisses after 5 seconds; a manual dismiss (×) button is available; the toast does not block any user action; the patient can still click "Add to calendar" manually from SCR-010 (Pattern 5, UXR-602 from IA interaction pattern, NFR-014).

5. **Given** an appointment is updated (e.g., slot swap via us_028) after the initial Outlook Calendar sync, **When** the swap transaction commits, **Then** a new `OutlookCalendarUpdateJob(appointment_id)` is enqueued; the job calls `PATCH /me/events/{id}` with the updated `start.dateTime` and `end.dateTime`; if `calendar_event_id_outlook IS NULL` (i.e., the original sync failed), the job falls back to `POST /me/events` to create the event from scratch; the same retry policy (3 retries, exponential backoff) and failure toast apply as in AC-3 and AC-4.

## Edge Cases

- What happens if the patient does not have a Microsoft account or has not granted calendar access? → The Microsoft Graph API returns HTTP 403 (Forbidden) or HTTP 404 (not found); the `OutlookCalendarSyncJob` treats these as permanent (non-retriable) failures by inspecting the HTTP status: 4xx responses (except 429) are logged as `CalendarSyncFailed_Permanent` and the job completes without retry; the booking remains confirmed; no toast is shown for a 403/404 since the patient has no expectation of sync without granting access; this mirrors the same pattern as Google (us_037) for consistency.
- What happens if Microsoft Graph API's OAuth 2.0 token expires during a sync job? → The Microsoft.Graph SDK handles token refresh automatically via its `IAuthenticationProvider` chain; if the refresh fails (e.g., token revoked), the exception is caught, `audit_logs (action=CalendarSyncFailed_TokenExpired, provider=Microsoft)` is inserted, and the job transitions to Hangfire `Failed` state; the booking is not affected; the error is surfaced in the Hangfire dashboard for Admin review.
- What happens if both `OutlookCalendarSyncJob` and `GoogleCalendarSyncJob` fail for the same appointment? → Each job writes its own `audit_logs` row independently; both failure toasts may appear concurrently on SCR-010; the SPA renders them as two separate stacked toasts (bottom-right), each with its own 5-second timer and dismiss button; the patient sees both messages but the booking confirmation banner remains visible and the 201 status is unaffected.
- What happens if two `OutlookCalendarSyncJob` instances run concurrently for the same appointment? → The job checks `appointments.calendar_event_id_outlook` under a PostgreSQL advisory lock before calling the API; if the field is already populated, the second job logs a no-op and exits without calling Microsoft Graph; this prevents duplicate calendar events on the patient's Outlook calendar, mirroring the same idempotency pattern as Google (us_037).
- What happens if the Microsoft Graph `Retry-After` header specifies a delay longer than Hangfire's calculated backoff? → The job reads the `Retry-After` response header on a 429 response; if the header value exceeds the next scheduled backoff interval, the job reschedules itself using the `Retry-After` value (capped at 120 seconds maximum) instead of the default exponential interval; this prevents wasted retry calls before Graph is ready to accept them.
