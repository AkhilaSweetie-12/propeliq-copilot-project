# User Story - US_037

## Story ID

- ID Format: US_037

## Story Title

- Google Calendar Sync for Confirmed Appointments with Graceful Degradation

## Description

- As a patient who confirms an appointment, I want the system to automatically create a Google Calendar event for my appointment, so that it appears in my Google Calendar without manual entry — and if the sync fails, I want to be informed by a non-blocking toast so that my confirmed booking is never put at risk.

## Acceptance Criteria

1. **Given** a patient's appointment booking transaction commits (HTTP 201 from `POST /api/appointments/book`), **When** the API enqueues jobs, **Then** a `GoogleCalendarSyncJob(appointment_id)` is enqueued via Hangfire immediately after the commit; the same job is enqueued for walk-in bookings (via us_033) when the patient has an email address that can be associated with a Google OAuth 2.0 account; the booking transaction does NOT wait for the calendar sync job to complete before returning 201 to the SPA (NFR-014, TR-014).

2. **Given** the `GoogleCalendarSyncJob` runs for a confirmed appointment, **When** the job calls the Google Calendar API v3, **Then** the job uses the configured OAuth 2.0 service account or user token (stored in environment secrets — never in source code); constructs a `CalendarEvent` object with: `summary = "Appointment at PropelIQ Health"`, `start.dateTime = slot_datetime`, `end.dateTime = slot_datetime + duration_minutes`, `location = clinic_address`, `description` containing provider name and appointment type; calls `events.insert` on the patient's calendar; on a successful HTTP 200/201 from Google the job updates `appointments SET calendar_event_id_google = event_id` and inserts `audit_logs (action=GoogleCalendarSynced, appointment_id)` (FR-022, TR-013).

3. **Given** the Google Calendar API returns a transient error (HTTP 429, 500, 503), **When** the `GoogleCalendarSyncJob` catches the exception, **Then** Hangfire retries the job with exponential backoff: first retry after 15 s, second after 30 s, third after 60 s (maximum 3 retries per TR-013); after the third failed attempt the job transitions to the Hangfire `Failed` state; `audit_logs (action=CalendarSyncFailed, provider=Google, appointment_id, error_message)` is written; the appointment record remains `status=Confirmed` — the calendar sync failure does NOT cancel or alter the booking (NFR-014).

4. **Given** all 3 retry attempts for the `GoogleCalendarSyncJob` have failed, **When** the SPA is displaying SCR-010 (Appointment Confirmation), SCR-008 (Booking Calendar), or SCR-014 (Same-Day Queue View), **Then** a non-blocking warning toast is displayed at bottom-right: "Google Calendar sync unavailable. Your appointment is still confirmed."; the toast auto-dismisses after 5 seconds; a manual dismiss (×) button is available; the toast does not block any user action; the patient can still click "Add to calendar" manually from SCR-010 (Pattern 5, UXR-602 from IA interaction pattern, NFR-014).

5. **Given** an appointment is updated (e.g., slot swap via us_028) after the initial Google Calendar sync, **When** the swap transaction commits, **Then** a new `GoogleCalendarUpdateJob(appointment_id)` is enqueued; the job calls `events.patch` on the existing `calendar_event_id_google` with the updated `start.dateTime` and `end.dateTime`; if `calendar_event_id_google IS NULL` (i.e., the original sync failed), the job falls back to `events.insert` to create the event from scratch; the same retry policy (3 retries, exponential backoff) and failure toast apply as in AC-3 and AC-4.

## Edge Cases

- What happens if the patient does not have a Google account or has not granted calendar access? → The Google Calendar API returns HTTP 403 (forbidden) or HTTP 404 (calendar not found); the `GoogleCalendarSyncJob` treats these as permanent (non-retriable) failures by inspecting the HTTP status code: 4xx responses (except 429) are logged as `CalendarSyncFailed_Permanent` and the job completes without retry; the booking remains confirmed; no toast is shown for a 403/404 since the patient has no expectation of sync without granting access.
- What happens if the Google Calendar API OAuth 2.0 token expires during a sync job? → The Google.Apis.Calendar.v3 SDK handles token refresh automatically via the stored refresh token; if the refresh also fails (e.g., token revoked), the job logs `CalendarSyncFailed_TokenExpired` and transitions to the Hangfire `Failed` state; the booking is not affected; the error is surfaced in the Hangfire dashboard for Admin review.
- What happens if two `GoogleCalendarSyncJob` instances run concurrently for the same appointment (e.g., a duplicate job enqueue bug)? → The job checks `appointments.calendar_event_id_google` under a PostgreSQL advisory lock before calling the API; if the field is already populated, the second job logs a no-op and exits without calling the Google API; this prevents duplicate calendar events on the patient's calendar.
- What happens if the Hangfire PostgreSQL job store is unreachable when the booking transaction commits? → The `GoogleCalendarSyncJob` enqueue call is wrapped in a try-catch; if enqueue fails, the exception is caught and logged to `audit_logs (action=CalendarJobEnqueueFailed)`; the booking HTTP 201 response is still returned to the SPA — the booking is committed regardless of job enqueue failure; the calendar sync can be retried manually from the admin interface or a subsequent booking event.
