# User Story - US_036

## Story ID

- ID Format: US_036

## Story Title

- SMS & Email Appointment Reminders via Hangfire CRON

## Description

- As a patient with a confirmed appointment, I want to receive an SMS and an email reminder at 48 hours and again at 2 hours before my appointment, so that I am prompted in advance and can take action to avoid missing my appointment.

## Acceptance Criteria

1. **Given** a Hangfire recurring CRON job (`AppointmentReminderJob`) is registered to run every 15 minutes, **When** the job fires, **Then** it executes two queries against PostgreSQL: `SELECT appointments JOIN users WHERE slot_datetime BETWEEN NOW()+48h AND NOW()+49h AND status=Confirmed` (48-hour batch) and `SELECT appointments JOIN users WHERE slot_datetime BETWEEN NOW()+2h AND NOW()+2h15m AND status=Confirmed` (2-hour batch); appointments already notified in the current window (i.e., with a `notifications` row for the same `channel + appointment_id + scheduled_at_window`) are skipped to prevent duplicate delivery; this idempotency check uses the `notifications` entity (TR-014).

2. **Given** an appointment falls within the 48-hour or 2-hour reminder window, **When** the `AppointmentReminderJob` processes it, **Then** for each appointment the job inserts `notifications (channel=SMS, status=Pending, appointment_id, scheduled_at=NOW())`; calls the Twilio SMS API (`POST /Messages`) using the patient's registered `phone` number and appointment date/time/provider in the message body; the Twilio Account SID and Auth Token are read from environment secrets and MUST NOT be embedded in source code (TR-012); on successful Twilio acknowledgement the job updates `notifications SET status=Sent, sent_at=NOW()`; on failure updates `notifications SET status=Failed, error_message, attempt_count=1` and inserts `audit_logs (action=SMSDeliveryFailed, appointment_id, error)` (FR-020, NFR-012).

3. **Given** an SMS delivery fails, **When** the job writes the failure to the `notifications` table, **Then** Hangfire enqueues a `NotificationRetryJob(notification_id)` with a minimum 5-minute delay (NFR-012); the retry job re-reads the notification record, re-attempts the Twilio call once; if the retry also fails, the `notifications.status` transitions to `PermanentFail` and `audit_logs (action=SMSPermanentFailure, appointment_id)` is written; no further automatic retries are made after the first retry attempt.

4. **Given** the same appointment is within the reminder window, **When** the job processes the email reminder step, **Then** the job inserts `notifications (channel=Email, status=Pending, appointment_id)`; constructs an HTML+plain-text reminder email using MailKit with the configurable SMTP relay (Gmail SMTP or SMTP2GO free tier, configured via environment variable `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`); the email body includes appointment date, time, provider name, and a "Add to calendar" link; the SMTP credentials are read from environment secrets — never hardcoded (TR-011, FR-021); on delivery success updates `notifications SET status=Sent`; on failure applies the same retry pattern as SMS (NFR-012).

5. **Given** a patient has no registered phone number (`users.phone IS NULL`), **When** the SMS reminder step runs for that patient's appointment, **Then** the SMS step is skipped without error; `notifications (channel=SMS, status=Skipped, reason=NoPhoneNumber)` is inserted; the email reminder step continues unaffected; `audit_logs (action=SMSSkipped_NoPhone, appointment_id)` is written; the absence of a phone number does not prevent email delivery or fail the overall `AppointmentReminderJob`.

## Edge Cases

- What happens if the same appointment falls in both the 48-hour and 2-hour query windows simultaneously (e.g., due to scheduler drift)? → The idempotency check in AC-1 prevents double notification: the job checks for an existing `notifications` row with `channel=SMS, appointment_id=X, scheduled_at >= window_start`; if one exists and its `status` is not `Failed`, the appointment is skipped for that batch; this ensures at-most-once delivery per reminder window per channel.
- What happens if the Hangfire PostgreSQL job store is unavailable and the CRON job fails to fire? → The CRON job will be retried by Hangfire on the next tick when the database recovers; Hangfire stores job state in PostgreSQL so no reminders are permanently lost; a monitoring alert (out of scope for this story) would surface prolonged job store unavailability; booking and queue operations are not impacted by the scheduler being down (NFR-014).
- What happens if an appointment is cancelled between when the reminder is queued and when it is delivered? → The `NotificationRetryJob` and `AppointmentReminderJob` both validate `appointments.status = Confirmed` before sending; if the status is `Cancelled` or `NoShow`, the send step is skipped and `notifications SET status=Cancelled_Appointment` is written; no SMS or email is sent to the patient.
- What happens if Twilio's free trial balance runs out? → Twilio returns an HTTP 402 error; the job treats this as a delivery failure — updates `notifications SET status=Failed, error_message="Twilio balance exhausted"`; retries once after 5 minutes (same as any other failure); if the retry also returns 402, transitions to `PermanentFail`; the job does not crash — other appointments in the same batch continue to be processed.
