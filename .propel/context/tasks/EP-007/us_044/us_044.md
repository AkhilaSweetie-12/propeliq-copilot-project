# User Story - US_044

## Story ID

- ID Format: US_044

## Story Title

- AI Code Suggestion Pipeline: ICD-10 & CPT RAG Generation

## Description

- As the clinical intelligence system, I want to generate ICD-10 diagnostic and CPT procedure code suggestions from a verified 360-Degree Patient View using a RAG pipeline — embedding the patient summary, searching against pre-indexed reference vector stores, and using the local LLM to confirm the top candidates with supporting source evidence — so that Staff can receive accurate, traceable code suggestions within 15 seconds at p90 without any PHI leaving the platform boundary.

## Acceptance Criteria

1. **Given** a Staff user calls `POST /api/patients/{id}/coding/request`, **When** the API handles the request, **Then** the API checks `patient_view_360 WHERE patient_id=X AND is_verified=true`; if the view exists and is verified, the API enqueues `CodingJob(patient_id)` in Hangfire and returns HTTP 202 Accepted; if `is_verified=false` or no `patient_view_360` row exists, the API returns HTTP 400 `{ "error": "PatientViewNotVerified", "message": "360° View must be verified before coding can be requested" }` with no job enqueued; `audit_logs (action=CodingRequested, actor_id=staff_id, patient_id)` is written on 202 (FR-031, FR-032, UC-008 step 1).

2. **Given** `CodingJob(patient_id)` is dequeued by Hangfire, **When** the ICD-10 pipeline runs, **Then** the job reads `patient_view_360.aggregated_data` (decrypted) and constructs a summarised patient context string (diagnoses, medications, allergies, vitals — max 3,072 tokens to leave 1,024-token headroom for LLM prompt + candidates); the AI Guard Layer applies PII redaction (patient name → `[PATIENT-NAME]`, DOB → `[DOB-REDACTED]`, address → `[ADDR-REDACTED]`) before any LLM invocation; the redacted summary is embedded via `nomic-embed-text` (`POST http://localhost:11434/api/embeddings`); the embedding is used to perform a cosine similarity search against the pre-indexed ICD-10 reference vector store in pgvector (`embedding <=> query_vector` cosine distance, `LIMIT 5`); only candidates with `score >= 0.65` are passed to the LLM; the LLM (`llama3.2:3b-instruct-q8_0` via Ollama REST API) receives the candidates + patient context and returns a JSON array `[{ code, description, source_evidence, confidence }]`; each code is validated against the ICD-10 JSON schema; codes passing validation are inserted into `medical_code_suggestions` per-code row (AIR-004, FR-031).

3. **Given** the ICD-10 pipeline has completed, **When** the CPT pipeline runs, **Then** the same RAG pattern as AC-2 is applied: embed the same PII-redacted patient context → cosine similarity search against the pre-indexed CPT reference vector store (separate pgvector index for CPT codes) → retrieve top-5 candidates with `score >= 0.65` → LLM confirm/refine CPT codes with source evidence → JSON schema validation → INSERT `medical_code_suggestions` per CPT code; the ICD-10 and CPT pipelines run sequentially within the same `CodingJob` to share the same patient context embedding — the embedding is computed once and reused for both queries (AIR-005, FR-032).

4. **Given** the LLM returns code suggestions, **When** each code is persisted, **Then** the job inserts one `medical_code_suggestions` row per code with: `patient_id`, `code` (ICD-10 or CPT string), `code_type` (ICD10|CPT), `description`, `source_evidence` (the clinical text excerpt from the retrieved chunks that justifies the code), `confidence` (0.0–1.0 from LLM response), `status=Pending`, `needs_review=false`; **IF** confidence < 0.6 → `needs_review=true`; `model_name=llama3.2:3b-instruct-q8_0`, `model_version=Ollama 0.5` are stored in every row regardless of confidence level; `created_at=NOW()`; `reviewed_by=null`, `final_code=null`, `finalised=false`; `audit_logs (action=CodeSuggestionGenerated, patient_id, code, code_type, confidence)` is written per code (AIR-O03, AIR-004, AIR-005).

5. **Given** the `CodingJob` completes (both ICD-10 and CPT suggestions inserted), **When** Staff calls `GET /api/patients/{id}/coding/suggestions`, **Then** the API returns HTTP 200 with the full suggestion list: `[{ suggestion_id, code, code_type, description, source_evidence, confidence, needs_review, status, model_name, model_version }]`; the response is ordered: ICD-10 codes first (sorted by confidence DESC), then CPT codes (sorted by confidence DESC); if no suggestions exist yet (job still running), the API returns HTTP 202 with `{ "status": "generating" }` so the React SPA can poll; if the job failed, returns HTTP 200 with `{ "status": "failed", "suggestions": [] }`; the entire `CodingJob` (embed → retrieve ICD-10 → retrieve CPT → LLM × 2 → validate × N → INSERT × N) MUST complete within 15 seconds at p90 (NFR-004).

## Edge Cases

- What happens if the LLM returns confidence < 0.6 for all ICD-10 and CPT codes? → All suggestions are inserted with `needs_review=true`; the job completes normally (not a failure); the GET endpoint returns all suggestions with `needs_review=true`; Staff can manually review each suggestion and accept/modify even the low-confidence ones on SCR-018; this is not a job failure — it is expected for clinically ambiguous patient summaries where the aggregated data lacks diagnostic specificity.
- What happens if the ICD-10 or CPT reference vector store returns 0 candidates above the 0.65 cosine similarity threshold? → For that code type (ICD-10 or CPT), no LLM invocation is made; no suggestions are inserted for that code type; `audit_logs (action=NoCandidatesAboveThreshold, patient_id, code_type=ICD10|CPT)` is written; the GET endpoint returns an empty array for that code type; Staff sees an empty state on SCR-018 for the affected table and can enter codes manually; the other code type still completes normally if it has candidates.
- What happens if the Ollama embedding or LLM call fails mid-job? → The `CodingJob` retries the failing call once (immediate retry); if the second attempt also fails, the job transitions to Hangfire `Failed` state; `audit_logs (action=CodingJobFailed, patient_id, step=Embedding|LLMInference, error_message)` is written; the GET endpoint returns `{ "status": "failed", "suggestions": [] }`; Staff sees an error state on SCR-018 with a "Retry code generation" button; partial suggestions from the completed code type (e.g., ICD-10 succeeded, CPT failed) are retained in `medical_code_suggestions` and returned by GET.
- What happens if the total `CodingJob` elapsed time approaches the 15-second p90 budget? → The job monitors elapsed time from Hangfire dequeue; if ICD-10 pipeline has been processed and the elapsed time exceeds 12 seconds, the CPT pipeline is skipped for this run; CPT suggestions are not generated in this execution; `audit_logs (action=CodingBudgetExceeded, patient_id, skipped=CPT)` is written; the GET response returns only ICD-10 suggestions; Staff can re-trigger via "Request code suggestions" on SCR-018 to generate CPT suggestions in a subsequent job; NFR-004 (15s p90) is maintained at the cost of incomplete CPT results on first run.
- What happens if `patient_view_360.aggregated_data` is empty (all fields were flagged `_NeedsReview` in EP-006-II) and Staff somehow calls the coding request API? → The constructed patient context string is empty; the embedding query produces a near-zero vector; cosine similarity results are unreliable; the job detects empty aggregated_data and returns HTTP 400 at the API level with `{ "error": "InsufficientPatientData", "message": "No verified patient data available — ensure patient documents are uploaded and extracted before requesting code suggestions" }`; no Hangfire job is enqueued; the Staff UI shows this 400 message and links to SCR-015 (document upload).
