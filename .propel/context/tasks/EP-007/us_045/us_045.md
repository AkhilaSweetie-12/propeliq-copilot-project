# User Story - US_045

## Story ID

- ID Format: US_045

## Story Title

- Medical Coding Panel UI: Code Review, Source Evidence & Finalisation

## Description

- As a Staff clinician reviewing a patient's AI-generated medical codes, I want to view each suggested ICD-10 and CPT code alongside its collapsible source evidence, accept, modify or reject each suggestion individually, and finalise the code set — with the AI-Human Agreement Rate displayed after finalisation — so that I can confidently verify AI-suggested codes before they are submitted for billing, with full traceability to the clinical text that justifies each code.

## Acceptance Criteria

1. **Given** an authenticated Staff user opens SCR-018 (Medical Coding Panel) from the patient's 360° view, **When** the page loads, **Then** the SPA calls `POST /api/patients/{id}/coding/request` (if no suggestions exist for this patient) or `GET /api/patients/{id}/coding/suggestions` (if suggestions already generated); while suggestions are being generated the page renders a skeleton loading state with "Generating code suggestions…" subtitle (UXR-502); once `GET` returns HTTP 200 with suggestions, the page renders two separate tables: ICD-10 Diagnosis Codes table (sorted by confidence DESC) and CPT Procedure Codes table (sorted by confidence DESC); each row contains: Code, Description, AI Confidence (visual progress bar — green ≥ 80%, amber 60–79%, red < 60%), Status badge (Pending / Accepted / Modified / Rejected / Needs Review), and Actions (View source, Accept, Edit, Reject); if `GET` returns `{ "status": "generating" }`, the SPA polls every 3 seconds until HTTP 200 with suggestions arrives or a 30-second polling timeout elapses; if the job failed, a full-page error state with "Retry" CTA is shown (UXR-603, FR-033, UC-008 step 4).

2. **Given** code suggestions are loaded on SCR-018, **When** the Staff views the table, **Then** codes with `needs_review=false` are rendered with an "AI Suggested" badge (amber) and muted background (`--neutral-100`) per the Trust-First visual hierarchy (UXR-403); codes with `needs_review=true` (confidence < 0.6) are rendered with a "Needs Review" badge (amber, distinct style) and a muted yellow background (`--color-warning-50`); all unreviewed rows display Accept and Reject action buttons; no auto-acceptance occurs — all codes start in `Pending` status and require explicit Staff action; the visual distinction between AI-suggested and Needs Review states MUST use both colour and label text (not colour alone) for colour-blind accessibility (UXR-403, UXR-201, FR-033).

3. **Given** the Staff views a code row, **When** they click "View source" (or "Source") on any row, **Then** a right-side Source Evidence drawer (420px wide, `role="complementary"`, `aria-label="Source evidence for code [CODE]"`) slides in displaying: code label + description in the drawer header, the supporting clinical text excerpts as visually highlighted quotes (amber left-border, `color.semantic.ai.bg` background), each quote's source document name + page number, and an AI reasoning paragraph summarising why the LLM selected this code; the drawer is collapsed by default when the page loads (UXR-104); clicking `×` or pressing Escape closes the drawer and returns focus to the "View source" trigger button; only one drawer is open at a time — opening a second drawer closes the first; the drawer content is fetched from the `source_evidence` field already returned by the GET API (no additional API call needed) (UXR-104, FR-033).

4. **Given** the Staff reviews a code suggestion, **When** they take a review action, **Then**:
   - **Accept**: Staff clicks "Accept" → SPA calls `PATCH /api/coding/suggestions/{suggestion_id} { status: "Accepted" }` → on 200 OK the row badge transitions to "Accepted" (green solid badge + green row tint); Accept/Reject buttons are replaced by an "Edit" ghost button; `audit_logs (action=CodeReviewed, actor_id=staff_id, patient_id, code, status=Accepted)` written
   - **Modify**: Staff clicks "Edit" (or when reviewing, clicks "Modify") → an inline edit input pre-filled with the current code value opens in the Code cell; Staff enters the corrected code string and clicks "Save" → SPA calls `PATCH /api/coding/suggestions/{suggestion_id} { status: "Modified", final_code: "[staff_code]" }` → on 200 OK the row shows the Staff-entered code with a "Modified" blue badge; `audit_logs (action=CodeReviewed, status=Modified, final_code=staff_code)` written
   - **Reject**: Staff clicks "Reject" → SPA calls `PATCH /api/coding/suggestions/{suggestion_id} { status: "Rejected" }` → on 200 OK the row is visually struck-through with a "Rejected" red badge; an optional "Enter replacement code" input appears for Staff to supply a manual replacement code; `audit_logs (action=CodeReviewed, status=Rejected)` written
   - Each PATCH action persists immediately; button press feedback is ≤ 200ms from click (UXR-501); all three actions are UC-008 loop steps (UC-008 steps 5a loop).

5. **Given** all code suggestions have been reviewed (no rows in `Pending` status), **When** Staff clicks "Finalise & submit codes", **Then** a confirmation modal appears: "Finalise codes for [patient name]? This will record your final code selections." with "Confirm" and "Cancel" buttons; Staff clicks "Confirm" → SPA calls `POST /api/patients/{id}/coding/finalise` → on 200 OK `{ agreement_rate: X% }` the panel shows a success banner: "Codes finalised — AI Agreement Rate: X%" (green, persists on screen); all row Action buttons become disabled (read-only state); `audit_logs (action=CodingFinalised, actor_id=staff_id, patient_id, agreement_rate=X%)` is written; the AI Agreement Rate (percentage of suggestions accepted without modification = Accepted / (ICD-10 + CPT total)) is computed server-side and returned; the "Finalise & submit codes" button is disabled (grey, tooltip: "Review all code suggestions before finalising") while any row remains in `Pending` status (AIR-Q01, FR-033, UC-008 step 6–7).

## Edge Cases

- What happens if Staff tries to finalise while one or more rows are still in `Pending` status? → The "Finalise & submit codes" button remains disabled and non-interactive; a tooltip "Review all code suggestions before finalising" is shown on hover; no API call is made; Staff must action (Accept/Modify/Reject) every row before the button becomes active; this ensures the Agreement Rate metric is computed on a complete review, not a partial one.
- What happens if Staff navigates away from SCR-018 mid-review (some codes accepted, some still Pending)? → Each PATCH is persisted immediately on click — there is no batch/draft mode; when Staff returns to SCR-018 and calls `GET /api/patients/{id}/coding/suggestions`, the response reflects the saved review state; already-accepted/rejected rows show their committed status; Pending rows remain actionable; progress is not lost; the "Finalise & submit codes" button remains disabled until all rows are reviewed.
- What happens if the GET returns HTTP 202 `{ "status": "generating" }` and the 30-second polling timeout elapses without a 200 OK? → The SPA stops polling and renders a timeout state: "Code generation is taking longer than expected." with a "Retry" CTA button; clicking "Retry" calls `POST /api/patients/{id}/coding/request` again to re-enqueue the job; the timeout state is non-blocking — Staff can navigate away and return later when the job completes.
- What happens if a Patient role navigates to SCR-018's URL directly? → The `POST /api/patients/{id}/coding/request` and `GET /api/patients/{id}/coding/suggestions` endpoints enforce `role=Staff` or `role=Admin` RBAC; a Patient JWT receives HTTP 403 Forbidden; the React Router guard redirects the Patient to their dashboard (SCR-004) and shows a toast "You do not have permission to access this page".
- What happens if the Agreement Rate returned after finalisation is below the target 98%? → The finalisation completes normally — the 98% Agreement Rate target (AIR-Q01) is an aspirational quality metric for aggregate model performance tracking, not a runtime gate; low Agreement Rate is recorded in `medical_code_suggestions` for aggregate quality reporting; the coding session is marked finalised regardless of the rate; no blocking alert is shown to Staff; a future admin reporting screen (out of EP-007 scope) can surface sessions with low Agreement Rate for model quality analysis.
