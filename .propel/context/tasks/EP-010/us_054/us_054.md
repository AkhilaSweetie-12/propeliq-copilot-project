# User Story - US_054

## Story ID

- ID Format: US_054

## Story Title

- Navigation Architecture: 3-Click Access, Breadcrumbs & Role-Sensitive Navigation

## Description

- As a patient, staff member, or admin navigating the platform, I want every primary feature to be reachable within 3 clicks from my role-specific dashboard, contextual breadcrumbs to appear on screens deeper than the dashboard, and the navigation bar to show only the sections relevant to my role — so that I can move efficiently between tasks without getting lost, always know where I am in the application, and never encounter navigation links to sections I am not permitted to access.

## Acceptance Criteria

1. **Given** any user (Patient, Staff, or Admin) is on their role-specific dashboard (SCR-004 Patient Dashboard, SCR-011 Staff Dashboard, SCR-019 Admin Users), **When** the user's primary task flows are audited, **Then** every primary feature is reachable within 3 clicks from the dashboard for each persona's top 3 tasks: Patient top tasks — (a) Book an Appointment: Dashboard → SCR-008 Booking Calendar → SCR-009 Booking & Insurance [2 clicks]; (b) Complete Intake: Dashboard → SCR-005 Intake Method → SCR-006 AI Chat or SCR-007 Manual Intake [2 clicks]; (c) Upload Document: Dashboard → SCR-015 Documents Upload [1 click]; Staff top tasks — (a) View Queue: Dashboard → SCR-014 Queue View [1 click]; (b) View Patient 360°: Dashboard → SCR-014 Queue View → SCR-016 Patient 360° [2 clicks]; (c) Add Walk-In: Dashboard → SCR-013 Patient Search → SCR-012 Walk-In Booking [2 clicks]; Admin top tasks — (a) View Users: Dashboard (SCR-019) [already at destination]; (b) View User Detail: Dashboard → SCR-020 User Detail [1 click]; (c) View Audit Log: Dashboard → SCR-021 Audit Log [1 click]; a Playwright click-path audit script navigates each of the 9 persona task flows and asserts the destination screen is reached within 3 clicks (counting from the dashboard as click 0); the audit fails the CI check if any task flow exceeds 3 clicks (UXR-001).

2. **Given** any authenticated user is on a screen at depth ≥ 2 from the dashboard (any screen deeper than the immediate dashboard), **When** the page renders, **Then** a breadcrumb navigation component appears immediately below the topbar and above the page title, displaying the full path from the dashboard to the current screen; breadcrumb items are: an anchor `<a>` for each ancestor screen (linking to that screen's route), a `›` separator between items, and a `<span aria-current="page">` for the current screen (non-linked); on mobile (< 768px) the breadcrumb truncates to show only the immediate parent and the current screen (e.g., "Queue › Jane Smith — 360° View" rather than the full path); the breadcrumb `<nav>` element carries `aria-label="Breadcrumb"` (per UXR-202 from us_051); example breadcrumb paths: SCR-016 shows "Queue › Jane Smith — 360° View"; SCR-017 shows "Queue › Jane Smith — 360° View › Resolve conflicts"; SCR-018 shows "Queue › Jane Smith — 360° View › Medical Coding"; SCR-020 shows "Users › [User Name]"; breadcrumbs are absent on SCR-001 (Login), SCR-002 (Register), SCR-003 (Email Verification), and the three role dashboards (SCR-004, SCR-011, SCR-019) — these are depth-1 screens (UXR-002).

3. **Given** an authenticated user with a specific role (Patient, Staff, or Admin) is on any screen, **When** the navigation sidebar (≥ 768px) or hamburger drawer (< 768px, implemented in us_052) renders, **Then** the navigation items visible to each role are exactly: Patient — "Dashboard", "Book Appointment", "Intake", "Documents"; Staff — "Dashboard", "Queue", "Walk-In Booking", "Patient Search"; Admin — "Users", "Audit Log"; navigation items for sections the current user cannot access are completely absent from the DOM (not merely hidden with `display: none` or `visibility: hidden`) — they are conditionally rendered based on the `role` claim in the decoded JWT token; the active navigation item (corresponding to the current route) is visually distinguished with the `active` class (`sidebar-nav-item active` as per `wireframe-shared.css`) and carries `aria-current="page"` on its anchor; the navigation renders identically in both the sidebar (desktop/tablet) and the hamburger drawer (mobile) — both are driven by the same `<NavigationItems role={userRole}>` component (UXR-003).

4. **Given** the role-sensitive navigation is rendered for a Staff user on SCR-016 (360° Patient View), **When** the navigation sidebar lists "Patient View" as the active section, **Then** the active item shows the "Patient View" sidebar item with `aria-current="page"` and the active visual style (`background: var(--color-teal-100); border-left: 3px solid var(--color-teal-500)` per wireframe); routing between role-specific navigation items uses React Router `<Link>` components — no `<a href>` with full page reload; navigating away from the current screen does not require a browser reload (SPA navigation); the `<NavigationItems>` component unit test covers all three roles and asserts: (a) only role-appropriate items are rendered; (b) the correct item is `active` for each route; (c) no forbidden items are present in the rendered DOM for any role (UXR-003, UXR-001).

5. **Given** the navigation architecture (3-click paths, breadcrumbs, role nav) is implemented across all 21 screens, **When** the full Playwright navigation audit runs in CI, **Then** all 9 persona click-path flows from AC-1 pass (each destination reached in ≤ 3 clicks); breadcrumbs are present on all depth-≥-2 screens (SCR-005 through SCR-010 for Patient, SCR-012 through SCR-018 for Staff, SCR-020 through SCR-021 for Admin) and absent on SCR-001 through SCR-004, SCR-011, SCR-019; role isolation is confirmed — a Playwright test authenticated as a Patient asserts that no Staff or Admin navigation items appear in the DOM; a test authenticated as Staff asserts no Patient-only items (e.g., "Book Appointment") appear (UXR-001, UXR-002, UXR-003).

## Edge Cases

- What happens if a Staff user attempts to navigate directly to a Patient-only route (e.g., `/booking`) by typing the URL? → The React Router route definitions wrap Patient-only routes in a `<RequireRole roles={['Patient']} />` guard component; if a Staff user navigates to a Patient-only route, the guard redirects to the Staff dashboard (SCR-011) with a `replace` navigation (no browser history entry for the forbidden route); an `audit_logs` entry (action=UnauthorisedNavigation, actor_id=staffUserId, entity_type=route, entity_id=/booking) is written to record the access attempt; this is distinct from the API-level RBAC enforcement (us_021) — both client-side routing guards and server-side API guards are in place as defence-in-depth.
- What happens if the breadcrumb path becomes very long (e.g., 4+ levels deep) and overflows the breadcrumb bar on a narrow viewport? → The breadcrumb component applies an `overflow: hidden; text-overflow: ellipsis; white-space: nowrap` style on the full path string at widths below 768px, and independently truncates to parent + current only (per AC-2) — the two mechanisms together ensure the breadcrumb never overflows the topbar area; the `aria-label="Breadcrumb"` `<nav>` element remains fully accessible even when visually truncated, because screen readers read the full `aria-label` text of each `<a>` link regardless of truncation; the platform is designed with a maximum screen depth of 3 (dashboard → list → detail), so a 4+ level breadcrumb is not expected in Phase 1.
- What happens if a JWT token contains a role claim not in the expected set (e.g., `"role": "SuperAdmin"` added in a future sprint) and the `<NavigationItems>` component receives an unknown role? → The `<NavigationItems>` component's role prop is typed as `'Patient' | 'Staff' | 'Admin'` (a TypeScript union) — any unknown role value is caught as a TypeScript compile error; at runtime, if an unexpected role value somehow reaches the component (e.g., via a manually crafted JWT), the component renders an empty navigation list (no items) and logs a `console.error("Unknown role in navigation: [role]")` — the user sees a navigation-less authenticated shell and must log out; this fails safe (no navigation items) rather than exposing forbidden items.
- What happens if the click-path audit in AC-1 detects that a future sprint adds a new feature that is not reachable within 3 clicks from the dashboard? → The Playwright click-path audit is not limited to the 9 hardcoded task flows — it also accepts a configuration file (`.propel/config/navigation-audit.json`) listing all audited paths; adding a new screen requires updating the navigation audit config with the new path; the CI gate fails if any configured path exceeds 3 clicks, forcing the developer to add a navigation shortcut (e.g., a dashboard card) before the PR can merge; this makes the 3-click constraint a continuous architectural enforcer, not a one-time check.
- What happens if the active navigation item is wrong after a deep-link navigation (e.g., the user navigates directly to `/patients/123/coding` via a bookmark and the sidebar shows no active item)? → The `<NavigationItems>` component uses `useMatch` from React Router to determine the active item based on the current URL path — it does not depend on navigation click history; deep-linking to any route will correctly highlight the matching nav item (`/patients/:id/coding` matches the "Queue" navigation path prefix for Staff); Playwright tests in AC-5 include a direct URL navigation test (not click-through) that asserts the correct nav item is active after a `page.goto()` call.
