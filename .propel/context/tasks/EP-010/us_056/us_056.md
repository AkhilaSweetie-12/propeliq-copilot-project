# User Story - US_056

## Story ID

- ID Format: US_056

## Story Title

- Error & Empty States: Non-Critical Toasts, Full-Page Errors & Empty-State CTAs

## Description

- As a patient, staff member, or admin using the platform, I want non-critical background failures (calendar sync, SMS delivery) to surface as a brief, dismissible notification that does not block my work, critical failures to show a clear error page with a way to retry, and empty list views to present a helpful prompt instead of a blank space — so that I can distinguish between minor disruptions I can ignore and blocking errors that need attention, and so I am never stranded on an empty screen without knowing what to do next.

## Acceptance Criteria

1. **Given** a non-critical background operation fails (Google Calendar sync failure, Microsoft Outlook sync failure, or SMS delivery failure originating from EP-005 Hangfire jobs) on any screen, **When** the failure event is propagated to the frontend (via a WebSocket push event, polling of the notification endpoint, or as a non-blocking field in an API response), **Then** a toast notification renders in the bottom-right corner of the viewport (`position: fixed; bottom: var(--space-6); right: var(--space-6); z-index: 50`) containing: a warning icon (`⚠`), a concise descriptive message ("Calendar sync failed — your appointment was booked successfully", "SMS reminder could not be sent — your appointment is confirmed"), and a manual dismiss `✕` button (`aria-label="Dismiss notification"`); the toast auto-dismisses after exactly 5 seconds (via `setTimeout` — the timer resets if a second toast is stacked); the toast does NOT block interaction with any underlying content — it does not dim the background, does not intercept clicks, and does not move page content; multiple simultaneous toasts stack vertically (newest on top) with a maximum of 3 visible at once (oldest dismissed when the 4th arrives); a toast `<div>` carries `role="status"` and `aria-live="polite"` so screen readers announce the message without interrupting the user's current focus; the toast appears on SCR-008 (booking confirmation calendar), SCR-010 (appointment confirmation), and SCR-014 (Staff queue) as these are the screens where calendar and SMS failures are surfaced (UXR-602).

2. **Given** a critical API call fails (patient view load on SCR-016, queue load on SCR-014, login failure on SCR-001, or code suggestions load on SCR-018), **When** the API call returns a non-2xx response or times out after 30 seconds (configured as the `TanStack Query` retry exhaustion timeout), **Then** the screen transitions from the skeleton loading state (us_055) to a full-page error state occupying the main content area; the full-page error state contains: (a) a descriptive error heading — "Patient view unavailable" (SCR-016), "Queue could not be loaded" (SCR-014), "Login failed" (SCR-001), "Code suggestions unavailable" (SCR-018); (b) a supporting sentence explaining the situation without technical jargon: "We couldn't load this data. This is usually temporary — please try again."; (c) a primary "Try again" button that re-triggers the failed API call (TanStack Query `refetch()` call); (d) a secondary "Go to dashboard" link that navigates to the user's role-specific dashboard; the error state does not expose HTTP status codes, stack traces, or internal error identifiers to the user; the page `<title>` is updated to reflect the error (e.g., "Error — Queue | PropelIQ Health") for browser tab clarity; on SCR-001 (Login failure), the error is an inline form-level error (not full-page) showing "Invalid email or password" beneath the submit button — the full-page error pattern applies to data-loading screens, not authentication forms (UXR-603).

3. **Given** a list or data view has no records to display — no appointments booked (SCR-008 booking calendar month view), no documents uploaded (SCR-015 Documents), no users found matching an admin search (SCR-019 Users list), no code suggestions generated yet (SCR-018 Coding Panel), no audit log entries matching a filter (SCR-021 Audit Log), or no queue entries for today (SCR-014 Queue View), **When** the API call completes successfully but returns an empty dataset, **Then** the empty state renders in place of the list/table/grid with: (a) a contextual illustration (SVG icon, not a photograph) relevant to the content type — calendar illustration for no appointments, document illustration for no documents, people illustration for no users, medical-chart illustration for no code suggestions, clipboard illustration for no queue entries; (b) a short descriptive heading: "No appointments yet", "No documents uploaded", "No users found", "Generating code suggestions…" (for pending AI state), "No queue entries today"; (c) a contextual CTA button or link: "Book your first appointment" (→ SCR-008), "Upload your first document" (→ SCR-015 upload trigger), "Invite a user" (→ new user form on SCR-020), "Retry code generation" (→ re-triggers the coding API call), "Check back later" (non-linked text for empty queue); the empty state never renders while data is still loading (the skeleton state from us_055 takes precedence while `isLoading = true`); empty state renders only when `isSuccess = true && data.length === 0` (UXR-604).

4. **Given** a toast notification is visible and the user dismisses it manually (clicks the `✕` button), **When** the dismiss interaction is performed, **Then** the toast slides out to the right (CSS `transform: translateX(110%)` over 200ms — consistent with the 200ms visual feedback budget from UXR-501) and is removed from the DOM after the animation completes; if the 5-second auto-dismiss timer is still running at the time of manual dismiss, the timer is cancelled; if the same non-critical failure recurs (e.g., the next SMS delivery also fails within the same session), a new toast is triggered — toasts are not de-duplicated (each failure event generates its own notification); the toast manager is implemented as a React Context singleton (`<ToastProvider>`) wrapping the application root — any component in the tree can trigger a toast via the `useToast()` hook without prop drilling; Playwright tests assert the toast slides in from the right within 500ms of the triggering event and is absent from the DOM 5.5 seconds after appearing (UXR-602, UXR-501).

5. **Given** error and empty states are implemented across all target screens, **When** the Playwright error-handling test suite runs in CI (using API mocking to simulate failures and empty responses), **Then** all of the following pass: (a) toast appears within 500ms of simulated calendar sync failure, auto-dismisses at 5 seconds ± 200ms tolerance, does not block pointer events on underlying content (Playwright `page.click()` on content behind toast succeeds while toast is visible); (b) full-page error state renders on SCR-016 within 300ms of a mocked 500 API response, "Try again" button re-triggers the API call (network request count increments by 1 on click), "Go to dashboard" navigates to SCR-011 for Staff; (c) empty state renders on SCR-014 (empty queue) and SCR-015 (no documents) with correct illustration and CTA visible; (d) empty state CTA on SCR-015 triggers the file upload input dialog (Playwright `page.waitForFileChooser()` asserts the chooser opens on CTA click); (e) no full-page error renders while the skeleton is still loading (UXR-602, UXR-603, UXR-604).

## Edge Cases

- What happens if a critical API failure (UXR-603 full-page error) occurs AND a non-critical toast (UXR-602) is simultaneously triggered (e.g., calendar sync fails at the same moment the patient view API call fails)? → The full-page error replaces the skeleton in the main content area while the toast renders in the fixed bottom-right overlay — these are two independent rendering layers and do not conflict; the toast appears on top of the full-page error state (z-index 50 vs. the page content z-index 0); the user sees both the full-page error ("Patient view unavailable — Try again") and the toast ("Calendar sync failed — your appointment was booked successfully") simultaneously, which is the correct behaviour since they represent two independent failure events.
- What happens if the toast's 5-second timer is paused when the user switches browser tabs (page visibility changes to `hidden`) and resumed when the tab regains focus? → The toast timer uses `setTimeout` which continues counting even while the tab is in the background (JavaScript timers are not paused on visibility change in major browsers); the 5-second countdown is not paused — the toast may have already auto-dismissed by the time the user returns to the tab; this is acceptable behaviour for non-critical notifications — the user does not need to act on them, and they are designed to be interruptible; if the notification is important enough to require acknowledgement, it must be promoted to a modal or persistent notification (which would require a higher-severity escalation decision outside the scope of UXR-602).
- What happens if the empty state CTA for "Generating code suggestions…" on SCR-018 triggers a second code generation API call while the first is still in progress (the AI pipeline is slow)? → The "Retry code generation" CTA button enters a loading state (per UXR-501 from us_055) immediately on click, disabling itself to prevent duplicate API calls — this is the same `isLoading` button guard from us_055 AC-4; while the retry is in flight, the empty state transitions to the skeleton loading state (per UXR-502 from us_055) to indicate the request was received; if the retry also results in an empty response (AI pipeline not started yet), the empty state returns; if the retry returns data, the code suggestions render — ensuring the empty state → skeleton → data or error state machine is consistent.
- What happens if an SVG illustration referenced in the empty state fails to load (e.g., network error serving static assets)? → The SVG illustrations are inlined as React SVG components (not `<img src="...">` tags) — they are part of the JavaScript bundle and cannot independently fail to load after the SPA has loaded; inline SVG components eliminate the external asset dependency; the component inventory specifies all empty-state illustrations as inline SVG components, not as image files, specifically to avoid this failure mode.
- What happens if the SCR-019 Admin Users list shows an empty state because an admin's search filter matches zero users — should the empty state say "No users found" or "No users exist on the platform"? → The empty state message is context-aware: when the URL includes a search query parameter (`?q=...` or `?filter=...`), the empty state heading is "No users match your search" with a secondary CTA "Clear search" that removes the filter and reloads the full list; when no search filter is active and the list is genuinely empty (no users exist), the heading is "No users yet" with the CTA "Invite a user"; the empty state component accepts an `isFiltered: boolean` prop that drives the message and CTA variant.
