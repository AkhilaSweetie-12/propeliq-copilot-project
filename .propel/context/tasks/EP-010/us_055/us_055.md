# User Story - US_055

## Story ID

- ID Format: US_055

## Story Title

- Interaction Feedback: 200ms Visual States, Skeleton Loading & Inline Form Validation

## Description

- As a patient or staff member interacting with forms and data-fetching screens, I want every button press to show a visual response within 200ms, data-loading screens to display skeleton placeholders instead of blank pages, and form validation errors to appear immediately when I leave a field — so that the platform always feels responsive and I receive clear, timely feedback on my actions without having to wait to discover that a field was filled in incorrectly.

## Acceptance Criteria

1. **Given** a user performs any primary action (clicking a button, submitting a form, toggling a control) on any screen, **When** the interaction is registered, **Then** the button or control renders its active/pressed visual state within 200ms of the user event — no action produces a blank or unchanged UI for longer than 200ms; specifically: primary buttons render a `10% darkened` background on `:active` state (implemented via `filter: brightness(0.9)` using the `transition-all duration-micro` CSS transition from `wireframe-shared.css`); buttons performing asynchronous operations (form submit, booking confirm, code finalise) enter a loading state within 200ms — the button label is replaced by a spinner (`aria-busy="true"` + `aria-label="Loading"`) and the button becomes `disabled` to prevent double-submission; spinner/loading state persists until the async operation resolves or rejects; for operations expected to take > 500ms (e.g., AI extraction job enqueue, code suggestion request), a full-page or section-scoped spinner overlay appears within 500ms of the action being triggered; Playwright performance tests assert button active-state renders ≤ 200ms using `page.waitForSelector('[aria-busy="true"]', { timeout: 200 })` after a button click (UXR-501).

2. **Given** the booking calendar (SCR-008), staff queue (SCR-014), patient 360° view (SCR-016), medical coding panel (SCR-018), or admin users list (SCR-019) is navigated to or refreshed, **When** the component mounts and the data-fetching API call is in-flight, **Then** a skeleton loading state renders immediately (within 200ms of navigation, before the first API response) — no blank screen or spinner-only state is shown; each skeleton is a layout-matching placeholder: SCR-008 renders a 7-column calendar grid skeleton with grey animated shimmer blocks in each cell; SCR-014 renders 5 skeleton rows in the queue table (position number placeholder, name placeholder, time badge placeholder, status badge placeholder, action button placeholder); SCR-016 renders the patient header skeleton (name + MRN placeholder) and section skeletons for each clinical data category (vitals, medications, allergies, diagnoses); SCR-018 renders 3 skeleton code rows (code + description + confidence bar + action buttons placeholders); SCR-019 renders 8 skeleton user rows; skeletons use the CSS animation `@keyframes shimmer` (background-position sweep from -200px to 100%) with `animation-duration: 1.5s; animation-iteration-count: infinite`; React `Suspense` with skeleton fallback is used for route-level data fetching; TanStack Query (React Query) `isLoading` state drives the skeleton conditional render (UXR-502).

3. **Given** a user is filling in a form on any form-containing screen (SCR-002 Registration, SCR-006 AI Intake supplementary fields, SCR-007 Manual Intake, SCR-009 Booking & Insurance, SCR-013 Patient Search/Create), **When** the user leaves a field (field `blur` event fires — not form submit), **Then** if the field value fails validation, the inline error state renders: the field's border changes to `var(--color-error)` (red) with a 2px solid border; a `✕` error icon appears to the right of the field input; a descriptive error message appears beneath the field in `var(--color-error)` text (`text-caption` size); the error message is linked to the field via `aria-describedby` (per UXR-204 from us_051); valid error messages are specific and actionable — not generic: "Email address is required" (not "This field is required"), "Password must be at least 8 characters" (not "Invalid password"), "Date of birth must be in MM/DD/YYYY format" (not "Invalid date"); if a previously errored field is corrected and blurred again, the error state clears immediately and the field border returns to `var(--neutral-200)`; form submit button remains enabled (form is not blocked from attempting submission) — submission re-validates all fields and focuses the first errored field if any fail (UXR-601).

4. **Given** the button loading state from AC-1 is active (async operation in flight), **When** the same button is clicked again before the operation resolves, **Then** the button's `disabled` attribute blocks duplicate clicks — no duplicate API call is made; the `<Button>` component's loading state prop (`isLoading: boolean`) sets `disabled={isLoading}` and `aria-disabled="true"` simultaneously; forms with a submit button in loading state cannot be submitted via keyboard (Enter key) — the `onKeyDown` handler checks `isLoading` and prevents default on Enter when true; a Playwright test confirms that rapid double-clicks on the "Confirm Booking" button (SCR-009) result in exactly one `POST /api/appointments` network request (verified via `page.waitForRequest` count assertion); this prevents duplicate bookings, duplicate code finalisation, and duplicate intake submissions (UXR-501).

5. **Given** interaction feedback (AC-1 through AC-4) is implemented across all 21 screens, **When** the Playwright interaction test suite runs in CI, **Then** all of the following pass: (a) button active-state assertion: `[aria-busy="true"]` appears within 200ms of clicking any async submit button on SCR-002, SCR-007, SCR-009, and SCR-018; (b) skeleton rendering assertion: skeleton elements are present in the DOM within 200ms of navigating to SCR-008, SCR-014, SCR-016, SCR-018, SCR-019 (verified by `page.waitForSelector('.skeleton', { timeout: 200 })`); (c) inline validation assertion: on SCR-002 Registration, blurring the email field with an invalid value triggers the error border and error message within one render cycle (`page.waitForSelector('[data-error="true"]', { timeout: 100 })`); (d) no blank flash: a Playwright screenshot comparison at 50ms post-navigation confirms no blank white-screen state on any of the 5 skeleton screens; (e) double-click protection: exactly one API call made per button click on all async-submit buttons (UXR-501, UXR-502, UXR-601).

## Edge Cases

- What happens if the skeleton screen for SCR-016 (Patient 360° View) takes longer than 30 seconds (the AI extraction pipeline timeout) to return data? → The skeleton renders immediately and persists until the API responds; after 30 seconds (the `staleTime` and `gcTime` thresholds configured in TanStack Query for the patient view query), the query transitions to an error state — the skeleton is replaced by the full-page error state from us_056 (UXR-603) with a "Retry" CTA; the skeleton is never permanently stuck — it has a 30-second maximum display duration before transitioning to either the data view or the error state.
- What happens if a form field uses conditional rendering (a field that appears after another field is filled, e.g., "Other insurance provider" text input appearing when "Other" is selected in a select) — does the inline validation handle late-appearing fields? → Conditionally rendered fields register their `blur` validation handler on mount via `useEffect`; the field's initial state is "untouched" (no error shown); validation only fires on the first blur after the field is rendered — it does not pre-validate on appearance; if the form is submitted while a conditional field is visible and empty, the submit-time validation catches it and focuses the first errored field; this ensures consistent validation behaviour for both static and dynamic form fields.
- What happens if a network-slow environment causes the async button operation to take longer than 500ms — should the skeleton overlap with the button spinner? → The button spinner (AC-1) handles the button's own loading state and prevents double-submission; the 500ms page-level spinner is a separate overlay mechanism used for page-level loading (e.g., when navigating away after form submit while awaiting the redirect); these are two distinct feedback layers — the button spinner confirms the action was received; the page-level spinner confirms the application is awaiting a significant async response; they do not conflict and can both be visible simultaneously (the button is in loading state AND the page-level spinner overlays the content).
- What happens if the shimmer animation causes a flash of unstyled content (FOUC) on screens with many skeleton rows (e.g., SCR-019 with 8 user rows rendering the shimmer animation simultaneously)? → Each skeleton row's shimmer animation starts with a staggered `animation-delay` (`nth-child(n) { animation-delay: n * 0.15s }`) so the shimmer sweeps sequentially across rows rather than all rows animating simultaneously; this reduces perceived FOUC and gives the skeleton a polished progressive appearance; the shimmer animation is GPU-accelerated via `transform` (not `background-position` which would trigger paint) using a pseudo-element gradient sweep.
- What happens if an inline validation error message contains personalised data that could expose sensitive information (e.g., "Your existing account [email@example.com] is already registered")? → Inline validation error messages are pre-defined static strings sourced from a client-side validation message map — they never include server-returned error details that might contain account existence information; the "Email already registered" error displays "An account with this email address already exists. Please log in instead." without echoing the email address; this prevents user enumeration (OWASP A07:2021 Identification and Authentication Failures) at the form validation layer, complementing the API-level protection in us_017 (EP-001).
