# User Story - US_005

## Story ID

- ID Format: US_005

## Story Title

- JWT Authentication Infrastructure (Identity + BCrypt + Redis Blocklist)

## Description

- As a developer, I want ASP.NET Core Identity with BCrypt password hashing, short-lived JWT access tokens (15-minute TTL), rotating refresh tokens, and an Upstash Redis blocklist configured, so that all feature endpoints can be secured with role-based access control from day one with no PHI stored in the token or cache layer.

## Acceptance Criteria

1. **Given** a registered user sends a valid `POST /api/auth/login` request with correct credentials, **When** the request is processed, **Then** the API returns HTTP 200 with a JSON body containing a signed JWT access token (TTL = 15 minutes, algorithm = HS256 or RS256) and a rotating refresh token; the refresh token is stored in Upstash Redis with a TTL equal to the refresh token lifetime.

2. **Given** an issued JWT access token with a 15-minute TTL, **When** the token is used to call any `[Authorize]`-protected endpoint after 15 minutes have elapsed (or after 15 minutes of inactivity per NFR-009), **Then** the API returns HTTP 401 Unauthorized; the Redis session entry for that token is cleared and the user must re-authenticate.

3. **Given** a user calls `POST /api/auth/logout` with a valid JWT, **When** the request is processed, **Then** the JWT token ID (`jti` claim) is added to the Redis blocklist with a TTL matching the token's remaining expiry; all subsequent requests bearing that JWT — even before expiry — return HTTP 401 Unauthorized.

4. **Given** the RBAC policy is registered with three roles (Patient, Staff, Admin), **When** a `Patient` role user calls a `[Authorize(Roles = "Staff")]`-decorated endpoint, **Then** the API returns HTTP 403 Forbidden before any handler logic executes; no patient data or application state is leaked in the 403 response body.

5. **Given** a new user account is created via `POST /api/auth/register`, **When** the account is persisted to the database, **Then** the stored `password_hash` column contains a BCrypt hash generated with a work factor ≥ 12; the plaintext password does not appear in any log, error message, database field, or HTTP response at any point in the registration flow.

6. **Given** TR-006 (no PHI in Redis), **When** any Redis key set during auth flows is inspected via `redis-cli SCAN` and `GET`, **Then** the stored values contain only opaque token identifiers (JWT IDs, refresh token hashes) and TTL metadata; no patient name, DOB, address, insurance ID, clinical data, or any other PHI field appears in any Redis value.

7. **Given** a valid refresh token is submitted to `POST /api/auth/refresh`, **When** the token is processed, **Then** the old refresh token is immediately invalidated in Redis (deleted), a new access token and new refresh token are issued, and the new refresh token is stored in Redis; submitting the old refresh token a second time returns HTTP 401 Unauthorized (replay prevention).

## Edge Cases

- What happens when Upstash Redis is temporarily unreachable during a token validation request? → The auth middleware falls back to stateless JWT signature-only validation, logs a `WARNING` circuit-open event to Serilog with the Redis unavailability reason, and proceeds; the fallback is logged to the audit trail and does not expose a security bypass.
- What happens when the JWT signing key is missing from environment variables at startup? → The application throws a `InvalidOperationException` ("JWT_SIGNING_KEY environment variable is required") during startup configuration validation and does not reach `app.Run()`; no default or hardcoded key is used.
- What happens when a malformed or tampered JWT is submitted to a protected endpoint? → ASP.NET Core's JWT Bearer middleware returns HTTP 401 with `error="invalid_token"` in the `WWW-Authenticate` header; no internal exception details are exposed in the response body (OWASP A02:2021).
- What happens when the same refresh token is submitted concurrently from two sessions (race condition)? → Redis `SET NX` (set-if-not-exists) with a lock key on the refresh token ID ensures only one concurrent refresh succeeds; the second concurrent attempt returns HTTP 401 and the entire token family is revoked as a precaution.
