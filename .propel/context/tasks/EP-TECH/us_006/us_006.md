# User Story - US_006

## Story ID

- ID Format: US_006

## Story Title

- Test Framework Scaffolding (xUnit + Moq + Playwright)

## Description

- As a developer, I want xUnit, Moq, and Playwright test projects scaffolded, wired to the application, and runnable in CI, so that unit, integration, and end-to-end tests can be written and executed immediately for every feature, with a load-test baseline confirming the 100 concurrent user and p95 latency NFRs.

## Acceptance Criteria

1. **Given** the `/api.tests` xUnit project referencing the `/api` project, **When** `dotnet test` is executed, **Then** the test runner discovers all test classes in the xUnit project, runs the included smoke test suite against an in-memory or Testcontainers-based PostgreSQL instance, and reports all tests as passed with results output in JUnit XML format at `/test-results/api-tests.xml`.

2. **Given** Moq is installed and a repository interface is defined in `/api`, **When** a unit test creates a mock with `new Mock<IAppointmentRepository>()` and configures a `.Setup()` call, **Then** the mock returns the configured value in isolation, no real database connection is required, and `mockRepo.Verify()` confirms the expected call was made.

3. **Given** the `/e2e` Playwright project, **When** `npx playwright test` is executed with the full application stack running (API + SPA), **Then** the test runner launches a headless Chromium browser, navigates to `http://localhost:3000`, asserts the page title contains "PropelIQ Health", and reports at least one passing test; total execution time for this smoke test is under 30 seconds.

4. **Given** the `WebApplicationFactory<Program>` integration test harness is configured, **When** 100 sequential HTTP requests are made to `GET /api/health` via the test client in a single integration test, **Then** all 100 requests complete with HTTP 200 and the p95 response time remains under 2 seconds, establishing the NFR-001 and NFR-015 baselines.

5. **Given** the CI configuration stub (`/.github/workflows/ci.yml`), **When** any pull request is opened targeting the `main` branch, **Then** the CI pipeline runs `dotnet test` and `npx playwright test` (smoke tests only); the pipeline exits with code 0 when all tests pass and with a non-zero exit code when any test fails, blocking merge on failure.

6. **Given** an integration test modifies data in the test database, **When** the test completes (pass or fail), **Then** the `IAsyncLifetime.DisposeAsync()` implementation rolls back the wrapping transaction, leaving the database in its pre-test state for the next test in the suite.

## Edge Cases

- What happens when the Playwright browser binary is not installed in the devcontainer? → `npx playwright install chromium` is included in the `devcontainer.json` `postCreateCommand`; the `e2e` package.json `pretest` script also calls `playwright install chromium --with-deps` to self-heal in environments where the binary is missing.
- What happens when `dotnet test` is run before the devcontainer database service is healthy? → The integration test base class checks for the `DATABASE_URL` environment variable at class initialisation; if the variable is absent or the database is unreachable, the test class is skipped with a `Skip("Database unavailable — run inside devcontainer")` message and the suite does not fail with a misleading error.
- What happens when a test leaks an open HTTP connection to the test server? → `WebApplicationFactory` is disposed in `IAsyncLifetime.DisposeAsync()`; a `using` block pattern in all integration test fixtures ensures the factory is always disposed even on test failure, preventing port exhaustion.
- What happens when the Playwright E2E smoke test runs against a locally started SPA that has not yet reached TTI? → The Playwright test uses `page.waitForLoadState('networkidle')` before asserting the page title, ensuring the assertion does not fire before the SPA is interactive.
