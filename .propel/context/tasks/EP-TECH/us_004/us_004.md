# User Story - US_004

## Story ID

- ID Format: US_004

## Story Title

- EF Core 9 Database Migration Framework Setup

## Description

- As a developer, I want Entity Framework Core 9 configured with the Npgsql provider and versioned migration scripts applied automatically on startup, so that all database schema changes are traceable, reproducible, and free from manual schema alterations in every environment.

## Acceptance Criteria

1. **Given** EF Core 9 + Npgsql is added to the `/api` project, **When** `dotnet ef migrations add InitialCreate` is executed, **Then** a timestamped migration file is generated in `/api/Migrations/` containing valid `Up()` and `Down()` methods with no compilation errors.

2. **Given** the application startup pipeline calls `await context.Database.MigrateAsync()`, **When** the API starts, **Then** all pending EF Core migrations are applied to the target PostgreSQL database before the first HTTP request is served; the `__EFMigrationsHistory` table confirms each applied migration by name and applied timestamp.

3. **Given** the pgvector extension requirement (TR-003), **When** the `InitialCreate` migration runs its `Up()` method, **Then** it executes `CREATE EXTENSION IF NOT EXISTS vector;` as the first SQL statement, enabling pgvector in PostgreSQL without requiring a manual DBA step.

4. **Given** NFR-018 (no manual schema alterations), **When** a developer runs `dotnet ef migrations add CheckDrift` after manually executing an `ALTER TABLE` on the live database, **Then** EF Core detects the schema divergence and generates a migration capturing the difference; the `dotnet ef database update` command then re-aligns the migration history.

5. **Given** the migration framework is operational, **When** `dotnet ef migrations list` is run against the configured database, **Then** all applied migrations are listed with their applied timestamps in ascending sequential order, and zero migrations show `pending` status after a fresh `dotnet ef database update`.

6. **Given** the connection string is sourced exclusively from the `DATABASE_URL` environment variable, **When** a developer runs any `dotnet ef` command without `DATABASE_URL` set, **Then** the EF design-time factory throws an `InvalidOperationException` with the message "DATABASE_URL environment variable is required for migrations" and does not fall back to any hardcoded connection string.

## Edge Cases

- What happens when two developers create EF Core migration files concurrently with identical last-migration snapshots? → `dotnet ef migrations add` on the second developer's branch detects the snapshot conflict; the developer must merge the first migration and re-generate before proceeding; the CI pipeline's migration lint step (`dotnet ef migrations script --idempotent`) fails on merge conflicts to prevent accidental deployment.
- What happens when a migration `Up()` fails midway through execution (e.g., a constraint violation)? → EF Core wraps each migration in an implicit transaction (where the database supports it); on failure, the transaction is rolled back and the `__EFMigrationsHistory` row is not written, leaving the database in its pre-migration state.
- What happens when the pgvector extension is already installed in a pre-existing database? → `CREATE EXTENSION IF NOT EXISTS vector;` is idempotent; running the migration against an existing database with pgvector already installed completes without error.
