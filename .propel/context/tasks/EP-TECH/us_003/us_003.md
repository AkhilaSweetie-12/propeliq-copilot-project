# User Story - US_003

## Story ID

- ID Format: US_003

## Story Title

- GitHub Codespaces DevContainer & Full Service Stack Setup

## Description

- As a developer, I want a fully containerized development environment configured in GitHub Codespaces with all platform services (PostgreSQL 16 + pgvector, Upstash Redis, Seq, Ollama) provisioned automatically via docker-compose, so that any team member can start development consistently in under 5 minutes without manual infrastructure setup.

## Acceptance Criteria

1. **Given** the repository is opened in GitHub Codespaces, **When** the devcontainer build completes, **Then** all required services start automatically via `docker-compose up`: PostgreSQL 16 (port 5432, pgvector extension available), a Redis-compatible cache (port 6379), Seq community server (port 5341), and Ollama (port 11434) — all within 5 minutes of container creation.

2. **Given** the devcontainer is running, **When** `docker-compose ps` is executed, **Then** every service container reports status `healthy` based on configured Docker health checks; no service remains in `starting` or `unhealthy` state after the 5-minute provisioning window.

3. **Given** the devcontainer is active, **When** `dotnet run` (API, port 5000) and `npm run dev` (SPA, port 3000) are started, **Then** both ports are automatically forwarded by Codespaces and accessible in the browser; the Codespaces ports panel shows both as `Public` or `Private` (per team security policy).

4. **Given** the post-create command in `devcontainer.json`, **When** the devcontainer finishes building for the first time, **Then** `ollama pull llama3.2:3b-instruct-q8_0` and `ollama pull nomic-embed-text` complete successfully and `curl http://localhost:11434/api/tags` returns a JSON response listing both model names.

5. **Given** the platform health endpoint, **When** `GET /api/health` is called after all services are up, **Then** the API returns HTTP 200 with a JSON body confirming database connectivity (PostgreSQL), cache connectivity (Redis), and AI model availability (Ollama), satisfying the 99.9% uptime monitoring baseline (NFR-011).

6. **Given** the `.env.example` file at the repository root, **When** a developer copies it to `.env` and starts the devcontainer, **Then** all required environment variables (database URL, Redis URL, JWT signing key, Ollama base URL) are pre-populated with development-safe defaults; `.env` is listed in `.gitignore` and never committed to source control.

## Edge Cases

- What happens when the Ollama model download is interrupted by a network timeout? → The `post-create` shell script wraps the `ollama pull` command in a retry loop (max 3 attempts, 10-second delay between attempts) and logs each failure to stdout; the devcontainer completes startup regardless, with a warning message explaining which models are missing.
- What happens when the Codespaces machine is provisioned with less than 16 GB RAM? → The `devcontainer.json` specifies `"hostRequirements": { "memory": "16gb" }` to request an adequate machine; the Codespaces creation UI warns the user if the selected machine does not meet the requirement.
- What happens when PostgreSQL fails its health check due to a slow first-time initialisation? → The `depends_on` condition in `docker-compose.yml` uses `condition: service_healthy` for the API service, ensuring the API waits for PostgreSQL to pass its health check before starting.
- What happens when port 5432 is already in use on the host machine (local Docker Desktop scenario)? → The `docker-compose.yml` maps PostgreSQL to host port `5433` for local development; the `DATABASE_URL` default in `.env.example` references port `5433`.
