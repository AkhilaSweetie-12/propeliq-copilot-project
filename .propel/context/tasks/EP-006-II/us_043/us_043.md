# User Story - US_043

## Story ID

- ID Format: US_043

## Story Title

- 360° Patient View UI, Trust-First Display & Conflict Acknowledgement

## Description

- As a Staff user reviewing a patient's clinical history, I want to view a unified 360-Degree Patient View with AI-suggested fields clearly distinguished from human-verified data, acknowledge any data conflicts surfaced by the system, and mark the view as verified, so that I can confidently use the patient's aggregated clinical record for downstream coding and billing workflows.

## Acceptance Criteria

1. **Given** an authenticated Staff user navigates to SCR-016 (360-Degree Patient View) for a patient, **When** the page loads, **Then** the SPA calls `GET /api/patients/{id}/view360`; while data is loading a skeleton loading state is rendered (no blank screen per UXR-502); once data loads the view renders all aggregated field groups (Vitals, Medications, Allergies, Diagnoses, Surgical History) in a structured layout; each field group header shows a count badge (e.g., "3 medications"); AI-suggested fields (those not yet verified by Staff) are rendered with an "AI Suggested" badge (amber, icon + text label) and a muted background colour (`--neutral-100`); fields that have been verified by Staff are rendered with a solid navy border and no badge; the visual distinction MUST use both colour and text label (not colour alone) for colour-blind accessibility; empty field groups show "No data extracted" with a source link to upload documents (UXR-403, FR-028).

2. **Given** the 360° view contains conflict flags (`conflict_flags` array non-empty), **When** the view is rendered on SCR-016, **Then** a "Conflicts Detected" banner is displayed at the top of the page with the count of unacknowledged conflicts (e.g., "2 conflicts require your review"); each conflicted field group displays a red conflict indicator badge; a "Review Conflicts" CTA button is visible; clicking "Review Conflicts" or the conflict indicator badge navigates the Staff user to SCR-017 (Conflict Acknowledgement Panel); fields with unacknowledged conflicts are rendered with a red border highlight; the "Mark as Verified" button at the bottom of SCR-016 is disabled while any unacknowledged conflicts remain (FR-030, UC-007 step 7).

3. **Given** the Staff user is on SCR-017 (Conflict Acknowledgement Panel), **When** the page loads, **Then** the SPA renders each conflict as a card showing: the field type, the conflicting values side-by-side with their source document names and extraction dates, and an "Acknowledge" button per conflict; the conflict card header displays the conflict type in a colour-coded tag (e.g., Medication, Allergy, Vital); Staff clicks "Acknowledge" on a conflict card; this triggers `POST /api/patients/{id}/view360/acknowledge { conflict_id }`; on 200 OK the conflict card transitions to a "Resolved" visual state (green border, checkmark icon, "Acknowledged by [staff name] at [time]"); the "Acknowledge" button is replaced with a disabled "Acknowledged" label; `audit_logs (action=ConflictAcknowledged, actor_id=staff_id, patient_id, conflict_id)` is written (FR-030, UC-007 steps 7a loop).

4. **Given** all conflicts have been acknowledged on SCR-017, **When** the Staff clicks "Back to Patient View" and returns to SCR-016, **Then** the "Conflicts Detected" banner is no longer shown; all conflict indicator badges are cleared; the "Mark as Verified" button is now enabled (blue, primary variant); Staff clicks "Mark as Verified"; a confirmation modal appears: "Mark this patient's 360° view as verified? This confirms all AI-extracted data has been reviewed."; Staff confirms; the SPA calls `POST /api/patients/{id}/view360/verify`; on 200 OK the view header updates to show a "Verified" badge (green, with Staff name and timestamp); all "AI Suggested" field badges are replaced by "Staff Verified" labels with a solid border; `audit_logs (action=ViewVerified)` is written (FR-030, UC-007 step 8).

5. **Given** one or more fields are flagged as `_NeedsReview` (confidence < 0.6, from us_041 AIR-Q04), **When** the Staff views those fields on SCR-016, **Then** `_NeedsReview` fields are rendered with an amber "Needs Review" badge (distinct from the regular "AI Suggested" badge) and a muted yellow background (`--color-warning-50`); an inline "Enter value manually" link is shown next to each Needs Review field; clicking "Enter value manually" opens an inline edit input for that field; Staff enters the value and clicks "Save"; the SPA calls `PATCH /api/patients/{id}/view360/fields { field_type, field_value, source="StaffManual" }`; on 200 OK the field renders as "Staff Verified" with the entered value; this allows manual resolution of low-confidence fields without re-running the AI pipeline (AIR-Q04, UXR-403).

## Edge Cases

- What happens if the Staff navigates away from SCR-017 mid-acknowledgement (with some conflicts acknowledged, some not)? → Acknowledged conflicts are persisted immediately on each `POST /acknowledge` API call — there is no batch/draft mode; navigating away and returning to SCR-017 shows the partially-acknowledged state accurately (acknowledged conflicts show "Acknowledged", unacknowledged show "Acknowledge" button); progress is not lost; the "Mark as Verified" button on SCR-016 remains disabled until all remaining unacknowledged conflicts are resolved.
- What happens if a new document is uploaded and processed while the Staff is actively viewing SCR-016? → The `AggregatePatientViewJob` re-runs and sets `is_verified=false` (per us_042 AC-1); the next time the Staff calls `GET /api/patients/{id}/view360` (e.g., on page refresh or auto-polling every 30 seconds), the response reflects the updated data with `is_verified=false`; the UI shows a non-blocking info toast "New document data available — view has been reset for re-verification" (warning variant, auto-dismisses 5 s); the "Mark as Verified" button is disabled again; previously acknowledged conflicts remain acknowledged unless the new document introduces new conflicts.
- What happens if a Patient navigates to SCR-016's URL directly (attempting to view their own 360° record)? → The `GET /api/patients/{id}/view360` endpoint enforces `role=Staff` or `role=Admin` RBAC; a Patient JWT receives HTTP 403 Forbidden; the React Router guard redirects the Patient to their dashboard (SCR-004) and shows a brief toast "You do not have permission to access this page"; Patient-facing clinical data is surfaced through the patient intake review (SCR-007b), not the 360° view.
- What happens if the `GET /api/patients/{id}/view360` call fails with HTTP 500 (e.g., aggregation job still running)? → The SPA renders the full-page error state with a "Retry" CTA (UXR-603); the "Retry" button re-triggers the GET request; no partial data is shown during the error state; the skeleton loading state (UXR-502) is shown again on retry while the request is in-flight; if the retry also fails, the error state persists with a "Contact support" message.
- What happens if two Staff users acknowledge conflicts for the same patient simultaneously? → The `POST /api/patients/{id}/view360/acknowledge` API uses an optimistic concurrency check via `conflict_flags` JSONB patch: it reads the current `conflict_flags`, sets the specific `conflict_id.acknowledged=true`, and UPSERTs with a version check; if two concurrent acknowledgements for different conflicts arrive simultaneously, each succeeds independently (no conflict between them); if two requests acknowledge the same `conflict_id`, the second request returns 200 OK idempotently (already-acknowledged returns success, not error) — this prevents duplicate acknowledgement UI states.
