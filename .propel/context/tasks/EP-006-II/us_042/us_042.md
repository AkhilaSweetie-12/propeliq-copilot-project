# User Story - US_042

## Story ID

- ID Format: US_042

## Story Title

- 360Â° Patient View: Aggregation, De-duplication & Conflict Detection API

## Description

- As an authorised Staff user, I want the system to automatically aggregate all extracted clinical data from a patient's documents into a unified 360-Degree Patient View, de-duplicate repeated entries, and surface critical data conflicts, so that I can review a single, reliable patient record without manually reconciling across multiple documents.

## Acceptance Criteria

1. **Given** one or more `extracted_clinical_data` rows exist with `document_id` belonging to a patient and `upload_status=Extracted`, **When** the `AggregatePatientViewJob(patient_id)` Hangfire job runs (triggered automatically after each successful `ExtractionJob` completes), **Then** the job queries all `extracted_clinical_data` rows for `patient_id` where `field_type` is NOT a `_NeedsReview` variant; groups rows by `field_type`; builds the aggregated JSONB structure (encrypted before persist per NFR-007): `{ vitals: [...], medications: [...], allergies: [...], diagnoses: [...], surgical_history: [...] }` with source attribution (`document_id`, `extracted_at`, `source_text`) for each value; UPSERTs `patient_view_360 (patient_id, aggregated_data, last_updated_at=NOW(), is_verified=false)` â€” sets `is_verified=false` on every aggregation run to require fresh Staff review after new documents are added (FR-028).

2. **Given** the aggregated data contains repeated values across multiple documents (e.g., "Metformin 500mg" appears in three documents), **When** the de-duplication step runs, **Then** the job identifies duplicate entries by field value normalisation (case-insensitive, whitespace-trimmed comparison of the core value string); for each group of duplicates the job retains exactly one entry â€” the most recently extracted occurrence (sorted by `extracted_at DESC`); the retained entry includes all source document attributions from the duplicate group (not just the most recent source) so Staff can trace which documents contributed the value; `audit_logs (action=DeduplicationApplied, patient_id, field_type, duplicate_count=N)` is written for each field type where duplicates were removed (FR-029).

3. **Given** the aggregated data contains values for the same field type from different documents that appear contradictory, **When** the conflict detection step runs, **Then** the job applies deterministic conflict detection rules per field type:
   - **Medications**: same medication name, dosage differs by more than 10% â†’ conflict
   - **Allergies**: allergy listed in one document and absent (not just unlisted) in a second document that explicitly states "No Known Allergies" â†’ conflict
   - **Vitals**: same vital type, values differ by more than the defined threshold (BP: â‰Ą 20 mmHg systolic, Weight: â‰Ą 10 kg) across documents taken within the same 30-day window â†’ conflict
   - **Diagnoses / SurgicalHistory**: contradictory presence/absence of a condition (one document asserts diagnosis, another explicitly denies it) â†’ conflict
   For each conflict detected, the job appends an entry to `patient_view_360.conflict_flags` JSONB: `{ conflict_id: UUID, field_type, field_name, values: [{document_id, value, extracted_at}], acknowledged: false }`; `is_verified` remains `false`; `audit_logs (action=ConflictDetected, patient_id, field_type, conflict_id)` is written (FR-030).

4. **Given** the `AggregatePatientViewJob` has completed, **When** Staff calls `GET /api/patients/{id}/view360`, **Then** the API returns the `patient_view_360` payload: `{ aggregated_data (decrypted), conflict_flags, is_verified, verified_by, verified_at, last_updated_at }`; the endpoint enforces RBAC â€” only `role=Staff` or `role=Admin` may access this endpoint; a `role=Patient` request receives HTTP 403; `audit_logs (action=PatientView360Accessed, actor_id, patient_id)` is written on every successful read (PHI access log per NFR-017); if no `patient_view_360` row exists for the patient yet (no documents processed), the API returns HTTP 404 with body `{ "status": "no_data", "message": "No extracted documents available for this patient." }` (FR-028).

5. **Given** the Staff is ready to verify the 360Â° view with no unacknowledged conflicts, **When** Staff calls `POST /api/patients/{id}/view360/verify`, **Then** the API checks `patient_view_360.conflict_flags` for any entries where `acknowledged=false`; if any unacknowledged conflicts remain, the API returns HTTP 400 `{ "error": "AllConflictsMustBeAcknowledged" }`; if all conflicts are acknowledged (or no conflicts exist), the API sets `patient_view_360 SET is_verified=true, verified_by=staff_id, verified_at=NOW()`; inserts `audit_logs (action=ViewVerified, actor_id=staff_id, patient_id)`; returns HTTP 200 `{ "is_verified": true, "verified_at": "..." }` (FR-030, UC-007 step 8).

## Edge Cases

- What happens if new documents are uploaded for a patient whose view is already `is_verified=true`? â†’ The `AggregatePatientViewJob` re-runs on every successful extraction; it sets `is_verified=false` on every UPSERT (AC-1); this means Staff must re-verify the view after each new document is processed; this is intentional â€” new documents may introduce new conflicts or supersede existing verified values; `audit_logs (action=ViewResetUnverified, patient_id, triggered_by=document_id)` is written to make the re-verification event traceable.
- What happens if the `AggregatePatientViewJob` is triggered concurrently for the same patient (e.g., two documents extracted simultaneously)? â†’ The job acquires a PostgreSQL advisory lock `pg_try_advisory_lock(patient_id_hash)` before the UPSERT; if the lock is not available (another aggregation is in progress), the job reschedules itself with a 5-second delay (Hangfire `Schedule` call) and retries up to 3 times; if all retries fail to acquire the lock, the job transitions to `Failed` state; the `patient_view_360` remains consistent because only one job holds the lock at a time; concurrent extraction does not produce a corrupted aggregate.
- What happens if all extracted values for a patient are `_NeedsReview` variants (confidence < 0.6 from us_041) and no confirmed values exist? â†’ The aggregation query (AC-1) filters out `_NeedsReview` field types; if no rows pass the filter, `aggregated_data` is an empty JSONB object `{}`; `patient_view_360` is upserted with empty data and `is_verified=false`; `conflict_flags=[]`; the `GET /api/patients/{id}/view360` response still returns HTTP 200 (the record exists) but with empty aggregated data; Staff can use the "Enter data manually" fallback (SCR-015) to populate data that the AI could not confidently extract.
- What happens if the de-duplication step identifies a medication name with different brand/generic representations (e.g., "Metformin" vs "Glucophage")? â†’ Phase 1 de-duplication uses exact string normalisation (case-insensitive, whitespace-trimmed) â€” brand/generic synonym resolution is not in scope; "Metformin" and "Glucophage" are treated as different medications and both retained; this is a known limitation documented in the de-duplication logic and accepted for Phase 1 accuracy; the Staff review step is the safety net for this class of false negatives.
- What happens if Staff attempts to verify the view via `POST /api/patients/{id}/view360/verify` before the aggregation job has run? â†’ `patient_view_360` row does not exist; the API returns HTTP 404 (same as AC-4 no-data response); the Staff UI shows the 404 state and prompts upload of documents before verification is possible.
