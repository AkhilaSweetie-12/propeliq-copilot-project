# User Story - US_041

## Story ID

- ID Format: US_041

## Story Title

- RAG Pipeline: Chunking, Embeddings, Similarity Retrieval & MMR Re-ranking

## Description

- As the clinical intelligence system, I want to enhance the document extraction pipeline with a full RAG retrieval layer — chunking PDF text into overlapping segments, generating vector embeddings, retrieving the most semantically relevant chunks via pgvector cosine similarity, and re-ranking with MMR — so that the LLM receives focused, non-redundant context for each clinical field type, producing higher-quality extractions within the 30-second p90 SLA.

## Acceptance Criteria

1. **Given** the `ExtractionJob(document_id)` has extracted full raw text from a PDF via PdfPig (from us_040), **When** the RAG pipeline chunking step runs, **Then** the raw text is split into segments of approximately 512 tokens each with a 51-token (10%) overlap at boundaries to preserve sentence context; the chunking is token-count-based using the same tokenizer consistent with the embedding model (nomic-embed-text); all chunks are held in memory for the duration of the job — they are NOT persisted as separate database records; if the raw text is fewer than 512 tokens the entire text is treated as a single chunk; chunk boundaries MUST NOT split mid-sentence where possible (AIR-R01).

2. **Given** the PDF text has been split into overlapping chunks, **When** the embedding step runs, **Then** the job calls the Ollama-hosted `nomic-embed-text` model via `POST http://localhost:11434/api/embeddings` for each chunk, producing a 768-dimensional vector per chunk; on successful embedding each chunk's vector is temporarily stored in memory alongside its source text; after all extraction results are validated and persisted to `extracted_clinical_data`, the corresponding chunk embedding is written to `extracted_clinical_data.embedding` (vector(384) — NOTE: if nomic-embed-text produces 768-dim vectors but the schema stores vector(384), the embedding is truncated to the first 384 dimensions to match DR-007's schema definition); the Ollama embedding call for all chunks in a document MUST complete within the overall 30-second p90 budget (AIR-002, AIR-Q02).

3. **Given** chunks have been embedded, **When** the retrieval step executes for each of the 5 target field types (Vitals, Medications, Allergies, Diagnoses, SurgicalHistory), **Then** the job generates a field-type-specific query embedding (e.g., "patient vital signs blood pressure pulse temperature weight") and executes a pgvector cosine similarity query: `SELECT chunk_text, 1 - (embedding <=> query_vector) AS score FROM chunk_store WHERE patient_id = {patient_id} ORDER BY score DESC LIMIT 5`; only chunks with `score >= 0.65` are included in the retrieval result for that field type; if fewer than 2 chunks meet the 0.65 threshold for any field type, the job logs `audit_logs (action=LowConfidenceRetrieval, field_type=X, chunk_count=N)` and marks that field as "Needs Review" in the extraction output rather than proceeding with LLM invocation for that field (AIR-003, AIR-R02, AIR-S04 — patient_id WHERE clause enforced on all pgvector queries).

4. **Given** retrieved chunks pass the 0.65 similarity threshold for a field type, **When** the MMR re-ranking step runs, **Then** the top-5 retrieved chunks are re-ranked using Maximal Marginal Relevance with a diversity factor of 0.3; the MMR algorithm scores each candidate chunk as `MMR_score = 0.7 × similarity(chunk, query) - 0.3 × max_similarity(chunk, already_selected)`; chunks are selected iteratively from the candidate set until 5 are selected (or the set is exhausted); the final ordered list of chunks (post-MMR) is assembled into the LLM context window as the input for structured field extraction for that field type; the LLM is invoked once per field type with the MMR-reranked context (AIR-R03).

5. **Given** the LLM returns an extraction result for a field type, **When** the JSON schema validation step runs, **Then** the raw LLM response is validated against the target JSON schema for the field type (e.g., `Medication` schema: `{ name: string, dosage: string, frequency: string, source: string }`); validation is implemented via the output JSON schema validator (TR-017); if the response fails schema validation the field is flagged as "Extraction Failed" (`upload_status=Failed` for that field); if the response passes but contains a confidence value (`confidence_score`) below 0.6, the field is written to `extracted_clinical_data` with `field_type` set to `{type}_NeedsReview` rather than the base field type (AIR-Q03, AIR-Q04, TR-017).

## Edge Cases

- What happens if the Ollama `nomic-embed-text` model returns an error for a specific chunk's embedding? → The embedding call is retried once (immediate retry); if the retry also fails, the entire document extraction is marked as `Failed` with `audit_logs (action=EmbeddingFailed, document_id, chunk_index=N, error_message)`; the "Enter data manually" fallback CTA is surfaced on SCR-015 via the `upload_status=Failed` state set by us_040's failure path; partial chunk embeddings are not persisted.
- What happens if pgvector returns 0 chunks above the 0.65 cosine similarity threshold for all 5 field types? → All 5 field types are flagged as "Needs Review" (`LowConfidenceRetrieval` for each); the job updates `clinical_documents SET upload_status=Extracted` (the extraction ran, even if no fields were confidently populated); `extracted_clinical_data` rows are inserted with `field_type={type}_NeedsReview` and empty `field_value`; the Staff sees a "Needs Review" indicator for all fields in the 360° view (EP-006-II us_042/us_043); the patient is not shown a "Failed" status since the extraction process completed.
- What happens if the MMR re-ranking step produces fewer than 2 chunks after diversity filtering (e.g., all retrieved chunks are near-identical duplicates)? → The MMR output includes only the unique chunks available; the LLM is invoked with fewer than 5 context chunks; if only 1 chunk remains after MMR the LLM is invoked with that single chunk; the resulting extraction is flagged as potentially low-confidence in `audit_logs` but is NOT automatically rejected — the confidence threshold check in AC-5 (AIR-Q04) handles the confidence-based gating after LLM response.
- What happens if the total embedding time for all chunks in a large document exceeds the 30-second p90 budget? → The job monitors elapsed time from Hangfire dequeue; if the embedding + retrieval + LLM steps collectively approach 25 seconds, the remaining unprocessed field types are skipped and marked as "Needs Review"; the job completes within the budget rather than timing out; `audit_logs (action=ExtractionBudgetExceeded, document_id, processed_field_types=[...])` is written; NFR-003 (30s p90) is maintained at the cost of partial extraction; Staff can re-trigger extraction for the patient after reviewing the partial results.
- What happens if the `extracted_clinical_data.embedding` column is vector(384) but nomic-embed-text returns 768-dimensional vectors? → As specified in AC-2, the embedding vector is truncated to the first 384 dimensions before INSERT; this matches DR-007 schema definition; the truncation is applied consistently for all embeddings; the pgvector cosine similarity queries in AC-3 use the same 384-dimension query vectors for consistency; this technical constraint is documented in the extraction job configuration for future schema evolution.
