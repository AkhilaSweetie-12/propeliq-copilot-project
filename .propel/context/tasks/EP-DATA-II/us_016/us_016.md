# User Story - US_016

## Story ID

- ID Format: US_016

## Story Title

- PHI 6-Year Data Retention Policy & Automated Backup with WAL PITR

## Description

- As a developer, I want a documented and script-enforced PHI data retention policy (minimum 6 years from record creation) and a PostgreSQL daily automated backup configuration with 7-day retention and WAL-based point-in-time recovery, so that the platform satisfies the HIPAA data retention mandate and can recover to any point in the last 7 days without data loss.

## Acceptance Criteria

1. **Given** the retention policy requirement, **When** the EF Core migration `AddRetentionMetadata` runs, **Then** the following PHI tables each have a `created_at` column confirmed as NOT NULL with a TIMESTAMPTZ type (already present from prior migrations): `patient_intakes`, `clinical_documents`, `extracted_clinical_data`, `patient_view_360`, `audit_logs`; a new `data_retention_policy` table is created with columns: `table_name` (TEXT PK), `retention_years` (INTEGER, NOT NULL), `archival_policy` (TEXT, NOT NULL), `last_reviewed_at` (DATE, NOT NULL); seed rows documenting 6-year retention for each PHI table are inserted via the migration's `Up()` method.

2. **Given** the PostgreSQL WAL archiving configuration, **When** the devcontainer's `postgresql.conf` (mounted via docker-compose volume) is applied, **Then** `wal_level = replica`, `archive_mode = on`, and `archive_command = 'cp %p /var/lib/postgresql/wal_archive/%f'` are active; running `SHOW wal_level` in psql returns `replica`; the `/var/lib/postgresql/wal_archive/` directory receives WAL segment files within 5 minutes of the database starting.

3. **Given** the daily backup requirement, **When** the devcontainer `post-start` script runs `pg_basebackup -D /var/lib/postgresql/backups/$(date +%Y%m%d) -Ft -z -P`, **Then** a timestamped compressed base backup directory is created; `ls /var/lib/postgresql/backups/` lists the backup folder; an integration test asserts this directory exists and is non-empty within 60 seconds of devcontainer startup.

4. **Given** the 7-day backup retention, **When** a cleanup script (`/scripts/cleanup-old-backups.sh`) is executed (scheduled as a daily cron job in the devcontainer), **Then** backup directories older than 7 days (based on directory name date prefix) are removed; a dry-run mode (`--dry-run` flag) lists files that would be deleted without deleting them; the script is tested with synthetic dated directories in an integration shell test.

5. **Given** a disaster-recovery scenario, **When** a point-in-time recovery procedure (`pg_restore` from base backup + WAL replay to a target timestamp) is documented in `/docs/disaster-recovery.md` and executed against a test PostgreSQL instance, **Then** the restored database contains all data up to the target timestamp; the restoration procedure is tested in a CI step that creates data, simulates a failure point, restores to that point, and asserts the expected row counts match.

6. **Given** the retention policy is enforced at the application layer, **When** a repository method attempts a hard DELETE on any PHI table row with `created_at` less than 6 years ago, **Then** the repository raises a `RetentionPolicyViolationException("Record is within the 6-year HIPAA retention window — deletion is prohibited")`; the deletion is blocked before reaching the database.

## Edge Cases

- What happens when the WAL archive directory fills up (disk quota exceeded in Codespaces)? → The `archive_command` script checks available disk space before archiving; if free space drops below 500 MB, it logs a `CRITICAL` warning to the Serilog sink and disables WAL archiving temporarily (`archive_mode = off` via `ALTER SYSTEM`), sending a startup notification to the Seq dashboard; this prevents PostgreSQL from halting due to archive failures.
- What happens when a backup is created but the source database is in an inconsistent state (e.g., mid-transaction)? → `pg_basebackup` uses an online backup mechanism that checkpoints the database before starting; the resulting base backup is always transactionally consistent and can be restored without manual recovery intervention.
- What happens when the retention cleanup script tries to delete a backup that is currently being read for a restore? → The cleanup script acquires a file lock (`.lock` file in the backup directory) before deleting; an active restore operation holds the same lock; the cleanup script detects the lock and skips the directory, logging a warning that the backup is in use.
