# User Story - US_013

## Story ID

- ID Format: US_013

## Story Title

- `extracted_clinical_data` Entity with pgvector Embedding Column

## Description

- As a developer, I want the `extracted_clinical_data` EF Core entity and migration defined with a field-type enum, AES-256 encrypted field values, source text attribution, and a nullable `vector(384)` embedding column backed by a pgvector HNSW index, so that the RAG extraction pipeline has a schema-enforced, query-efficient persistence layer capable of cosine similarity retrieval with patient-scoped ACL filtering from the first migration.

## Acceptance Criteria

1. **Given** the EF Core migration for `extracted_clinical_data` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `extract_id` (UUID PK, default `gen_random_uuid()`), `document_id` (UUID, FK → `clinical_documents.document_id`, NOT NULL), `patient_id` (UUID, FK → `users.user_id`, NOT NULL), `field_type` (TEXT CHECK IN ('Vital','Medication','Allergy','Diagnosis','SurgicalHistory'), NOT NULL), `field_value` (TEXT, NOT NULL — AES-256 encrypted), `source_text` (TEXT, NOT NULL), `extracted_at` (TIMESTAMPTZ, NOT NULL), `embedding` (vector(384), nullable).

2. **Given** the pgvector extension is enabled (from us_003/us_004), **When** the migration runs `CREATE INDEX ON extracted_clinical_data USING hnsw (embedding vector_cosine_ops) WHERE embedding IS NOT NULL`, **Then** a named HNSW index exists on the `embedding` column; `\d extracted_clinical_data` in psql confirms the index name, type (`hnsw`), and distance operator (`vector_cosine_ops`).

3. **Given** the RAG pipeline performs a cosine similarity search for a specific patient's document chunks, **When** a LINQ/raw SQL query executes `SELECT ... FROM extracted_clinical_data WHERE patient_id = @patientId ORDER BY embedding <=> @queryVector LIMIT 5`, **Then** the query returns at most 5 rows belonging exclusively to the specified `patient_id` (ACL filter enforced via `WHERE patient_id = @patientId`); rows from other patients are never returned regardless of embedding proximity (AIR-S04 enforcement at schema level).

4. **Given** the `field_value` column stores encrypted clinical data, **When** a `ExtractedClinicalData` entity is persisted, **Then** the EF Core value converter encrypts `field_value` with AES-256 before writing; a direct `SELECT field_value FROM extracted_clinical_data` returns ciphertext; the application layer decrypts transparently, producing the original string — confirmed by a round-trip integration test that writes then reads the same entity.

5. **Given** the `source_text` column stores the raw PDF text excerpt that justified the extraction, **When** a row is inserted, **Then** `source_text` is stored as-is (unencrypted plaintext); it does not contain patient identifiers (PII redaction is enforced by the AI pipeline service layer per AIR-S02 before the value reaches the repository); the column is NOT NULL and may not be empty.

6. **Given** the `embedding` column is nullable, **When** a row is inserted before the embedding is generated (e.g., text extracted but embedding job still queued), **Then** `embedding` persists as `NULL`; the HNSW index's `WHERE embedding IS NOT NULL` partial filter ensures null rows are excluded from similarity searches without affecting index size.

7. **Given** a composite index on `(patient_id, field_type)` is defined in the migration, **When** the Clinical module queries all `Medication` extractions for a specific patient, **Then** the query uses the composite index (confirmed via `EXPLAIN ANALYZE` returning Index Scan) rather than a sequential scan across the full table.

## Edge Cases

- What happens when a `vector(384)` embedding value of incorrect dimensionality (e.g., 768 dimensions) is inserted? → pgvector enforces the declared dimension at the column definition level; an INSERT with a mismatched vector raises a PostgreSQL error (`expected 384 dimensions, not 768`); the repository layer maps this to a domain `EmbeddingDimensionMismatchException`.
- What happens when the `document_id` FK references a document that has been logically deleted (upload_status changed)? → `clinical_documents` rows are never physically deleted (no hard-delete exposed per us_012); the FK constraint is always satisfiable; `upload_status = 'Failed'` does not cascade to extracted rows — they are retained for audit purposes.
- What happens when the HNSW index build fails due to insufficient memory on a resource-constrained Codespaces instance? → The `CREATE INDEX ... USING hnsw` statement uses `WITH (m = 16, ef_construction = 64)` (reduced from defaults) in the migration to lower the memory footprint; if the index still fails, the migration catches the error, logs a warning, and continues — the table is functional without the index (slower similarity queries) until the index is rebuilt.
