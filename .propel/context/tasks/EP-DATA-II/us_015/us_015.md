# User Story - US_015

## Story ID

- ID Format: US_015

## Story Title

- `audit_logs` Append-Only Entity with Permission Guards

## Description

- As a developer, I want the `audit_logs` EF Core entity and migration defined as a strictly append-only table with a PostgreSQL RULE or trigger that rejects UPDATE and DELETE operations at the database level, so that every PHI-touching action has an immutable, tamper-evident record that satisfies the HIPAA 6-year audit retention mandate.

## Acceptance Criteria

1. **Given** the EF Core migration for `audit_logs` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `log_id` (UUID PK, default `gen_random_uuid()`), `actor_id` (UUID, FK → `users.user_id`, NOT NULL), `actor_role` (TEXT, NOT NULL), `action_type` (TEXT, NOT NULL), `entity_type` (TEXT, NOT NULL), `entity_id` (UUID, NOT NULL), `change_summary` (TEXT, NOT NULL), `ip_address` (TEXT, NOT NULL), `occurred_at` (TIMESTAMPTZ, NOT NULL, default `NOW()`).

2. **Given** the append-only requirement, **When** the migration's `Up()` method runs, **Then** it executes the following PostgreSQL statements:
   - `CREATE RULE no_update_audit_logs AS ON UPDATE TO audit_logs DO INSTEAD NOTHING;`
   - `CREATE RULE no_delete_audit_logs AS ON DELETE TO audit_logs DO INSTEAD NOTHING;`
   Verified by attempting `UPDATE audit_logs SET actor_role = 'Hacked' WHERE 1=1` in an integration test — the query returns `UPDATE 0` (zero rows affected) and no row is modified.

3. **Given** the PostgreSQL RULE guards, **When** an integration test calls `context.Database.ExecuteSqlRawAsync("DELETE FROM audit_logs")`, **Then** the command completes without error (RULE suppresses it silently) and `SELECT COUNT(*) FROM audit_logs` returns the same count as before the DELETE — confirming tamper-evidence at the database engine level.

4. **Given** an audit log entry is written by the Audit middleware, **When** `context.AuditLogs.Add(newEntry)` followed by `SaveChangesAsync()` is called, **Then** the row is inserted with all non-null columns populated; the `log_id` is auto-generated by PostgreSQL; `occurred_at` defaults to `NOW()` at the database level, so the middleware does not need to supply it.

5. **Given** the EF Core `AuditLog` entity, **When** any EF Core SaveChanges interceptor or repository method attempts to call `context.AuditLogs.Update(entry)` or `context.AuditLogs.Remove(entry)`, **Then** the application-layer repository raises an `InvalidOperationException("Audit log entries are immutable — update and delete are prohibited")` before the database call is made; the database RULE serves as a secondary defence-in-depth guard.

6. **Given** the audit log is queried by an Admin user to view recent PHI actions, **When** a LINQ query executes `OrderByDescending(l => l.OccurredAt).Where(l => l.EntityType == "Appointment").Take(100)`, **Then** an index on `(entity_type, occurred_at DESC)` in the migration allows the query to use an Index Scan (not Seq Scan) — confirmed by `EXPLAIN ANALYZE` in the integration test.

## Edge Cases

- What happens when `actor_id` references a user who was subsequently soft-deleted (status = Inactive)? → The FK on `actor_id` is set to `RESTRICT`; soft-deleting a user does not physically remove the row, so the FK remains valid permanently; the audit log entry is fully preserved for the 6-year HIPAA retention period.
- What happens when the RULE-based append-only guard is bypassed by a superuser (`postgres` role) running a direct DELETE? → The PostgreSQL RULE applies to all roles including superuser; a RULE with `DO INSTEAD NOTHING` prevents the operation at the rewrite level, not the permission level — it cannot be bypassed by privilege escalation; only dropping the RULE itself (a DDL operation) would remove the guard.
- What happens when `change_summary` contains special characters or a very long description? → TEXT columns in PostgreSQL have no practical size limit; `change_summary` accepts any UTF-8 string; the service layer truncates summaries exceeding 4,000 characters before passing them to the repository, appending `"...[truncated]"` to indicate truncation.
