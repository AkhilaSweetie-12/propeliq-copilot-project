# User Story - US_014

## Story ID

- ID Format: US_014

## Story Title

- `patient_view_360` & `medical_code_suggestions` Entities

## Description

- As a developer, I want the `patient_view_360` and `medical_code_suggestions` EF Core entities and migrations defined with encrypted aggregated data, conflict flags, verification state, code-type enums, confidence scores, and Trust-First review tracking, so that the 360-Degree Patient View and medical coding workflows have a schema-enforced, audit-ready persistence layer from the first sprint.

## Acceptance Criteria

1. **Given** the EF Core migration for `patient_view_360` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `view_id` (UUID PK, default `gen_random_uuid()`), `patient_id` (UUID, FK ŌåÆ `users.user_id`, UNIQUE, NOT NULL), `aggregated_data` (TEXT, NOT NULL ŌĆö AES-256 encrypted JSONB payload), `conflict_flags` (JSONB, NOT NULL, default `'[]'::jsonb`), `is_verified` (BOOLEAN, NOT NULL, default `FALSE`), `verified_by` (UUID, FK ŌåÆ `users.user_id`, nullable), `verified_at` (TIMESTAMPTZ, nullable), `last_updated_at` (TIMESTAMPTZ, NOT NULL); the UNIQUE constraint on `patient_id` enforces one view record per patient.

2. **Given** the EF Core migration for `medical_code_suggestions` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `suggestion_id` (UUID PK), `patient_id` (UUID, FK ŌåÆ `users.user_id`, NOT NULL), `code_type` (TEXT CHECK IN ('ICD10','CPT'), NOT NULL), `suggested_code` (TEXT, NOT NULL), `code_description` (TEXT, NOT NULL), `source_evidence` (TEXT, NOT NULL), `confidence_score` (DECIMAL(5,4), NOT NULL, CHECK BETWEEN 0 AND 1), `status` (TEXT CHECK IN ('Pending','Accepted','Modified','Rejected'), NOT NULL, default `'Pending'`), `reviewed_by` (UUID, FK ŌåÆ `users.user_id`, nullable), `reviewed_at` (TIMESTAMPTZ, nullable), `final_code` (TEXT, nullable).

3. **Given** the `patient_view_360.aggregated_data` column stores AES-256 encrypted JSONB, **When** a `PatientView360` entity is saved after aggregation, **Then** the EF Core value converter encrypts the JSONB payload before write; a direct `SELECT aggregated_data FROM patient_view_360` returns ciphertext; decrypting via the application layer produces a valid JSON object ŌĆö verified by a round-trip integration test.

4. **Given** the `conflict_flags` column stores unencrypted JSONB (conflict metadata is non-PHI structural data), **When** a conflict is detected during aggregation, **Then** the service appends a conflict descriptor object to the `conflict_flags` array and calls `SaveChangesAsync()`; PostgreSQL stores and returns the native JSONB; `is_verified` remains `FALSE` until a Staff member explicitly acknowledges all flags.

5. **Given** the Trust-First verification flow, **When** a Staff member verifies the 360┬░ view (setting `is_verified = TRUE`, `verified_by = staffUserId`, `verified_at = DateTimeOffset.UtcNow`), **Then** `SaveChangesAsync()` persists all three fields atomically in a single UPDATE; a subsequent read confirms all three are non-null; the `last_updated_at` is refreshed by the `SaveChanges` interceptor.

6. **Given** the `confidence_score` column on `medical_code_suggestions` has a DECIMAL(5,4) CHECK BETWEEN 0 AND 1, **When** an AI pipeline attempts to insert a suggestion with `confidence_score = 1.5`, **Then** the CHECK constraint rejects the INSERT and EF Core raises a `DbUpdateException`; the service layer maps this to a domain `InvalidConfidenceScoreException`.

7. **Given** a Staff reviewer accepts a code suggestion, **When** `status = 'Accepted'` and `reviewed_by = staffUserId` and `reviewed_at = now` are saved, **Then** `final_code` is set to the same value as `suggested_code`; when a reviewer modifies the suggestion, `final_code` holds the Staff-corrected code and `status = 'Modified'`; when a suggestion is rejected, `final_code` remains `NULL`.

## Edge Cases

- What happens when a second `patient_view_360` row is inserted for the same `patient_id`? ŌåÆ The UNIQUE constraint on `patient_view_360.patient_id` rejects the INSERT; the repository pattern exposes an `UpsertAsync()` method (using `INSERT ... ON CONFLICT (patient_id) DO UPDATE`) so callers always write via upsert rather than blind insert.
- What happens when `conflict_flags` grows very large (hundreds of conflicts across many documents)? ŌåÆ The service layer caps the `conflict_flags` array at 100 entries; entries beyond 100 are rolled up into a summary object (`{"overflow": true, "additionalCount": N}`); the JSONB type handles large arrays efficiently but the cap prevents unbounded growth.
- What happens when `verified_by` references a Staff user who is later deactivated? ŌåÆ The FK on `verified_by` is set to `RESTRICT` (no CASCADE); deactivating a user sets `status = 'Inactive'` without deleting the row (soft-delete from us_007); the `verified_by` FK remains valid and the audit trail is preserved.
