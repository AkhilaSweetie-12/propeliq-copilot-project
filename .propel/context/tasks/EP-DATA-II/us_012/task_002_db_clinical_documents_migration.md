## Task Overview
Generate and verify the EF Core migration for the `clinical_documents` table. The migration must: create the table with UUID PK (server-default `gen_random_uuid()`), two independently named FK constraints to `users.user_id` (both `ON DELETE RESTRICT`), a CHECK constraint on `upload_status` added via `migrationBuilder.Sql()` (EF Core cannot express this from enum converters), a composite descending index `(patient_id, uploaded_at DESC)` for patient document listing queries, and a full index on `upload_status` for the Hangfire pending-job query. Verify both indexes are used via `EXPLAIN ANALYZE`.

## Implementation Plan

1. **Generate the migration** after task_001 entity and context changes are in place:
   ```bash
   cd api
   dotnet ef migrations add ClinicalDocuments \
     --project Api.csproj \
     --startup-project Api.csproj \
     --output-dir Migrations
   ```
   Then open the generated `<timestamp>_ClinicalDocuments.cs` and verify the scaffolded `Up()` — EF Core will generate `CreateTable`, the two FK `AddForeignKey` calls, and `HasDefaultValueSql("gen_random_uuid()")` for `document_id`. It will NOT generate the CHECK constraint — that must be added manually in step 2.

2. **Add CHECK constraint on `upload_status`** — add immediately after `CreateTable` call in `Up()`:
   ```csharp
   migrationBuilder.Sql("""
       ALTER TABLE clinical_documents
       ADD CONSTRAINT chk_clinical_documents_upload_status
       CHECK (upload_status IN ('Pending', 'Extracted', 'Failed'));
   """);
   ```
   And in `Down()`, drop the constraint before `DropTable`:
   ```csharp
   migrationBuilder.Sql("""
       ALTER TABLE clinical_documents
       DROP CONSTRAINT IF EXISTS chk_clinical_documents_upload_status;
   """);
   ```
   Design decision: EF Core's `HasConversion<string>()` on an enum stores the values as TEXT but does NOT add a database-level CHECK constraint. Without this manual SQL addition, any raw SQL INSERT can bypass the allowed-values restriction.

3. **Add composite descending index for AC-5** — patient document listing ordered by `uploaded_at DESC`:
   ```csharp
   // In Up() after CreateTable and CHECK constraint:
   migrationBuilder.CreateIndex(
       name: "ix_clinical_documents_patient_id_uploaded_at",
       table: "clinical_documents",
       columns: ["patient_id", "uploaded_at"],
       descending: [false, true]);   // patient_id ASC, uploaded_at DESC
   ```
   In `Down()`, drop before `DropTable`:
   ```csharp
   migrationBuilder.DropIndex(
       name: "ix_clinical_documents_patient_id_uploaded_at",
       table: "clinical_documents");
   ```
   This index supports the repository query:
   ```sql
   SELECT * FROM clinical_documents WHERE patient_id = $1 ORDER BY uploaded_at DESC;
   ```

4. **Add full index on `upload_status` for AC-6** — Hangfire pending-document query:
   ```csharp
   // In Up() after the composite index:
   migrationBuilder.CreateIndex(
       name: "ix_clinical_documents_upload_status",
       table: "clinical_documents",
       column: "upload_status");
   ```
   In `Down()`, drop before `DropTable`:
   ```csharp
   migrationBuilder.DropIndex(
       name: "ix_clinical_documents_upload_status",
       table: "clinical_documents");
   ```
   Design note: A partial index `WHERE upload_status = 'Pending'` would be more efficient for large tables (only Pending rows are indexed). However, EF Core cannot express partial index WHERE clauses via `CreateIndex()`. To use a partial index, replace the `CreateIndex` call with:
   ```csharp
   migrationBuilder.Sql("""
       CREATE INDEX ix_clinical_documents_upload_status_pending
       ON clinical_documents (upload_status)
       WHERE upload_status = 'Pending';
   """);
   ```
   And in `Down()`:
   ```csharp
   migrationBuilder.Sql("""
       DROP INDEX IF EXISTS ix_clinical_documents_upload_status_pending;
   """);
   ```
   **Recommended approach**: Use the partial index via `migrationBuilder.Sql()` for production-scale efficiency (consistent with the `partial UNIQUE` pattern used in us_009).

5. **Verify the full `Up()` method structure** — expected order of DDL operations:
   ```
   1. CreateTable("clinical_documents", ...)
      - columns: document_id, patient_id, file_name, file_path, upload_status, uploaded_by, uploaded_at
      - constraints: PK "pk_clinical_documents" on document_id
   2. migrationBuilder.Sql() — ADD CONSTRAINT chk_clinical_documents_upload_status CHECK (...)
   3. AddForeignKey("fk_clinical_documents_patient_id") → users.user_id ON DELETE RESTRICT
   4. AddForeignKey("fk_clinical_documents_uploaded_by") → users.user_id ON DELETE RESTRICT
   5. CreateIndex("ix_clinical_documents_patient_id_uploaded_at", descending: [false, true])
   6. migrationBuilder.Sql() — CREATE INDEX ... ix_clinical_documents_upload_status_pending WHERE upload_status = 'Pending'
   ```
   And the `Down()` method in reverse:
   ```
   1. migrationBuilder.Sql() — DROP INDEX IF EXISTS ix_clinical_documents_upload_status_pending
   2. DropIndex("ix_clinical_documents_patient_id_uploaded_at")
   3. DropForeignKey("fk_clinical_documents_uploaded_by")
   4. DropForeignKey("fk_clinical_documents_patient_id")
   5. migrationBuilder.Sql() — DROP CONSTRAINT IF EXISTS chk_clinical_documents_upload_status
   6. DropTable("clinical_documents")
   ```

6. **Apply the migration and verify**:
   ```bash
   DATABASE_URL="Host=localhost;Port=5433;Database=patientaccess;Username=devuser;Password=devpassword" \
   PHI_ENCRYPTION_KEY="dev-only-phi-key-change-in-production-32chars" \

   ```

7. **Verify schema correctness via psql**:
   ```sql
   -- AC-1: confirm all columns and defaults
   SELECT column_name, data_type, column_default, is_nullable
   FROM information_schema.columns
   WHERE table_name = 'clinical_documents'
   ORDER BY ordinal_position;

   -- AC-4: confirm both FK constraints exist
   SELECT constraint_name, constraint_type
   FROM information_schema.table_constraints
   WHERE table_name = 'clinical_documents';

   -- CHECK constraint values
   SELECT pg_get_constraintdef(oid)
   FROM pg_constraint
   WHERE conname = 'chk_clinical_documents_upload_status';

   -- AC-5: EXPLAIN ANALYZE — patient document listing (index must be used)
   EXPLAIN ANALYZE
   SELECT * FROM clinical_documents
   WHERE patient_id = '00000000-0000-0000-0000-000000000001'
   ORDER BY uploaded_at DESC;
   -- Expected: "Index Scan using ix_clinical_documents_patient_id_uploaded_at"

   -- AC-6: EXPLAIN ANALYZE — Hangfire pending query (partial index must be used)
   EXPLAIN ANALYZE
   SELECT * FROM clinical_documents
   WHERE upload_status = 'Pending';
   -- Expected: "Index Scan using ix_clinical_documents_upload_status_pending"

   -- Confirm CHECK prevents invalid status values
   INSERT INTO clinical_documents (document_id, patient_id, file_name, file_path, upload_status, uploaded_by, uploaded_at)
   VALUES (gen_random_uuid(), gen_random_uuid(), 'test.pdf', 'uploads/test.pdf', 'Invalid', gen_random_uuid(), now());
   -- Expected: ERROR: new row for relation "clinical_documents" violates check constraint "chk_clinical_documents_upload_status"
   ```

8. **Generate idempotent migration script** for deployment verification:
   ```bash
   dotnet ef migrations script --idempotent \
     --project api/Api.csproj \
     --startup-project api/Api.csproj \
     --output clinical_documents_idempotent.sql
   ```
   Review the output to confirm all DDL statements are wrapped in `IF NOT EXISTS` / migration history guards.
