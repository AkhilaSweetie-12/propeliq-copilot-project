# User Story - US_012

## Story ID

- ID Format: US_012

## Story Title

- `clinical_documents` Entity

## Description

- As a developer, I want the `clinical_documents` EF Core entity and migration defined with UUID primary key, patient and uploader FKs, server-side file storage path, and a tri-state upload-status enum, so that the clinical document upload feature has a reliable, FK-constrained metadata store that tracks each PDF's extraction processing state from upload through to completion or failure.

## Acceptance Criteria

1. **Given** the EF Core migration for `clinical_documents` runs against PostgreSQL, **When** `dotnet ef database update` completes, **Then** the `clinical_documents` table has columns: `document_id` (UUID PK, default `gen_random_uuid()`), `patient_id` (UUID, FK → `users.user_id`, NOT NULL), `file_name` (TEXT, NOT NULL), `file_path` (TEXT, NOT NULL), `upload_status` (TEXT CHECK IN ('Pending','Extracted','Failed'), NOT NULL, default `'Pending'`), `uploaded_by` (UUID, FK → `users.user_id`, NOT NULL), `uploaded_at` (TIMESTAMPTZ, NOT NULL).

2. **Given** a `ClinicalDocument` entity is created with `upload_status = 'Pending'` on upload, **When** `SaveChangesAsync()` is called, **Then** the row is persisted with `upload_status = 'Pending'`; the AI extraction Hangfire job subsequently updates `upload_status` to `'Extracted'` on success or `'Failed'` on error; any attempt to set a value outside the CHECK constraint set raises a `DbUpdateException` before the row is written.

3. **Given** the `file_path` column stores a server-side relative path (e.g., `uploads/patients/{patient_id}/{document_id}.pdf`), **When** the value is read by the Clinical module, **Then** the path is resolved against the `UPLOADS_ROOT` environment variable to produce an absolute path; no absolute paths or OS-specific separators are stored in the database.

4. **Given** the `patient_id` and `uploaded_by` FKs both reference `users.user_id`, **When** a document is uploaded by a Staff member on behalf of a patient, **Then** `patient_id` holds the patient's UUID and `uploaded_by` holds the Staff member's UUID — both FKs are independently enforced; inserting a non-existent UUID in either column raises a `DbUpdateException`.

5. **Given** a patient may have multiple uploaded documents, **When** a query retrieves all documents for a patient ordered by `uploaded_at DESC`, **Then** an index on `(patient_id, uploaded_at DESC)` exists in the migration and the query plan shows an Index Scan (not Seq Scan), confirmed via `EXPLAIN ANALYZE` in the integration test.

6. **Given** the `upload_status` column, **When** a Hangfire background job queries for all documents with `upload_status = 'Pending'` to process them, **Then** an index on `upload_status` (partial or full) exists in the migration, and the query returns only `'Pending'` rows without a full table scan.

## Edge Cases

- What happens when a document is uploaded but the file write to the server-side storage path fails before the database row is committed? → The file write and the `SaveChangesAsync()` are wrapped in a try-catch at the service layer; if the file write fails, `SaveChangesAsync()` is not called and no orphan database row is created; the error is logged with the attempted `file_path` value.
- What happens when `file_name` contains path-traversal characters (e.g., `../../etc/passwd`)? → The service layer sanitises `file_name` using `Path.GetFileName()` before persistence, stripping any directory component; the sanitised name is what is stored in the database (OWASP A03 prevention).
- What happens when a document's `upload_status` attempts to transition backwards (e.g., `Extracted` → `Pending`)? → The repository exposes only forward-transition methods (`MarkExtracted()`, `MarkFailed()`); no direct property mutation is allowed on `upload_status` from outside the domain entity; an attempted reverse transition raises a domain `InvalidStatusTransitionException`.
