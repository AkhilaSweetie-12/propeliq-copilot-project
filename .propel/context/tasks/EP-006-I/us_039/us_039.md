# User Story - US_039

## Story ID

- ID Format: US_039

## Story Title

- Clinical Document Upload Screen & File Validation API

## Description

- As an authenticated patient or Staff user, I want to upload one or more PDF clinical documents (lab reports, discharge summaries, historical records) to a patient profile, so that those records are securely stored and queued for AI extraction — giving me a clear status indicator for each document throughout the process.

## Acceptance Criteria

1. **Given** an authenticated Patient navigates to SCR-015 (Clinical Documents Upload) via the patient sidebar, **When** the page loads, **Then** the page renders a document list showing all previously uploaded documents for the patient (file name, upload date, status badge); documents with `upload_status=Pending` show "Processing" badge, `Extracted` show "Extracted" badge, and `Failed` show "Extraction Failed" badge with a visible "Enter data manually" CTA button; an empty state with "Upload your first document" CTA and an illustration is shown when no documents exist; a PDF preview overlay opens when the user clicks a document row (UXR-103, UXR-604, FR-026).

2. **Given** the user clicks the "Upload Document" button on SCR-015, **When** a file picker opens and the user selects a non-PDF file (e.g., .docx, .jpg, .png, .xlsx), **Then** the client-side validator immediately rejects the file before any network request is made; the file picker input clears; an inline error message "Only PDF files are supported" is displayed beneath the upload button; no `POST /api/documents` request is sent (FR-026, UC-007 extension 2a).

3. **Given** the user selects a valid PDF file via the file picker, **When** the user confirms the upload, **Then** the SPA sends `POST /api/documents` with `multipart/form-data` containing the PDF binary and `patient_id` in the request body; the API validates the MIME type server-side (must be `application/pdf`) as a defence-in-depth check regardless of the client-side validation; if the server-side MIME check fails, the API returns HTTP 400 with `"Only PDF files are supported"`; on passing validation the API saves the file to the access-controlled local directory `/uploads/{patient_id}/{document_id}.pdf` (TR-009); creates `clinical_documents (document_id, patient_id, file_name, file_path=/uploads/..., upload_status=Pending, uploaded_by, uploaded_at)`; inserts `audit_logs (action=DocumentUploaded, entity_type=clinical_documents, entity_id=document_id, actor_id, ip_address)` (FR-026).

4. **Given** the API has saved the document and created the `clinical_documents` record, **When** the API enqueues the extraction job, **Then** the API calls `Hangfire.BackgroundJob.Enqueue<IExtractionJobService>(j => j.ProcessDocument(document_id))` on the dedicated AI processing queue (TR-008); the Hangfire job store (PostgreSQL) persists the job; the API immediately returns HTTP 202 Accepted with body `{ "document_id": "...", "status": "Processing" }`; the total API response time for upload does NOT include LLM inference latency (NFR-016); the UI updates the document row badge to "Processing" and shows a progress spinner for that document.

5. **Given** an authenticated Staff user is viewing the patient profile, **When** they navigate to SCR-015 for any patient (not just their own), **Then** the same upload flow and status display applies; both Patient and Staff roles can upload to the patient profile; a Patient may only upload to their own profile (verified by JWT `sub` claim matching `patient_id`); a Staff user may upload to any patient profile (verified by JWT `role=Staff` claim); a Patient attempting to upload to another patient's profile receives HTTP 403 Forbidden (FR-026, NFR-008).

## Edge Cases

- What happens if the user uploads a file that has a `.pdf` extension but contains non-PDF binary content (e.g., a renamed .exe)? → The server-side MIME type detection (via ASP.NET Core's `IFormFile.ContentType` and magic-byte inspection of the first 5 bytes: `%PDF-`) rejects the file with HTTP 400 "Only PDF files are supported"; the client-side extension-only check is not sufficient, so both checks are required; no file is written to `/uploads` before the MIME check passes.
- What happens if the same PDF is uploaded twice for the same patient? → The system does NOT perform deduplication at upload time; each upload creates a new `document_id` and a separate row in `clinical_documents`; deduplication of extracted clinical data is handled downstream by the 360° Patient View aggregation in EP-006-II; this is intentional — the same document may have been updated with corrections and re-uploaded.
- What happens if the `/uploads` directory is unavailable (e.g., volume not mounted in Codespaces)? → The API catches the `IOException` from the file write attempt; returns HTTP 500 "Document storage unavailable"; no `clinical_documents` row is inserted; no Hangfire job is enqueued; `audit_logs (action=DocumentUploadFailed, error_message)` is inserted; the UI displays the full-page error state with retry CTA (UXR-603).
- What happens if the patient uploads multiple PDFs simultaneously (multi-file picker)? → The upload form supports `multiple` attribute on `<input type="file" accept=".pdf">`; the SPA sends one `POST /api/documents` per file sequentially (not concurrently) to avoid exceeding Codespaces memory limits; each upload is independent — a failure on one document does not prevent the others from uploading; each document gets its own status badge on SCR-015.
- What happens if a Patient navigates away from SCR-015 while a document shows "Processing"? → The "Processing" status is persisted in `clinical_documents.upload_status`; when the Patient returns to SCR-015 (GET /api/documents?patient_id=X), the current status is fetched from the database — if the Hangfire job completed, the badge will reflect the final state (Extracted or Failed); no real-time WebSocket is required for Phase 1; status refresh is polling-based (auto-refresh every 10 seconds while any document is in Pending/Processing state).
