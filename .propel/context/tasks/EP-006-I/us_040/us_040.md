# User Story - US_040

## Story ID

- ID Format: US_040

## Story Title

- AI Document Extraction Background Pipeline via Hangfire & PdfPig

## Description

- As a clinician or Staff user, I want the system to automatically extract structured clinical data (vitals, medications, allergies, diagnoses, and surgical history) from uploaded PDF documents using an AI pipeline, so that patient records are available without manual data entry — with the extraction completing within 30 seconds for 90% of documents and failing gracefully with a manual fallback when the AI cannot process a document.

## Acceptance Criteria

1. **Given** a `clinical_documents` record exists with `upload_status=Pending` and a corresponding PDF file exists at `/uploads/{patient_id}/{document_id}.pdf`, **When** the Hangfire `ExtractionJob(document_id)` is dequeued from the dedicated AI processing queue (TR-008), **Then** the job first updates `clinical_documents SET upload_status=Pending` to `Processing` (if not already set by the upload API) and proceeds with PdfPig text extraction; the job runs exclusively on the Hangfire AI processing queue, not on the default queue — this ensures LLM jobs do not starve user-facing background jobs (NFR-016, TR-008).

2. **Given** the `ExtractionJob` is processing a document, **When** PdfPig (MIT, TR-009) extracts full text from the PDF binary at the stored file path, **Then** the extracted raw text is passed to the Ollama-hosted `llama3.2:3b-instruct-q8_0` model (TR-007) via the local Ollama HTTP endpoint (e.g., `POST http://localhost:11434/api/generate`); the LLM prompt uses a structured extraction template requesting JSON output for the 5 target field types: `vitals`, `medications`, `allergies`, `diagnoses`, `surgical_history`; the prompt includes the PII-redacted document text (patient name, DOB, address, insurance ID substituted with opaque tokens per AIR-S02 from EP-008-II scope); the total input token count is validated to be ≤ 4,096 tokens before submission — documents exceeding this limit are chunked into segments before sending (FR-027, TR-007).

3. **Given** the Ollama LLM returns a structured JSON response, **When** the job processes the extraction result, **Then** for each extracted field type (Vital, Medication, Allergy, Diagnosis, SurgicalHistory) the job inserts one row in `extracted_clinical_data (extract_id, document_id, patient_id, field_type, field_value [encrypted], source_text, extracted_at, embedding=NULL)`; `field_value` is AES-256 encrypted before INSERT (PHI protection per NFR-007); after all rows are inserted the job updates `clinical_documents SET upload_status=Extracted`; inserts `audit_logs (action=ExtractionCompleted, entity_type=clinical_documents, entity_id=document_id)`; the entire job — from Hangfire dequeue to `upload_status=Extracted` — MUST complete within 30 seconds at the p90 percentile (NFR-003, FR-027).

4. **Given** the extraction job fails at any step (PdfPig I/O error, Ollama timeout, malformed JSON response, or JSON schema validation failure), **When** the exception is caught, **Then** the job updates `clinical_documents SET upload_status=Failed`; inserts `audit_logs (action=ExtractionFailed, entity_type=clinical_documents, entity_id=document_id, error_message=ex.Message)`; the Hangfire job transitions to the `Failed` state and does NOT auto-retry (extraction failure is considered a permanent failure requiring manual fallback — the patient or Staff uses the "Enter data manually" CTA from SCR-015 via us_039); the `upload_status=Failed` persists until a user manually enters data or re-uploads a corrected document (FR-027, UC-007 extension 3a).

5. **Given** the extraction job successfully produces `extracted_clinical_data` rows, **When** the final step of the job runs, **Then** the job does NOT aggregate data into `patient_view_360` (that aggregation is EP-006-II scope); the job scope ends at persisting individual extracted rows and updating `clinical_documents.upload_status=Extracted`; this boundary ensures the extraction pipeline can be built, tested, and deployed independently of the 360° view aggregation logic in EP-006-II (FR-027 — scoped to extraction only).

## Edge Cases

- What happens if the PDF file is password-protected and PdfPig cannot extract text? → PdfPig throws a `PdfDocumentFormatException` (or returns empty text); the job catches this, updates `upload_status=Failed`, inserts `audit_logs (action=ExtractionFailed, error_message="PDF is encrypted or password-protected")`, and transitions to `Failed` state; the "Enter data manually" fallback CTA is shown to the user on SCR-015; no LLM call is made if PdfPig produces empty text (to avoid wasting Ollama inference on a blank prompt).
- What happens if the Ollama server is unavailable or unresponsive (e.g., model not loaded)? → The job catches the `HttpRequestException` from the Ollama HTTP client; updates `upload_status=Failed`; inserts `audit_logs (action=ExtractionFailed_OllamaUnavailable)`; transitions to Hangfire `Failed` state without retry; the absence of a running Ollama server is a deployment issue, not a transient error — retrying would only consume queue capacity; the Hangfire dashboard surfaces the failure for Admin review.
- What happens if the PDF text extracted by PdfPig exceeds 4,096 tokens? → The job splits the extracted text into segments of ≤ 3,500 tokens (with 200-token overlap at boundaries) before LLM invocation (AIR-O01 — token budget enforcement from EP-008-II); each segment is processed as a separate LLM call; results from all segments are merged into a single extraction result per field type (most recent value wins on duplicates); if the number of required LLM calls exceeds 5 (i.e., document > 17,500 tokens), the job logs a warning and processes only the first 5 segments (capped at approximately 70 pages); this cap applies to Phase 1 and is documented in the extraction job configuration.
- What happens if the LLM returns JSON that does not match the expected extraction schema (e.g., missing fields, wrong types)? → The JSON schema validator in the extraction pipeline validates the LLM response against the target schema before any INSERT; schema-invalid responses are treated as extraction failures: `upload_status=Failed`, `audit_logs (action=ExtractionFailed_SchemaValidation)`; the Hangfire job transitions to `Failed`; the manual fallback CTA is shown on SCR-015; schema validation logic is shared infrastructure with EP-008-II (AIR-Q03) and must be available before this story can be fully tested.
- What happens if the Hangfire PostgreSQL job store becomes unavailable while an extraction job is in-flight? → If the job store is unavailable mid-execution, Hangfire cannot persist the job state update; the Hangfire worker retries the state transition automatically when the job store reconnects; from the application perspective, the `clinical_documents` row may temporarily remain in `Pending` state until the retry succeeds; `upload_status` will eventually converge to `Extracted` or `Failed` when the connection is restored; the patient-facing status polling in us_039 will reflect the persisted state accurately once connectivity is restored.
