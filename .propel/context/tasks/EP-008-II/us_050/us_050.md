# User Story - US_050

## Story ID

- ID Format: US_050

## Story Title

- AI Operational Reliability: Token Budget Enforcement & Ollama Circuit Breaker

## Description

- As the platform operations team, I want all LLM inference requests to be bounded by a 4,096-token input budget with automatic chunk-batching when exceeded, and the Ollama inference endpoint to be protected by a circuit breaker that opens after 3 consecutive failures and retries queued jobs after a 5-minute cool-down — so that LLM calls never exceed the model's context window, and a degraded or restarting Ollama instance does not cascade into unrecoverable job failures or indefinitely stalled Hangfire queues.

## Acceptance Criteria

1. **Given** an AI pipeline (extraction or coding) constructs an LLM prompt by combining the field-type query with retrieved RAG context chunks, **When** the `TokenBudgetEnforcer` service calculates the total input token count, **Then** the enforcer counts tokens using the same tokenizer consistent with the Ollama model's vocabulary (a byte-pair encoding approximation using `Microsoft.ML.Tokenizers` or a character-count heuristic of `total_characters / 4` as the Phase 1 acceptable approximation); if the calculated token count is ≤ 4,096, the prompt is passed to the LLM unchanged; if the token count exceeds 4,096, the enforcer splits the RAG context chunks into smaller batches such that each batch's token count (system prompt + query + context) is ≤ 4,096, and the LLM is invoked once per batch; results from multiple batches are merged by the calling pipeline service (union of extracted fields, taking the highest-confidence result per field type); a `audit_logs (action=TokenBudgetEnforced, entity_id=job_id, change_summary="prompt_tokens=N split_into=M batches")` entry is written whenever splitting occurs; the maximum allowed batch count is 10 — if 10 batches are still insufficient, the remaining chunks are skipped and flagged as `NeedsReview` (AIR-O01, NFR-001, NFR-016).

2. **Given** the `TokenBudgetEnforcer` is applied, **When** a single large PDF produces more than 10 × 4,096-token batches worth of RAG chunks, **Then** the enforcer processes the first 10 batches (covering the highest-similarity chunks, already top-5-ranked by the retrieval step) and skips the remaining lower-ranked chunks; the skipped chunks are not an extraction failure — they are lower-ranked candidates that did not meet the top-5 cutoff anyway; `audit_logs (action=TokenBudgetExceeded, job_id, change_summary="processed=10 batches skipped=N additional chunks")` is written; the extraction result for that field type is based on the best available evidence within the budget; the 30-second p90 SLA (AIR-Q02) is maintained because skipping low-ranked chunks does not add latency (AIR-O01).

3. **Given** the Ollama HTTP client makes an LLM inference call, **When** the call fails with any of the following error conditions: HTTP 5xx response, connection timeout (`>30s`), `HttpRequestException`, or `SocketException`, **Then** the `OllamaCircuitBreakerPolicy` (implemented using Polly `CircuitBreakerAsyncPolicy`) records the failure; after 3 consecutive failures within a 60-second sampling window, the circuit transitions to `Open` state; while the circuit is `Open`, any new LLM call attempt immediately throws `BrokenCircuitException` without contacting Ollama — Hangfire catches this exception, marks the job as `Failed`, and writes `audit_logs (action=CircuitBreakerOpen, change_summary="ollama circuit open after 3 failures within 60s")`; the `Open` state persists for the configured cool-down period (default 5 minutes, overridable via `OLLAMA_CIRCUIT_BREAKER_COOLDOWN_SECONDS` environment variable) before transitioning to `HalfOpen` (AIR-O02, NFR-011, NFR-014).

4. **Given** the circuit is in `HalfOpen` state after the 5-minute cool-down, **When** a Hangfire job retries (automatically, per Hangfire's retry policy with exponential backoff), **Then** the circuit allows one trial LLM call; if the trial call succeeds (HTTP 200 from Ollama), the circuit transitions back to `Closed` state and all subsequent calls proceed normally; `audit_logs (action=CircuitBreakerClosed, change_summary="ollama circuit closed after successful probe")` is written; if the trial call fails, the circuit returns to `Open` for another 5-minute cool-down period; Hangfire jobs that were `Failed` while the circuit was open are automatically retried by Hangfire's built-in retry scheduler (3 automatic retries with exponential backoff: 1 min, 5 min, 15 min) and succeed once the circuit is `Closed`; the Hangfire retry mechanism and the circuit breaker cool-down work independently — Hangfire may retry a job before the circuit closes, but the job will immediately fail again (BrokenCircuitException) and be rescheduled (AIR-O02, NFR-014, NFR-011).

5. **Given** the circuit breaker is open, **When** a Staff user requests code suggestions via `POST /api/patients/{id}/coding/request` or a document is uploaded triggering an extraction job, **Then** the job is enqueued in Hangfire normally (202 Accepted returned to the caller — the API layer does not check circuit breaker state); the job fails immediately with `BrokenCircuitException` on first Hangfire dequeue; the `GET /api/patients/{id}/coding/suggestions` polling endpoint returns `{ "status": "generating" }` for up to 30 seconds, then returns `{ "status": "failed", "error": "AI service temporarily unavailable — please retry in a few minutes" }` (based on the job's `Failed` state in Hangfire); the Staff UI on SCR-018 shows the full-page error state with a "Retry code generation" CTA that re-enqueues the job; core non-AI workflows (appointment booking, queue management, patient dashboard) continue operating normally regardless of circuit breaker state — AI pipeline degradation does not affect non-AI API endpoints (NFR-014, AIR-O02).

## Edge Cases

- What happens if the circuit breaker policy is misconfigured (e.g., `OLLAMA_CIRCUIT_BREAKER_COOLDOWN_SECONDS` set to 0 by accident)? → The `OllamaCircuitBreakerPolicy` validates the cool-down value on startup: if the value is ≤ 0 or missing, it defaults to 300 (5 minutes) and logs `"Warning: OLLAMA_CIRCUIT_BREAKER_COOLDOWN_SECONDS invalid or missing — using default 300s"`; the application does not fail to start due to this misconfiguration; the default protects against infinite-loop cycling between Open and HalfOpen states.
- What happens if Ollama is temporarily slow (e.g., 25-second response time) but not failing — does the circuit breaker open? → The circuit breaker fires only on hard failures (5xx, connection timeout > 30s, socket exception) — not on slow-but-successful responses; a 25-second response that returns HTTP 200 does NOT increment the failure counter; the 30-second p90 budget (AIR-Q02, NFR-004) is monitored independently by the job's elapsed-time tracker (from us_041 and us_044), not by the circuit breaker; slow-but-successful Ollama responses are tracked as performance degradation in application logs, not as circuit breaker events.
- What happens if the `TokenBudgetEnforcer` token count heuristic (`characters / 4`) over-estimates tokens and batches a prompt that would have fit in a single call? → Over-estimation leads to unnecessary batching (worse performance, more audit log entries) but never to a context-window overflow error; this is the safe failure direction — better to batch unnecessarily than to exceed the 4,096-token limit and receive a malformed or truncated response; the `Phase 1 acceptable approximation` note in AC-1 acknowledges this trade-off; a Phase 2 improvement can replace the heuristic with an exact tokenizer when the ML.Tokenizers library is validated for Llama 3.2's vocabulary.
- What happens if a coding job is split into multiple batches by the `TokenBudgetEnforcer` and one batch returns a valid ICD-10 code while a later batch returns a contradicting ICD-10 code for the same diagnosis? → The merge strategy takes the highest-confidence result per code: if both batches suggest code `I10` (hypertension) with confidence 0.94 and 0.87 respectively, the 0.94 result is retained and the 0.87 duplicate is discarded; if two batches suggest different codes for the same clinical presentation, both are inserted as separate `medical_code_suggestions` rows (both are presented to Staff for review on SCR-018); de-duplication within the suggestion set is NOT performed by the merger — Staff review is the resolution mechanism for contradictions within a single patient's coding session.
- What happens if the Polly circuit breaker is not thread-safe and concurrent Hangfire workers increment the failure counter incorrectly? → Polly's `CircuitBreakerAsyncPolicy` is designed to be thread-safe and shared across concurrent callers — the Hangfire background processor may have multiple worker threads executing jobs concurrently; the `OllamaCircuitBreakerPolicy` is registered as a singleton in the DI container so all Hangfire worker threads share the same circuit state; Polly uses atomic state transitions internally to prevent race conditions; no additional locking is required in the calling code.
