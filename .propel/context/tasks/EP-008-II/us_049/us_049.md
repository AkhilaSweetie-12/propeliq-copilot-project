# User Story - US_049

## Story ID

- ID Format: US_049

## Story Title

- AI PHI Safety Controls: Local-Only Inference, PII Redaction, LLM Audit Logging & pgvector ACL

## Description

- As the platform security team, I want all LLM inference to be enforced through the locally-hosted Ollama server only, patient identifiers to be stripped from prompts before they reach the model, every LLM call and response to be recorded in the audit log, and all pgvector similarity searches to be filtered by patient ownership — so that PHI never leaves the platform boundary during AI operations and a complete, tamper-evident record of every model interaction is maintained for HIPAA compliance.

## Acceptance Criteria

1. **Given** any AI pipeline (document extraction, ICD-10/CPT coding, or conversational intake) prepares an LLM inference call, **When** the HTTP client is configured, **Then** the `OllamaHttpClient` is registered as a singleton in the DI container with `BaseAddress = "http://localhost:11434"` (or the Codespaces-internal host configured by `OLLAMA_HOST` environment variable); no other HTTP client or SDK method targeting an external AI endpoint (OpenAI, Azure OpenAI, Anthropic, Google Gemini, Cohere, or any non-localhost base URL) is registered or called within the AI pipeline services; a startup validator (`IHostedService.StartAsync`) verifies connectivity to `GET http://{OLLAMA_HOST}/api/tags` on application boot and logs `"AI Safety: Ollama reachable at {host}"` or `"Fatal: Ollama endpoint unreachable — AI jobs will fail until resolved"` accordingly; an architecture fitness test (using `NetArchTest` or reflection-based assertion) scans all types in the `Clinical`, `Coding`, and `Intake` feature assemblies and asserts no registered `HttpClient` has a `BaseAddress` containing `openai.com`, `anthropic.com`, `googleapis.com`, `azure.com`, or `cohere.ai` — the test fails the build if any non-Ollama external AI endpoint is detected (NFR-010, AIR-S01).

2. **Given** any AI pipeline constructs a prompt containing patient-derived content, **When** the `PiiRedactionMiddleware` processes the prompt, **Then** the middleware applies the following substitutions to the prompt string before it reaches the LLM: patient legal name (first + last) → `[PATIENT-NAME]`, date of birth (any ISO 8601 or common format: `YYYY-MM-DD`, `MM/DD/YYYY`, `DD/MM/YYYY`) → `[DOB-REDACTED]`, street address (detected via street-number + street-name pattern or explicit "Address:" prefix) → `[ADDR-REDACTED]`, insurance ID / member ID (alphanumeric patterns matching `[A-Z]{2,5}-?[0-9]{6,10}` or prefixed with "Insurance:", "Member ID:", "Policy:") → `[INS-REDACTED]`, email address → `[EMAIL-REDACTED]`; the patient summary passed to the LLM MUST contain only clinical content (vitals, medications, diagnoses, chief complaint) with patient identity replaced by opaque tokens; the redaction is performed in memory — the redacted prompt string is never persisted; the original unredacted patient context is sourced from the decrypted `patient_view_360.aggregated_data` and is only referenced within the calling process (TR-016, AIR-S02, NFR-005).

3. **Given** the `PiiRedactionMiddleware` has been applied, **When** any LLM inference call is dispatched via `OllamaHttpClient`, **Then** the `AiAuditInterceptor` (an `HttpMessageHandler` decorator wrapping `OllamaHttpClient`) captures: the redacted prompt string (post-PII-redaction — never the original unredacted prompt), the full LLM response text, `actor_id` (the staff or system user ID triggering the job), `occurred_at` (UTC), `model_name` (from the request body `"model"` field), `token_count_input` and `token_count_output` (from the Ollama response `"prompt_eval_count"` and `"eval_count"` fields); the interceptor writes one `audit_logs` row per LLM call with `action_type=LLMInvoked`, `entity_type=ai_inference`, `entity_id=job_id` (the Hangfire job ID), `change_summary="model={model_name} tokens_in={N} tokens_out={M} pipeline={ExtractionJob|CodingJob|IntakePipeline}"`; the interceptor writes synchronously (blocks the LLM call response until the audit row is committed, consistent with audit event synchronous write policy from us_048); `audit_logs` rows for LLM calls are retained under the same 6-year policy as all PHI-touching events (AIR-S03, NFR-017, DR-010).

4. **Given** any pgvector similarity search is executed in the extraction or coding pipeline, **When** the SQL query is constructed by the `VectorSearchService`, **Then** every `SELECT` against tables containing document embeddings (`extracted_clinical_data` or the ICD-10/CPT reference index) includes an explicit `WHERE patient_id = @patientId` clause (for patient document chunks) or is restricted to the read-only ICD-10/CPT reference index (which contains no PHI); the `VectorSearchService` constructor requires `patientId` as a mandatory parameter — there is no overload that omits it; a repository-layer unit test for each search method asserts that the generated SQL contains `WHERE patient_id = @patientId` by inspecting the EF Core `FormattedCommandText` or by verifying mock calls; a test fixture with two patients' document embeddings in the same test database verifies that a search for patient A never returns chunks belonging to patient B (AIR-S04, NFR-008, DR-007).

5. **Given** the four AI safety controls (AC-1 through AC-4) are implemented, **When** a full AI pipeline integration test runs against the Codespaces-local Ollama + PostgreSQL stack, **Then** the following assertions all pass: (a) document extraction for patient A completes without any HTTP call to a non-localhost endpoint (verified by inspecting `HttpClient` outbound request log); (b) the LLM prompt captured in `audit_logs` for the test run does NOT contain the test patient's name, DOB, or address literals (verified by substring scan); (c) the pgvector query for patient A's document chunks does NOT return any chunk seeded under patient B's `patient_id` (verified by count assertion); (d) `audit_logs` contains one `LLMInvoked` row per LLM call made during the test run (verified by count match against mock call count); (e) the architecture fitness test from AC-1 passes with zero violations (NFR-010, AIR-S01, AIR-S02, AIR-S03, AIR-S04).

## Edge Cases

- What happens if the `PiiRedactionMiddleware` fails to detect a PII pattern in the prompt (false negative) and a patient name passes through to the LLM? → The middleware is a best-effort safety layer, not a cryptographic guarantee; the primary PHI containment control is local-only Ollama inference (AC-1) — since the LLM runs locally, PHI in the prompt never reaches an external endpoint; the redaction layer reduces the PHI surface area within LLM training exposure and internal log exposure; false negatives are addressed by expanding the PII pattern list through periodic review; unit tests cover 100+ name, DOB, address, and insurance ID variants to minimise undetected patterns; any redaction escape is logged by the `AiAuditInterceptor` (AC-3) for retrospective review.
- What happens if the `AiAuditInterceptor` write to `audit_logs` fails (e.g., DB timeout) — does it block the LLM response? → Per the synchronous audit write policy (us_048 AC-2), the interceptor retries once (immediate); if the retry also fails, the failed audit event is written to the overflow file sink (`/logs/audit-overflow-{Date}.log`) and the LLM call response is returned to the calling pipeline — the AI job is NOT aborted due to audit log failure; the failed audit event is flagged in the overflow file for manual replay after database recovery; this trade-off (availability over audit completeness) is the same policy applied to all audit log failures per us_048.
- What happens if the `VectorSearchService` is called with a `null` or empty `patientId` (e.g., a coding job where the patient ID is lost mid-pipeline)? → The `VectorSearchService` constructor validates `patientId` with a `Guard.Against.NullOrEmpty(patientId, nameof(patientId))` guard clause; a `null` or empty patient ID throws `ArgumentException` immediately, before any database call is made; the Hangfire job catches this exception, marks the job as `Failed`, writes `audit_logs (action=CodingJobFailed|ExtractionFailed, error=MissingPatientId)`, and does NOT execute the pgvector query; this prevents a potentially unconstrained table scan that could return cross-patient data.
- What happens if the architecture fitness test in AC-1 detects a new legitimate external HTTP client added by a developer for a non-AI purpose (e.g., a new calendar sync HTTP client) that happens to have `googleapis.com` in its base address? → The fitness test's HTTP client base address check is scoped specifically to services registered in the `Clinical`, `Coding`, and `Intake` feature assemblies — not the full application; the `Notifications` and `Booking` assemblies (which host Google Calendar and Twilio integrations) are explicitly excluded from the AI safety fitness check scope; the test uses assembly-scoped reflection, not a global scan, so calendar/SMS HTTP clients do not trigger false positives.
- What happens if the Ollama model name changes (e.g., updated to `llama3.2:3b-instruct-q8_1`) and the audit log records stale model names for old jobs? → The `model_name` stored in `audit_logs` is sourced from the request body `"model"` field at call time — it reflects the model actually used for that specific call, not a configuration constant; model name updates only affect new jobs going forward; historical audit records accurately reflect the model version that was used when the job ran; `AIR-O03` (model version tracking per coding result) is handled in the `medical_code_suggestions` table separately (us_044) and serves as the application-level model provenance record; the `audit_logs` entry serves the HIPAA audit trail purpose.

### Requirement Tags

- NFR-010 (System MUST store all PHI solely on the platform's local infrastructure; no PHI MUST be transmitted to any external third-party AI or cloud service)
- AIR-S01 (System MUST ensure all LLM inference is performed exclusively via locally-hosted Ollama; no PHI or patient data transmitted to any external API endpoint)
- AIR-S02 (System MUST redact patient identifiers from LLM prompts before invocation, substituting opaque internal identifiers)
- AIR-S03 (System MUST log every LLM prompt and response payload — excluding direct PHI — to the audit log with actor ID, timestamp, model name, and token counts)
- AIR-S04 (System MUST enforce document ACL filtering in pgvector retrieval queries, ensuring only the target patient's chunks are returned)
- TR-016 (System MUST implement a PII redaction middleware in the AI pipeline that strips patient-identifiable fields from LLM prompts before invocation, substituting opaque internal reference identifiers)
