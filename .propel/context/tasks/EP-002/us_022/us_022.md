# User Story - US_022

## Story ID

- ID Format: US_022

## Story Title

- AI-Assisted Conversational Intake via Ollama Tool Calling

## Description

- As an authenticated patient, I want to complete my pre-appointment intake through a conversational AI chat that understands my natural language responses and incrementally saves each structured field as I answer, so that I can provide my health history in a natural and guided way without filling out a form.

## Acceptance Criteria

1. **Given** an authenticated patient navigates to `/intake` and selects "AI-Assisted" on the Intake Method Selection screen (SCR-005), **When** the AI-Assisted Intake Chat screen (SCR-006) loads, **Then** the API calls `GET /api/intake/status` to retrieve any existing `patient_intakes` row; if one exists, the chat opens with a greeting acknowledging previously captured fields (e.g., "Welcome back — I have your chief complaint on file. Let's continue with medication history"); if none exists, the LLM opens with a greeting and the first intake question about chief complaint.

2. **Given** the patient submits a natural language message in the chat input, **When** the API processes `POST /api/intake/chat {message, session_id}`, **Then** the AI Guard Layer redacts patient identifiers (`[PATIENT-ID]`, `[DOB-REDACTED]` replacing name and date of birth per AIR-S02) before passing the message and conversation history to Ollama; the Ollama `llama3.2:3b-instruct-q8_0` model is invoked with the tool definitions for `save_demographic`, `save_medication`, `save_allergy`, and `save_chief_complaint`; when the LLM emits a tool call, the Guard Layer executes the corresponding tool, calling `UPSERT patient_intakes` (partial update of the specific JSONB field) in PostgreSQL; the API returns `{next_prompt, extracted_fields_so_far}` to the SPA; the SPA renders the AI response bubble and updates the field-completion chips in the sidebar.

3. **Given** the LLM is generating a response (after the patient submits a message), **When** the API call is in-flight, **Then** the SPA renders the TypingIndicator component (3 animated bouncing dots, `aria-label="AI is typing"`, `role="status"`) in the chat log (UXR-504); the indicator is removed and replaced by the AI response bubble as soon as the API returns; the patient's send button is disabled during this window to prevent duplicate submissions.

4. **Given** the LLM inference call is in progress, **When** the request exceeds 4,096 input tokens (conversation history + tool definitions + new message), **Then** the Guard Layer truncates conversation history from the oldest turn first until the total is ≤ 4,096 tokens (AIR-O01) and logs a `WARN` entry to Serilog indicating token budget enforcement; the truncation is transparent to the patient — no error is shown; the intake continues normally.

5. **Given** the Ollama inference endpoint fails 3 consecutive times within a 60-second window, **When** the circuit breaker (AIR-O02) opens, **Then** the API returns HTTP 503 with message "AI intake is temporarily unavailable — please switch to the manual form"; the SPA displays this message inline in the chat log as a system bubble with a "Switch to Manual Form" button; no data already captured is lost; the circuit breaker closes after the 5-minute cool-down period and the AI path becomes available again.

6. **Given** the patient has completed all five required intake sections (demographics, medical history, medications, allergies, chief complaint) and clicks "Review answers →", **When** the SPA navigates to the Intake Review & Summary screen (SCR-007b), **Then** all captured fields are displayed in read-only form for review; a "Confirm & submit intake" button calls `POST /api/intake/submit` which sets `submitted_at = NOW()`; an `audit_logs` entry with `action_type = IntakeSubmitted` is written; the API returns HTTP 200 and the SPA navigates to the Patient Dashboard (SCR-004).

## Edge Cases

- What happens when the LLM returns a malformed tool call (missing required arguments for `save_demographic`)? → The Guard Layer catches the JSON schema validation failure, logs it to Serilog at `WARN`, does NOT persist the partial data, and returns a fallback prompt to the patient: "I didn't quite catch that — could you provide your date of birth in MM/DD/YYYY format?"; the conversation continues without data loss.
- What happens when the patient closes the browser mid-conversation (e.g., phone dies)? → All tool-call saves are UPSERT operations that persist incrementally to `patient_intakes` as each field is collected; on next session start, `GET /api/intake/status` returns the partially saved intake; the AI greeting resumes from the last captured field, preventing duplicate data collection.
- What happens when the patient sends a message containing PHI that the Guard Layer's regex-based redaction misses (e.g., an unusual name format)? → The Guard Layer applies a secondary NER-based pass (spaCy-style pattern matching in .NET) as a defence-in-depth measure; any remaining potential PHI tokens are replaced with `[REDACTED]`; the AI Guard logs the redaction event at `DEBUG` level; the patient-facing experience is unaffected.
