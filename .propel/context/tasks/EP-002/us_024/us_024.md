# User Story - US_024

## Story ID

- ID Format: US_024

## Story Title

- Intake Method Switching with Full Data Preservation

## Description

- As an authenticated patient, I want to switch freely between the AI-assisted chat and the manual form at any point during my intake session so that I can choose the approach that suits me best without losing any data I have already provided.

## Acceptance Criteria

1. **Given** a patient is on the AI-Assisted Intake Chat (SCR-006) with partially captured fields, **When** they click "Switch to manual form" (`#btn-switch-manual`), **Then** the SPA calls `GET /api/intake/current-data`; the API returns all field values currently stored in `patient_intakes` for that patient; the SPA navigates to the Manual Intake Form (SCR-007) with all returned fields pre-filled in the correct form sections; no data captured via the AI path is lost; the patient can continue filling in remaining fields manually.

2. **Given** a patient is on the Manual Intake Form (SCR-007) with partially completed fields, **When** they click "Switch to AI chat" (`#btn-switch-ai`), **Then** the SPA first calls `POST /api/intake/draft {current_form_state}` to persist any unsaved manual edits before switching; after a successful save, the SPA calls `GET /api/intake/current-data` and navigates to the AI-Assisted Intake Chat (SCR-006); the AI greeting acknowledges previously filled fields (e.g., "I can see you've already provided your medications — let me continue with your allergies"); the field-completion chips in the SCR-006 sidebar reflect the pre-filled status of all already-saved sections.

3. **Given** the switch is triggered from SCR-006 to SCR-007 (or vice versa), **When** `GET /api/intake/current-data` returns the existing `patient_intakes` row, **Then** the `intake_method` field in the database is updated to reflect the current mode on the next `POST /api/intake/draft` or `POST /api/intake/submit` call — specifically `intake_method = Manual` after switching to manual, `intake_method = AI` after switching back to AI; the method recorded at submission time reflects the mode in which the patient completed the final section.

4. **Given** the switch involves a mid-session unsaved AI tool call that has not yet been UPSERTed (e.g., the LLM response is still in-flight when the patient clicks "Switch"), **When** the switch is executed, **Then** the in-flight API call is aborted by the SPA using an `AbortController`; the SPA proceeds immediately to `GET /api/intake/current-data` using the last successfully persisted state; no partial or uncommitted LLM output is surfaced in the manual form; the patient is not shown an error.

5. **Given** either intake screen (SCR-006 or SCR-007), **When** the patient has already completed all five sections and all fields are saved, **Then** the "Switch" button is still visible and functional (patient can still switch modes for review or re-entry); switching at this point routes the patient to the other screen pre-filled with all five sections complete; the patient can proceed directly to "Review answers →" (SCR-007b) from either screen.

## Edge Cases

- What happens when `GET /api/intake/current-data` returns an empty response (no `patient_intakes` row exists for this patient yet)? → The target screen (SCR-007 manual or SCR-006 AI) opens with all fields empty — the same initial state as if the patient had entered via the normal intake method selection flow; no error is shown; the patient can begin from scratch in the new mode.
- What happens when the `POST /api/intake/draft` (save-before-switch) call fails due to a temporary network error? → The SPA shows a toast notification "Could not save your progress before switching — please try again"; the switch is cancelled and the patient remains on the current screen; their in-memory form state is not lost; they can retry switching after the network recovers.
- What happens when the patient switches modes more than once in the same session (AI → Manual → AI → Manual)? → Each switch calls `GET /api/intake/current-data` to load the latest persisted state; the `intake_method` enum is updated on the next save; no maximum switch count is enforced; all field values accumulated across all modes are preserved across every switch; the only record of the final mode is the `intake_method` value at submission.
