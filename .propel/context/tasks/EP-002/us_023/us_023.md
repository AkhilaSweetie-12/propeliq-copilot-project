# User Story - US_023

## Story ID

- ID Format: US_023

## Story Title

- Manual Form Intake with 60-Second Auto-Save

## Description

- As an authenticated patient, I want to complete my pre-appointment intake using a structured manual form that saves my progress automatically every 60 seconds so that I can fill in my health information at my own pace without losing data if my session times out.

## Acceptance Criteria

1. **Given** an authenticated patient selects "Manual Form" on the Intake Method Selection screen (SCR-005), **When** the Manual Intake Form (SCR-007) loads, **Then** the SPA calls `GET /api/intake/status`; if a partial `patient_intakes` row exists, all previously saved field values are pre-filled in the form sections (demographics, medical history, medications, allergies, chief complaint); a progress indicator shows the fraction of sections completed (e.g., "3 / 7 sections completed") — matching the SCR-007 wireframe.

2. **Given** the patient is on SCR-007 with unsaved changes, **When** 60 seconds elapse since the last auto-save (or the last user interaction that reset the timer), **Then** the SPA calls `POST /api/intake/draft {partial_fields}` which UPSERTs the current form values to `patient_intakes` (setting `intake_method = Manual`); on success, the topbar auto-save indicator updates to "✓ Saved X seconds ago" (UXR-004); on failure (network error), the indicator shows "⚠ Save failed — retrying…" and the SPA retries once after 5 seconds; no user interaction is required to trigger the save.

3. **Given** the patient has filled all required fields (demographics, medical history, medications, allergies, chief complaint) on SCR-007 and clicks "Review answers →", **When** the SPA navigates to the Intake Review & Summary screen (SCR-007b), **Then** all field values are displayed in read-only grouped sections (one per intake category); each section has an "Edit" link that navigates back to SCR-007 focused on that section; the "Confirm & submit intake" button calls `POST /api/intake/submit`, sets `submitted_at = NOW()`, writes an `audit_logs` entry with `action_type = IntakeSubmitted`, and navigates to the Patient Dashboard (SCR-004).

4. **Given** a patient fills a field and moves focus away (blur event), **When** the field value fails client-side validation (e.g., a required field is empty, or date of birth is in the future), **Then** the field renders a red border, an error icon, and a descriptive message beneath the field (UXR-601 — "Date of birth must be in the past"); the form submit button remains disabled until all required fields pass validation; the API is NOT called for validation — it is performed exclusively client-side on blur.

5. **Given** the patient is on the Manual Intake Form (SCR-007), **When** they click the "Switch to AI chat" button in the page header, **Then** the SPA triggers the intake method switch flow (handled by us_024); all data currently entered in the form is preserved and carried into the switch.

6. **Given** the patient has previously submitted the intake (i.e., `submitted_at` is set) and returns to `/intake`, **When** the Manual Intake Form loads, **Then** all fields are pre-filled from the existing `patient_intakes` row; the form is in edit mode (not locked); the patient can modify any field and resubmit — this triggers `PUT /api/intake {updated_fields}` (handled in full by us_025); the "Confirm & submit intake" button on SCR-007b calls `PUT /api/intake/submit` updating `updated_at = NOW()` and writes an `audit_logs` entry with `action_type = IntakeUpdated`.

## Edge Cases

- What happens when the 60-second auto-save `POST /api/intake/draft` returns HTTP 401 (JWT expired mid-session)? → The SPA interceptor catches the 401, attempts a token refresh via `POST /api/auth/refresh`; if refresh succeeds, the auto-save is retried transparently; if the refresh token is also expired, the session expiry modal (UXR-503) appears — the patient's unsaved changes since the last successful auto-save are preserved in React state and will be re-submitted once they extend the session.
- What happens when a very large free-text field (e.g., medical history) causes the auto-save payload to exceed the API's request size limit? → The API applies a 64 KB JSON body limit (configured in ASP.NET Core middleware per DR-005); if exceeded, the API returns HTTP 413; the SPA shows an inline error "Your entry is too long — please shorten the medical history field"; the previous successful auto-save data is preserved in the database.
- What happens when the patient submits the manual form without completing any AI-assisted fields from a previous AI session? → The `patient_intakes` row stores `intake_method` as an enum; on manual submission, `intake_method` is set to `Manual` (overwriting the previous `AI` value); all previously AI-extracted field values remain in the JSONB columns and are pre-filled in the manual form; the patient can overwrite or keep them; the submitted intake reflects the final confirmed values regardless of original source.
