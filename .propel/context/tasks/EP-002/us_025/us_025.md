# User Story - US_025

## Story ID

- ID Format: US_025

## Story Title

- Patient-Initiated Intake Edit Without Staff Intervention

## Description

- As an authenticated patient who has previously submitted my intake, I want to update any intake field at any time from my dashboard without needing to contact or wait for Staff approval so that I can keep my health information current ahead of my appointment.

## Acceptance Criteria

1. **Given** a patient navigates to `/intake` and a `patient_intakes` row with `submitted_at` set already exists for them, **When** the Manual Intake Form (SCR-007) loads, **Then** the form displays in edit mode: all previously submitted field values are pre-filled; a page subtitle reads "Update your intake — changes are saved automatically and applied immediately"; no lock state or "request edit" step is required; the patient can begin modifying any field immediately.

2. **Given** the patient edits one or more fields in the Manual Intake Form and clicks "Review answers →", **When** the Intake Review & Summary screen (SCR-007b) is displayed, **Then** it shows the updated field values alongside the original submission date; the "Confirm & submit intake" button calls `PUT /api/intake/{intake_id} {updated_fields}`; the API updates the matching `patient_intakes` row (`SET updated_at = NOW()`, preserves `submitted_at`); an `audit_logs` entry with `action_type = IntakeUpdated` capturing `actor_id`, `actor_role = Patient`, `entity_type = PatientIntake`, `entity_id = intake_id`, and a `change_summary` listing the field names that changed is written; the API returns HTTP 200; the SPA navigates to the Patient Dashboard (SCR-004) with a success toast "Intake updated successfully."

3. **Given** a Staff or Admin user attempts to call `PUT /api/intake/{intake_id}` for a `patient_intakes` row they do not own (i.e., `patient_id ≠ caller's user_id`), **When** the API processes the request, **Then** the API returns HTTP 403 Forbidden; only the owning patient (authenticated user whose `user_id` matches the row's `patient_id`) can edit their own intake; Staff can read intake data for clinical purposes but cannot submit edits on behalf of a patient through this endpoint.

4. **Given** the patient edits their intake after a clinical document has already been extracted and a `patient_view_360` row exists, **When** the `PUT /api/intake/{intake_id}` call completes successfully, **Then** the API enqueues a Hangfire background job (`RefreshPatientView360Job`) that re-aggregates the updated intake data into `patient_view_360.aggregated_data`; this job runs asynchronously — the patient's save confirmation is NOT blocked by the re-aggregation; the patient receives no UI indication of the background job (it is a silent update).

5. **Given** the patient saves an intake edit via the 60-second auto-save (`POST /api/intake/draft`) while in edit mode, **When** the auto-save API call completes, **Then** the draft endpoint writes a partial update to `patient_intakes` using the same UPSERT pattern as the initial intake; no `audit_logs` entry is written for draft saves — only for confirmed submit/update (`POST /api/intake/submit` or `PUT /api/intake/{intake_id}`); this prevents the audit log from being flooded with incremental draft events.

## Edge Cases

- What happens when the patient starts editing and then navigates away from SCR-007 without submitting (e.g., clicks "Dashboard" in the sidebar)? → The SPA displays a browser `beforeunload` confirmation "You have unsaved changes — are you sure you want to leave?"; if confirmed, the current auto-saved draft (written by the last 60-second timer cycle) remains in `patient_intakes`; the `submitted_at` timestamp from the previous confirmed submission is preserved; the patient can return to `/intake` and their draft edits will be pre-filled.
- What happens when two browser sessions (e.g., phone and laptop) submit concurrent edits to the same intake? → The API uses an optimistic concurrency check via an `updated_at` timestamp in the PUT payload; if the client's `updated_at` does not match the database value (another session already updated it), the API returns HTTP 409 Conflict with message "Your intake was updated in another session — please refresh to see the latest version"; no data is silently overwritten.
- What happens when a patient tries to clear a previously submitted required field (e.g., deletes the chief complaint) during an edit? → Client-side blur validation (UXR-601) prevents submission of a blank required field and shows an inline error "Chief complaint is required"; the "Confirm & submit intake" button remains disabled; the API also enforces NOT NULL constraints on `chief_complaint` as a secondary guard, returning HTTP 400 if the field is empty on a PUT call.
