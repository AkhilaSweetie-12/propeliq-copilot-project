# User Story - US_047

## Story ID

- ID Format: US_047

## Story Title

- PHI Column-Level Encryption (AES-256 at Rest)

## Description

- As the platform security team, I want all Protected Health Information stored in PostgreSQL to be encrypted at the column level using AES-256, so that even if the database files or backups are accessed without authorisation, patient PHI is unreadable without the application-level encryption key — satisfying the HIPAA Security Rule's encryption and decryption technical safeguard at rest.

## Acceptance Criteria

1. **Given** the PHI field inventory is defined, **When** the encryption implementation is applied, **Then** the following columns across all entities are encrypted using AES-256 (application-layer transparent encryption via an EF Core value converter backed by `System.Security.Cryptography.Aes`): `users.email` (encrypted, but a separate deterministic SHA-256 HMAC hash of the email is stored in a parallel `email_hash` column to preserve unique-constraint lookups without exposing plaintext), `patient_intakes.demographics` (JSONB), `patient_intakes.medical_history` (JSONB), `patient_intakes.medications` (JSONB), `patient_intakes.allergies` (JSONB), `patient_intakes.chief_complaint` (TEXT), `extracted_clinical_data.field_value` (TEXT), `patient_view_360.aggregated_data` (JSONB); the AES encryption key is a 256-bit key loaded from the environment variable `PHI_ENCRYPTION_KEY` (base64-encoded) — never hardcoded; IV (initialisation vector) is generated freshly per encryption operation and prepended to the ciphertext; encrypted values are stored as Base64-encoded `TEXT` columns (NOT in binary `BYTEA` to preserve EF Core compatibility) (FR-037, NFR-007).

2. **Given** the EF Core value converters are implemented, **When** the application reads or writes a PHI entity, **Then** encryption is transparent to the rest of the application: all repository and service classes operate on plaintext values; the EF Core `ValueConverter<string, string>` registered for each PHI column automatically encrypts on write and decrypts on read; no service or controller class calls `Aes.Encrypt()` or `Aes.Decrypt()` directly — all encryption is handled through the value converter; a unit test for each encrypted entity verifies that: (a) writing a PHI value and reading it back returns the original plaintext, (b) the raw database column value is not equal to the plaintext input (confirming encryption was applied), (c) a different IV is used for each encryption call so identical plaintext produces different ciphertext (FR-037, NFR-007).

3. **Given** the `users.email` field is encrypted and the authentication flow requires a unique email lookup, **When** a user registers or logs in with their email address, **Then** the login query does NOT perform a plaintext email scan against the encrypted column (which would require decrypting every row); instead, the authentication service computes the HMAC-SHA-256 of the provided email address (using a separate `EMAIL_HMAC_KEY` environment variable) and queries `WHERE email_hash = computed_hmac`; this produces an O(1) indexed lookup without decrypting the full table; the `email_hash` column has a UNIQUE constraint enforced at the database level; the `email_hash` column is NOT considered PHI (it is a keyed hash, not reversible without the HMAC key) and is NOT encrypted; the HMAC key is stored as an environment variable separate from the AES key (NFR-007, FR-037, NFR-005).

4. **Given** the PHI encryption is applied, **When** the EF Core migration for this story runs, **Then** all existing rows in each encrypted table are migrated to ciphertext: the migration script reads each row in batches of 1,000, encrypts the PHI column values using the `PHI_ENCRYPTION_KEY` from the migration environment, writes the encrypted values back, and verifies the row count matches pre- and post-migration; a compensating rollback migration is provided that does NOT decrypt (to prevent accidental PHI exposure on rollback — a rollback requires reseeding from backup); the `PHI_ENCRYPTION_KEY` environment variable is validated to be exactly 32 bytes (256 bits) on application startup: if the key is absent or the wrong length, the application refuses to start and logs `"Fatal: PHI_ENCRYPTION_KEY must be 32 bytes — application cannot start without valid encryption configuration"` (FR-037, NFR-007, NFR-005).

5. **Given** the encryption implementation is complete, **When** a penetration test or security integration test reads a PostgreSQL row directly from `psql` (bypassing the application), **Then** all PHI columns contain only ciphertext (Base64-encoded strings); no plaintext patient name, email, demographics, or clinical data is readable from the raw database; a dedicated CI security test connects to the test PostgreSQL instance with a read-only database user, queries the five encrypted tables (`users`, `patient_intakes`, `extracted_clinical_data`, `patient_view_360`, and a sample `clinical_documents` row), and asserts that none of the returned column values match the known test fixture PHI values; this test runs in the CI pipeline on every merge to the main branch (FR-035, FR-037, NFR-005, NFR-007).

## Edge Cases

- What happens if the `PHI_ENCRYPTION_KEY` environment variable is rotated (new key deployed)? → Phase 1 does not implement key rotation (key rotation is a Phase 2 concern); the key is set once at deployment and immutable for the life of the Phase 1 deployment; if a key rotation is ever needed, all encrypted rows must be re-encrypted using the data migration pattern from AC-4; the key rotation risk is documented in the security runbook; the application startup validation (AC-4) ensures the key is always present and the correct length before any PHI operation runs.
- What happens if decryption fails for a specific row (e.g., ciphertext corruption from a partial database write)? → The `AesValueConverter` catches `CryptographicException` per column; a failed decryption returns `null` for the affected field and writes `audit_logs (action=DecryptionFailure, entity_type, entity_id, column_name)` synchronously; the application does not surface the raw ciphertext to the client; the affected record is marked as requiring data recovery; the partial decryption failure is logged at Error level to Serilog so the operations team can identify and restore from the 7-day backup window (DR-013).
- What happens if the AES-256 encryption of a large JSONB field (e.g., `patient_view_360.aggregated_data` with many documents) produces a ciphertext larger than the `TEXT` column can store? → PostgreSQL `TEXT` columns have no maximum length limit (up to 1 GB); the Base64 encoding adds approximately 33% overhead to the ciphertext length; even a 500 KB aggregated_data JSONB would be stored as a ~670 KB Base64 text — well within PostgreSQL TEXT limits; no length truncation risk exists for the expected Phase 1 data volumes (max ~20 documents per patient, ~5 fields each).
- What happens if two simultaneous writes encrypt the same PHI value differently (due to fresh IV per call) and a query tries to match on encrypted value? → Only `users.email` requires query-time matching, which is handled via the deterministic HMAC hash (AC-3); all other encrypted PHI columns (demographics, medications, field_value, aggregated_data) are never queried by value — they are always retrieved by `patient_id` or `document_id` foreign key; therefore, non-deterministic encryption (fresh IV per call) is safe for all PHI columns except email, which uses the separate HMAC index.
- What happens in automated tests that need to assert on PHI field values but the fields are encrypted in the test database? → Test fixtures use the same `PHI_ENCRYPTION_KEY` set in the CI environment variable; the EF Core value converters apply transparently in tests (the in-memory EF Core provider does NOT encrypt — encryption only activates with the real Npgsql provider connecting to PostgreSQL); integration tests targeting the real PostgreSQL test instance can read decrypted values through the application layer (EF Core), while the raw-database CI test in AC-5 uses a separate read-only database user to verify ciphertext; this dual-layer test strategy covers both business logic and storage-layer encryption.
