# User Story - US_048

## Story ID

- ID Format: US_048

## Story Title

- Immutable Audit Log & Structured Logging (Serilog + Seq)

## Description

- As the platform compliance officer, I want every PHI-touching create, read, update, and delete operation to be recorded in a complete, append-only, immutable audit log retained for 6 years, and all application and audit events to be written to a structured Serilog + Seq logging pipeline, so that the platform satisfies the HIPAA Security Rule's audit control standard and provides a complete, tamper-evident chain of custody for every patient data access.

## Acceptance Criteria

1. **Given** the Serilog and Seq logging infrastructure is configured, **When** the ASP.NET Core 9 application starts, **Then** Serilog is registered as the application's `ILogger` provider via `UseSerilog()` in `Program.cs`; Serilog sinks include: (a) Seq community server sink (`WriteTo.Seq("http://seq:5341")` — Seq running as a devcontainer service on port 5341) for structured log indexing and search, (b) Console sink (structured JSON format) for container log aggregation, and (c) File sink (`/logs/app-{Date}.log`, rolling daily, 7-day retention) for local log backup; audit events are written using a dedicated `AuditLogger` service that calls `Log.Write()` synchronously using `LogEventLevel.Warning` to ensure audit events are never dropped by async buffering; application diagnostics (request logging, performance traces) are written asynchronously via the default Serilog internal queue; minimum log level is `Information` for application logs and `Warning` for audit events (TR-015, NFR-017).

2. **Given** any PHI-touching operation occurs in the system, **When** the operation completes (success or failure), **Then** the `AuditLogger` service writes one `audit_logs` row synchronously with the following fields populated: `log_id` (UUID v4 generated by application), `actor_id` (authenticated user UUID from JWT claims — `AnonymousUser` UUID for unauthenticated requests), `actor_role` (Patient | Staff | Admin | System from JWT claims), `action_type` (one of: PatientRegistered, PatientLogin, PatientLogout, IntakeSubmitted, IntakeRead, AppointmentCreated, AppointmentRead, AppointmentCancelled, DocumentUploaded, DocumentRead, ExtractionCompleted, ExtractionFailed, PatientView360Accessed, ConflictAcknowledged, ViewVerified, CodingRequested, CodeReviewed, CodingFinalised, CodeSuggestionGenerated, DeduplicationApplied, ConflictDetected, ViewResetUnverified, LowConfidenceRetrieval, EmbeddingFailed, CodingJobFailed, CodingBudgetExceeded, DecryptionFailure, RedisPhiViolation, AdminUserCreated, AdminUserDeactivated, AdminRoleChanged), `entity_type` (users | patient_intakes | appointments | clinical_documents | extracted_clinical_data | patient_view_360 | medical_code_suggestions | audit_logs), `entity_id` (UUID of the affected record), `change_summary` (human-readable description, max 500 chars, NO PHI values — only field names and action context), `ip_address` (from `HttpContext.Connection.RemoteIpAddress`, with X-Forwarded-For header respected for proxy scenarios), `occurred_at` (UTC timestamp, `DateTimeOffset.UtcNow`) (NFR-017, DR-010, FR-035).

3. **Given** the `audit_logs` table is migrated, **When** the database migration runs, **Then** the migration applies the following database-level immutability controls to the `audit_logs` table: (a) a PostgreSQL `RULE` or `TRIGGER` `BEFORE UPDATE ON audit_logs` that raises `EXCEPTION 'Audit logs are immutable — UPDATE is prohibited'` is created, (b) a PostgreSQL `RULE` or `TRIGGER` `BEFORE DELETE ON audit_logs` that raises `EXCEPTION 'Audit logs are immutable — DELETE is prohibited'` is created, (c) the `audit_log_writer` application database role is granted only `INSERT` and `SELECT` on `audit_logs` — no `UPDATE` or `DELETE` permissions are granted to any application role; the application database connection string uses the `audit_log_writer` role for audit log operations; a separate `app_writer` role handles all other table writes; this database-level enforcement ensures that even a compromised application layer cannot delete or overwrite audit records (DR-010, NFR-017, NFR-005).

4. **Given** PHI-touching events are written to audit_logs, **When** the `change_summary` field is populated, **Then** the summary contains ONLY: the action type, entity type, entity ID, and contextual metadata (e.g., "Staff accessed patient view for patient_id=<UUID>", "Appointment status changed from Confirmed to Arrived for appointment_id=<UUID>") — it MUST NOT contain any PHI values such as patient names, email addresses, diagnosis text, medication names, or clinical field values; a Serilog enricher applies a PHI scrubber that checks `change_summary` values against a pattern list before write: patterns that match `"[A-Z][a-z]+ [A-Z][a-z]+"` (name pattern), `"[^@]+@[^@]+"` (email pattern), or known medication name prefixes are replaced with `[REDACTED]`; the PHI scrubber is covered by unit tests with sample PHI and non-PHI summaries (NFR-005, NFR-017, AIR-S03).

5. **Given** the 6-year retention policy is defined, **When** the `audit_logs` table grows over time, **Then** a PostgreSQL table partition strategy is implemented: `audit_logs` is partitioned by year (`PARTITION BY RANGE (occurred_at)`) with one partition per calendar year; the current-year partition is `NOT NULL` (write-active); partitions older than 6 years (72 months from `occurred_at`) are eligible for archival (export to compressed cold storage), not deletion — the HIPAA 6-year retention minimum (45 CFR § 164.530(j)) requires retention, not active querying; an automated check runs on first day of each calendar year to create the new year partition and identify partitions eligible for archival; in Phase 1 (single-year deployment), only the initial year partition is created; no partition is ever dropped (NFR-017, DR-012).

## Edge Cases

- What happens if the `AuditLogger` synchronous `INSERT` to `audit_logs` fails (e.g., database connectivity loss)? → The `AuditLogger` catches the database exception; it immediately retries once (immediate, no delay); if the retry also fails, the failed audit event is written to the file sink (`/logs/audit-overflow-{Date}.log`) as a structured JSON entry with full event details; the overflow file is monitored by operations and replayed manually after database recovery; the primary operation (e.g., patient login) is NOT rolled back due to audit log failure — the patient action succeeds; this is the HIPAA-accepted trade-off for availability vs. audit completeness; all overflow events are marked for replay in the operations runbook.
- What happens if the Seq community server is unavailable (e.g., container restart)? → Seq unavailability does NOT affect `audit_logs` database writes — the `AuditLogger` writes directly to PostgreSQL (AC-2, AC-3) regardless of Seq availability; the Seq sink in Serilog uses a durable buffer (`bufferBaseFilename` set to `/logs/serilog-seq-buffer`) so that events written while Seq is unavailable are buffered on disk and replayed to Seq when it recovers; the console and file sinks continue to receive all log events during Seq downtime; application functionality is not impacted by Seq unavailability.
- What happens if the `BEFORE UPDATE` trigger on `audit_logs` conflicts with EF Core's change tracking (e.g., EF Core attempts an UPDATE during an UPSERT operation)? → The `AuditLogger` only ever calls `dbContext.AuditLogs.Add()` followed by `dbContext.SaveChanges()` — never `Update()` or UPSERT; EF Core's change tracker is configured with `AsNoTracking()` for all audit log reads to prevent EF Core from tracking audit log entities and attempting updates; the trigger serves as a database-level backstop, not a primary defence; an integration test asserts that calling `dbContext.AuditLogs.Update()` results in an exception (confirming the trigger fires), and that normal `Add()` operations succeed.
- What happens if a high-throughput operation (e.g., batch document upload for 50 documents) generates 50+ synchronous audit log writes in rapid succession? → The synchronous writes are batched within a single database transaction where possible (AC-2 specifies synchronous write, not isolated transaction per event); for batch operations the `AuditLogger` accepts a `List<AuditEvent>` and performs a single `INSERT ... SELECT` multi-row insert within the batch operation's transaction; this reduces per-write overhead from 50 round trips to 1; the synchronous guarantee is maintained (the transaction commits only after all audit events are written); individual single-operation paths still use single-row inserts.
- What happens if the `change_summary` PHI scrubber in AC-4 incorrectly redacts a legitimate non-PHI value (e.g., "appointment_id=<UUID-that-looks-like-a-name>")? → The PHI scrubber's patterns are specifically targeted: the name pattern `[A-Z][a-z]+ [A-Z][a-z]+` only matches strings with two consecutive Title-Case words (space-separated) — UUID strings (hex + hyphens) do not match; the email pattern only matches strings containing `@`; medication name prefixes are a whitelist of ~50 common generic names loaded from a configuration file; the scrubber is conservative (prefer false negative over false positive for audit readability) and is covered by unit tests with 200+ sample values including UUIDs, timestamps, and entity IDs to confirm zero false positives on structural identifiers.
