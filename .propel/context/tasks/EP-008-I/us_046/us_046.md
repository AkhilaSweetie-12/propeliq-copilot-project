# User Story - US_046

## Story ID

- ID Format: US_046

## Story Title

- TLS 1.2+ Enforcement & Redis PHI Prohibition

## Description

- As the platform security team, I want all API traffic to be encrypted in transit using TLS 1.2 or higher with automatic rejection of plaintext HTTP, and all Redis cache operations to be strictly prohibited from storing any Protected Health Information, so that the platform satisfies the HIPAA Security Rule's transmission security and minimum necessary access standards without relying on application code correctness to protect PHI in transit or cache.

## Acceptance Criteria

1. **Given** the ASP.NET Core 9 API starts up, **When** any HTTP request arrives on a non-TLS port, **Then** the HTTPS redirection middleware (`UseHttpsRedirection`) redirects the request with HTTP 301 to the HTTPS equivalent; the `UseHsts` middleware adds an `Strict-Transport-Security: max-age=31536000; includeSubDomains` response header to all HTTPS responses; no API endpoint accepts or processes a plaintext HTTP request body containing patient data — the redirect occurs before the request body is parsed; HTTPS-only enforcement is applied in all environments (Development, Staging, Production) with a self-signed or Let's Encrypt certificate configured in the devcontainer (NFR-006, FR-036).

2. **Given** the ASP.NET Core Kestrel server is configured, **When** TLS connections are negotiated, **Then** Kestrel's TLS configuration explicitly allows only TLS 1.2 and TLS 1.3 by setting `SslProtocols = SslProtocols.Tls12 | SslProtocols.Tls13` in the `ConfigureKestrel` block; TLS 1.0 and TLS 1.1 connections are rejected at the transport layer (connection closed, no response body); a cipher suite allowlist is configured that excludes `NULL`, `RC4`, `DES`, `3DES`, and `EXPORT` cipher suites per NIST SP 800-52r2; the TLS configuration is applied in `Program.cs` and validated by an integration test that asserts TLS 1.0/1.1 connections are refused (NFR-006, FR-036).

3. **Given** any application service (authentication, intake, booking, clinical, coding) performs a write operation to Upstash Redis, **When** the write is executed, **Then** only the following value types are permitted in Redis keys: JWT access token identifiers (opaque UUIDs, not containing PHI claims), refresh token identifiers (opaque UUIDs), and session lifecycle metadata (expiry timestamps); the following data types are explicitly NEVER written to any Redis key or value: patient names, email addresses, demographics JSONB, intake fields, clinical document content, extracted field values, 360° aggregated data, medical code suggestions, or any other PHI as defined under HIPAA 45 CFR § 164.514; a `RedisPhiGuard` middleware/service wrapper inspects value content before write: if the value length exceeds 128 bytes or contains a known PHI pattern (JSON object with `email`, `demographics`, `medications` keys), the write is rejected and `audit_logs (action=RedisPhiViolation, actor_id, attempted_key)` is written; the violation is also logged at Error level to Serilog (FR-038, NFR-005, NFR-010).

4. **Given** the JWT authentication flow issues or revokes a token, **When** the access token or refresh token is stored or checked in Redis, **Then** only the opaque token UUID (not the JWT payload) is used as the Redis key; the Redis value stores only token metadata: `{ expires_at: ISO8601, role: "Patient|Staff|Admin", revoked: bool }` — no PHI fields (name, email, patient data) appear in any Redis value; the JWT payload itself is signed (HS256 minimum, RS256 preferred) and contains only non-PHI claims: `sub` (UUID), `role`, `iat`, `exp`; PHI fields such as patient name or email are NOT embedded in JWT claims (TR-006, NFR-009, FR-038).

5. **Given** the TLS and Redis configurations are deployed, **When** a security integration test suite runs, **Then** the following automated assertions pass: (a) `GET http://{api-host}/health` returns HTTP 301 (TLS redirect, not 200); (b) `GET https://{api-host}/health` returns HTTP 200 with `Strict-Transport-Security` header present; (c) a TLS 1.0 connection attempt to the API port is rejected (socket closed, no HTTP response); (d) a Redis GET for any key that was written during a full booking and intake workflow does not return a value containing any of the PHI patterns defined in AC-3; (e) all assertions pass in both CI/CD pipeline runs and GitHub Codespaces devcontainer environment (NFR-005, NFR-006, FR-035, FR-036, FR-038).

## Edge Cases

- What happens if the HTTPS redirect creates an infinite redirect loop in development because the devcontainer self-signed certificate is not trusted by the test runner? → The `UseHttpsRedirection` middleware is configured with the development certificate via `dotnet dev-certs https --trust`; the integration test suite trusts the development certificate explicitly via `HttpClientHandler.ServerCertificateCustomValidationCallback` (allowed only in test context, never in production code); a `DangerousAcceptAnyServerCertificateValidator` is explicitly prohibited in production builds; a CI linter check flags any production usage of certificate bypass callbacks.
- What happens if the `RedisPhiGuard` in AC-3 produces false positives (e.g., a valid token UUID is flagged as PHI because it resembles a JSON key)? → The guard performs structural inspection: values are only checked if they can be deserialized as JSON objects — opaque UUID strings and short token metadata objects with only `expires_at`, `role`, `revoked` fields are explicitly whitelisted patterns; string values shorter than 128 bytes that match the UUID regex pattern `[0-9a-f-]{36}` are never flagged; the guard is covered by unit tests with PHI and non-PHI sample values to confirm false positive rate = 0 for valid token operations.
- What happens if a developer accidentally writes a patient email to Redis in a new feature branch? → The `RedisPhiGuard` rejects the write at runtime; the violation is recorded in `audit_logs` and logged as Error; the CI integration test that verifies no PHI in Redis after a full workflow run will catch this regression before merge; a code review rule (enforced by the code review instructions) flags any direct Redis write that does not route through the `RedisPhiGuard` wrapper.
- What happens if the Upstash Redis free tier connection is temporarily unavailable? → Redis unavailability affects only JWT blocklist checks and session revocation (TR-005, TR-006); it does NOT affect PHI storage since PHI is never in Redis; the authentication middleware falls back to accepting non-revoked tokens (fail-open for availability, per NFR-014 graceful degradation requirement); revoked tokens that cannot be checked against the unavailable blocklist may temporarily remain valid until TTL expiry (maximum 15 minutes per NFR-009) — this is the accepted trade-off for a free-tier Redis dependency.
- What happens if the HSTS header causes the browser to refuse HTTP access to the application during local development? → The `UseHsts` middleware is conditionally applied only in `!env.IsDevelopment()` environments; in the GitHub Codespaces development environment, HSTS is NOT applied to avoid preloading issues with localhost; HSTS is only enforced in the Staging and Production environments; the TLS 1.2+ cipher restriction and HTTPS redirect are still applied in Development but without the HSTS preload header.
