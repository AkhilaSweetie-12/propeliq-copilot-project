# User Story - US_008

## Story ID

- ID Format: US_008

## Story Title

- `appointment_slots` & `appointments` Entities

## Description

- As a developer, I want the `appointment_slots` and `appointments` EF Core entities and migrations defined with all required fields, foreign keys, status enums, and a nullable preferred-slot self-reference, so that the booking, slot-swap, and arrival-management features have a fully relational, constraint-enforced persistence layer ready to use.

## Acceptance Criteria

1. **Given** the EF Core migration for `appointment_slots` runs against PostgreSQL, **When** `dotnet ef database update` completes, **Then** the table has columns: `slot_id` (UUID PK, default `gen_random_uuid()`), `slot_date` (DATE, NOT NULL), `slot_time` (TIME, NOT NULL), `duration_minutes` (INTEGER, NOT NULL, CHECK > 0), `is_available` (BOOLEAN, NOT NULL, default `TRUE`), `created_at` (TIMESTAMPTZ, NOT NULL); a composite UNIQUE constraint on `(slot_date, slot_time)` prevents duplicate slot definitions.

2. **Given** the EF Core migration for `appointments` runs, **When** the migration completes, **Then** the table has columns: `appointment_id` (UUID PK), `patient_id` (UUID, FK â†’ `users.user_id`, NOT NULL), `slot_id` (UUID, FK â†’ `appointment_slots.slot_id`, NOT NULL), `status` (TEXT CHECK IN ('Confirmed','Arrived','Cancelled','NoShow'), NOT NULL), `preferred_slot_id` (UUID, FK â†’ `appointment_slots.slot_id`, nullable), `booking_type` (TEXT CHECK IN ('Online','WalkIn'), NOT NULL), `created_by` (UUID, FK â†’ `users.user_id`, NOT NULL), `created_at` (TIMESTAMPTZ, NOT NULL), `updated_at` (TIMESTAMPTZ, NOT NULL).

3. **Given** the `preferred_slot_id` nullable self-referencing FK on `appointments`, **When** a new `Appointment` entity is created without a preferred slot, **Then** `preferred_slot_id` persists as `NULL`; when an appointment is created with a preferred slot, `preferred_slot_id` is set to a valid `appointment_slots.slot_id` value and the FK constraint is enforced â€” an invalid UUID returns a `DbUpdateException`.

4. **Given** the `SELECT FOR UPDATE` locking pattern required by NFR-013, **When** a repository method acquires a row lock on an `appointment_slots` row via `context.AppointmentSlots.FromSqlRaw("SELECT ... FOR UPDATE")`, **Then** the lock is held for the duration of the transaction and concurrent attempts to update `is_available` on the same row block until the first transaction commits or rolls back; verified by a concurrent integration test using two parallel `Task.Run` calls.

5. **Given** the `appointments` entity has a CASCADE rule on `slot_id`, **When** a soft-delete of an `appointment_slot` is triggered (setting `is_available = FALSE`), **Then** the associated `appointments` rows are NOT cascade-deleted; referential integrity is preserved and the appointment record retains its `slot_id` FK for historical traceability.

6. **Given** the `updated_at` column on `appointments`, **When** any field on the `Appointment` entity is modified and `SaveChangesAsync()` is called, **Then** the `SaveChanges` interceptor (established in us_007) automatically updates `updated_at` to `DateTimeOffset.UtcNow`; the `appointment_slots` table does not have an `updated_at` column (it is immutable after creation except for `is_available`).

## Edge Cases

- What happens when an appointment is created pointing to a `slot_id` that does not exist? â†’ The FK constraint on `appointments.slot_id` causes PostgreSQL to raise a foreign key violation; EF Core surfaces a `DbUpdateException` with the constraint name; the service layer maps this to a domain `SlotNotFoundException` before returning to the caller.
- What happens when `booking_type` is 'WalkIn' but `patient_id` points to a guest (non-persisted patient)? â†’ Walk-in guest bookings create a minimal `users` row with `role = 'Patient'` and `status = 'Active'` before creating the appointment (guest flow covered in EP-004); the FK constraint is always satisfied before the appointment row is inserted.
- What happens when two threads race to set `is_available = FALSE` on the same slot after the `SELECT FOR UPDATE` lock is implemented? â†’ The `SELECT FOR UPDATE` row-level lock serialises the two transactions; the second transaction waits until the first commits, then reads the updated `is_available = FALSE` value and does not double-assign the slot.
