# User Story - US_011

## Story ID

- ID Format: US_011

## Story Title

- Insurance Dummy Seed Data & Migration Startup Verification

## Description

- As a developer, I want a versioned seed-data migration that populates the internal insurance dummy records table and a startup verification test that confirms all EP-DATA-I migrations applied correctly, so that the insurance pre-check feature has static reference data available from the first deployment and no schema drift goes undetected across environments.

## Acceptance Criteria

1. **Given** an EF Core migration named `SeedInsuranceDummyRecords` is authored, **When** `dotnet ef database update` runs in any environment, **Then** an `insurance_records` table is created with columns: `record_id` (UUID PK), `provider_name` (TEXT, NOT NULL), `insurance_id_prefix` (TEXT, NOT NULL), `created_at` (TIMESTAMPTZ, NOT NULL); and at least 10 pre-defined dummy insurance records (e.g., `BlueCross Shield`, `Aetna`, `UnitedHealth`, `Cigna`, `Humana`, `Medicare`, `Medicaid`, `Kaiser Permanente`, `Anthem`, `Centene`) are inserted as part of the migration's `Up()` method using raw SQL `INSERT` statements.

2. **Given** the seed migration runs, **When** a query executes `SELECT COUNT(*) FROM insurance_records`, **Then** the result is exactly 10 (or the defined seed count); re-running `dotnet ef database update` on a database where the migration was already applied does NOT duplicate the rows (idempotency enforced via `INSERT ... ON CONFLICT DO NOTHING` in the `Up()` body).

3. **Given** DR-014 requires migrations to execute automatically on startup, **When** `await context.Database.MigrateAsync()` is called in `Program.cs` before `app.Run()`, **Then** all pending migrations (including `SeedInsuranceDummyRecords`) are applied in sequence; the `__EFMigrationsHistory` table records each migration name and applied timestamp; an integration test asserts this table contains all expected migration names with non-null applied timestamps.

4. **Given** the full set of EP-DATA-I migrations has been applied, **When** an integration test calls `context.Database.GetPendingMigrationsAsync()`, **Then** the result is an empty collection, confirming zero pending migrations in the test database; this assertion forms a regression guard that fails CI if a migration is added without being applied.

5. **Given** the `insurance_records` table is populated with seed data, **When** a service performs a soft insurance validation by querying `WHERE provider_name ILIKE '%BlueCross%' OR insurance_id_prefix = 'BC'`, **Then** at least one matching record is returned, confirming the seed data structure supports the lookup pattern required by FR-024 (implemented in EP-003).

6. **Given** a developer runs `dotnet ef migrations script --idempotent` in CI, **When** the generated SQL script is executed against a fresh database, **Then** all EP-DATA-I tables (`users`, `appointment_slots`, `appointments`, `waitlist_entries`, `patient_intakes`, `notifications`, `insurance_records`) are created and the seed data is present; the script includes `IF NOT EXISTS` guards and is safe to re-run.

## Edge Cases

- What happens when the seed migration's `Down()` method is called (rollback)? → The `Down()` method deletes the 10 seed rows via `DELETE FROM insurance_records WHERE insurance_id_prefix IN (...)` and then drops the `insurance_records` table; rolling back further removes the table entirely; this is safe because no other entity has a FK to `insurance_records`.
- What happens when `context.Database.MigrateAsync()` is called concurrently by multiple API instances starting at the same time (e.g., rolling deployment)? → EF Core's `MigrateAsync()` acquires a PostgreSQL advisory lock (`pg_try_advisory_lock`) during migration; concurrent instances block on the lock and apply the migration only once; subsequent callers find zero pending migrations after the lock is released.
- What happens when an environment variable for the database connection string is incorrect at startup? → The startup guard added in us_004 throws `InvalidOperationException` before `MigrateAsync()` is called; no partial migration state is created; the error is logged to Serilog with the connection string variable name (not value) for diagnosis.
