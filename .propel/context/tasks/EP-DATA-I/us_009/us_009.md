# User Story - US_009

## Story ID

- ID Format: US_009

## Story Title

- `waitlist_entries` & `notifications` Entities

## Description

- As a developer, I want the `waitlist_entries` and `notifications` EF Core entities and migrations defined with FIFO queue positioning, channel and status enums, and retry tracking fields, so that waitlist management and multi-channel reminder delivery have a reliable, query-efficient persistence foundation from the first sprint.

## Acceptance Criteria

1. **Given** the EF Core migration for `waitlist_entries` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `entry_id` (UUID PK), `patient_id` (UUID, FK ā†’ `users.user_id`, NOT NULL), `slot_id` (UUID, FK ā†’ `appointment_slots.slot_id`, NOT NULL), `position` (INTEGER, NOT NULL, CHECK > 0), `requested_at` (TIMESTAMPTZ, NOT NULL), `status` (TEXT CHECK IN ('Waiting','Fulfilled','Cancelled'), NOT NULL, default `'Waiting'`); a composite UNIQUE constraint on `(patient_id, slot_id)` with a partial WHERE clause (`WHERE status = 'Waiting'`) prevents a patient from occupying more than one active waitlist position per slot.

2. **Given** the EF Core migration for `notifications` runs, **When** the migration completes, **Then** the table has columns: `notification_id` (UUID PK), `appointment_id` (UUID, FK ā†’ `appointments.appointment_id`, NOT NULL), `channel` (TEXT CHECK IN ('SMS','Email'), NOT NULL), `status` (TEXT CHECK IN ('Pending','Sent','Failed'), NOT NULL, default `'Pending'`), `attempt_count` (INTEGER, NOT NULL, default `0`, CHECK >= 0), `scheduled_at` (TIMESTAMPTZ, NOT NULL), `sent_at` (TIMESTAMPTZ, nullable), `error_message` (TEXT, nullable).

3. **Given** a waitlist query needs to identify the next patient in queue for a specific slot, **When** a LINQ query executes `OrderBy(e => e.RequestedAt).Where(e => e.SlotId == targetSlotId && e.Status == WaitlistStatus.Waiting).FirstOrDefaultAsync()`, **Then** the query returns the entry with the earliest `requested_at` timestamp; an index on `(slot_id, status, requested_at)` exists in the migration and makes this query plan efficient (confirmed by `EXPLAIN ANALYZE` output in the integration test showing Index Scan, not Seq Scan).

4. **Given** a notification delivery attempt fails, **When** the Notifications module increments `attempt_count` and sets `status = 'Failed'` and populates `error_message`, **Then** `SaveChangesAsync()` persists the update; a subsequent query filtering `status = 'Failed' AND attempt_count < 2` correctly surfaces the notification for retry, satisfying the at-least-one-retry requirement of NFR-012.

5. **Given** a waitlist entry is fulfilled (patient's slot opens), **When** `entry.Status = WaitlistStatus.Fulfilled` is saved, **Then** the row is retained in the database (no delete); the `position` value remains unchanged for historical audit; a query filtering `status = 'Waiting'` no longer returns that entry.

6. **Given** `sent_at` is nullable on `notifications`, **When** a notification is first created with `status = 'Pending'`, **Then** `sent_at` persists as `NULL`; once delivery succeeds and the service sets `sent_at = DateTimeOffset.UtcNow`, the value is persisted correctly and a non-null `sent_at` can be used as the delivery confirmation timestamp.

## Edge Cases

- What happens when a patient is already on the waitlist for a slot and attempts to join again? ā†’ The partial UNIQUE constraint on `(patient_id, slot_id) WHERE status = 'Waiting'` prevents the duplicate active entry; EF Core raises a `DbUpdateException`; the service layer returns a domain `AlreadyOnWaitlistException` to the caller.
- What happens when a notification row's `appointment_id` FK references a cancelled appointment? ā†’ The FK constraint on `notifications.appointment_id` is set to `RESTRICT` (not CASCADE); the appointment row remains in the database when cancelled (status change only); the notification row is preserved for delivery history and audit purposes.
- What happens when `position` values in the waitlist become non-contiguous after a cancellation (e.g., positions 1, 3, 5)? ā†’ The FIFO ordering relies on `requested_at` as the source of truth, not `position`; `position` is a display-friendly integer assigned at insertion time and is not recalculated on cancellations; queries always use `OrderBy(requested_at)` for correct queue ordering.
