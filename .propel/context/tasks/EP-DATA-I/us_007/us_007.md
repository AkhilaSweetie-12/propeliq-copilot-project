# User Story - US_007

## Story ID

- ID Format: US_007

## Story Title

- `users` Entity with PHI Column-Level Encryption

## Description

- As a developer, I want the `users` EF Core entity and corresponding migration defined with UUID primary key, AES-256 encrypted email, BCrypt password hash, role enum, and soft-delete status enum, so that all platform actors (Patient, Staff, Admin) can be persisted with HIPAA-compliant PHI protection from the first migration.

## Acceptance Criteria

1. **Given** the EF Core migration for the `users` table runs against a clean PostgreSQL database, **When** `dotnet ef database update` completes, **Then** the `users` table exists with columns: `user_id` (UUID, PK, default `gen_random_uuid()`), `email` (TEXT, unique, NOT NULL), `password_hash` (TEXT, NOT NULL), `role` (TEXT CHECK IN ('Patient','Staff','Admin'), NOT NULL), `status` (TEXT CHECK IN ('Active','Inactive'), NOT NULL, default `'Active'`), `created_at` (TIMESTAMPTZ, NOT NULL), `updated_at` (TIMESTAMPTZ, NOT NULL).

2. **Given** a `User` entity is created with a patient email address, **When** EF Core's `SaveChangesAsync()` is called, **Then** the `email` value written to the database is AES-256 encrypted (pgcrypto `pgp_sym_encrypt` or equivalent); a direct `SELECT email FROM users` query returns ciphertext, and only the application layer — using the stored encryption key — can decrypt it to the original email string.

3. **Given** the `users` EF Core entity class, **When** a query is executed via the `ApplicationDbContext`, **Then** the `email` column value is transparently decrypted on read through a value converter registered in `OnModelCreating`, so the application layer always receives the plaintext email without manual decryption calls at the service layer.

4. **Given** the `role` column is defined as an enum (Patient | Staff | Admin), **When** any code path attempts to insert a row with a `role` value outside this set (e.g., `"SuperAdmin"`), **Then** the PostgreSQL CHECK constraint rejects the INSERT and the EF Core layer raises a `DbUpdateException`; no partial row is written.

5. **Given** a user account must be deactivated (soft-delete), **When** a service calls `user.Status = UserStatus.Inactive` and `SaveChangesAsync()`, **Then** the `status` column is updated to `'Inactive'` and the row is retained in the database with all historical data intact; no hard-delete operation is exposed on the repository interface.

6. **Given** the `updated_at` column must stay current, **When** `SaveChangesAsync()` is called and the entity has been modified, **Then** an EF Core `SaveChanges` interceptor automatically sets `updated_at` to `DateTimeOffset.UtcNow` before the UPDATE is issued; no service-layer caller is required to set this field manually.

## Edge Cases

- What happens when two concurrent inserts attempt to register the same encrypted email? → The `UNIQUE` constraint on the `email` column enforces uniqueness at the database level; EF Core raises a `DbUpdateException` with the message identifying the unique constraint violation; the application layer catches this and returns a domain-level `EmailAlreadyRegisteredException` without leaking database internals.
- What happens when the AES encryption key is rotated? → The encryption key is sourced from the `PHI_ENCRYPTION_KEY` environment variable; key rotation requires a data migration script that re-encrypts all `email` values with the new key; this migration is versioned and tracked in EF Core migration history.
- What happens if the `email` column is queried in a `WHERE` clause (e.g., login lookup)? → Because the email is stored encrypted, lookups by email use the application-side encryption to produce the ciphertext of the target email, then match against the stored ciphertext; a `UNIQUE` index on the encrypted column ensures the lookup remains efficient.
