# User Story - US_010

## Story ID

- ID Format: US_010

## Story Title

- `patient_intakes` Entity with Encrypted JSONB PHI Fields

## Description

- As a developer, I want the `patient_intakes` EF Core entity and migration defined with UUID primary key, a patient FK, intake method enum, AES-256 encrypted JSONB columns for all PHI fields (demographics, medical history, medications, allergies), and an encrypted chief complaint text field, so that patient intake data is persisted and retrieved with full PHI protection at the column level from day one.

## Acceptance Criteria

1. **Given** the EF Core migration for `patient_intakes` runs, **When** `dotnet ef database update` completes, **Then** the table has columns: `intake_id` (UUID PK, default `gen_random_uuid()`), `patient_id` (UUID, FK ‚Üí `users.user_id`, NOT NULL), `intake_method` (TEXT CHECK IN ('AI','Manual'), NOT NULL), `demographics` (TEXT, NOT NULL ‚Äî stores encrypted JSONB payload), `medical_history` (TEXT, NOT NULL ‚Äî stores encrypted JSONB payload), `medications` (TEXT, NOT NULL ‚Äî stores encrypted JSONB payload), `allergies` (TEXT, NOT NULL ‚Äî stores encrypted JSONB payload), `chief_complaint` (TEXT, NOT NULL ‚Äî stores encrypted text), `submitted_at` (TIMESTAMPTZ, NOT NULL), `updated_at` (TIMESTAMPTZ, NOT NULL).

2. **Given** a `PatientIntake` entity is created with demographics, medical_history, medications, allergies, and chief_complaint values, **When** `SaveChangesAsync()` is called, **Then** each PHI column is individually encrypted with AES-256 before being written to PostgreSQL; a direct `SELECT demographics FROM patient_intakes` query returns ciphertext for each column; the application layer decrypts each column transparently via EF Core value converters registered in `OnModelCreating`.

3. **Given** an authenticated patient updates a previously submitted intake entry, **When** the service calls `SaveChangesAsync()` on the modified `PatientIntake` entity, **Then** the updated JSONB payload is re-encrypted before write; the `updated_at` timestamp is refreshed by the `SaveChanges` interceptor; the `submitted_at` timestamp remains unchanged, preserving the original submission time.

4. **Given** the `intake_method` column, **When** an intake is saved with `intake_method = 'AI'`, **Then** the value is persisted as `'AI'`; when saved with `intake_method = 'Manual'`, it persists as `'Manual'`; any other value causes a CHECK constraint violation and a `DbUpdateException` before the row is written.

5. **Given** a patient has multiple intake records (e.g., initial submission plus an edit), **When** the service queries `patient_intakes` for the most recent intake for a patient, **Then** an index on `(patient_id, submitted_at DESC)` allows efficient retrieval of the latest record without a full table scan (confirmed by `EXPLAIN ANALYZE` returning Index Scan in the integration test).

6. **Given** the JSONB PHI fields are stored encrypted as TEXT (not native PostgreSQL JSONB type), **When** a developer queries the column directly via `psql`, **Then** the raw value is unreadable ciphertext; when the application layer deserialises the decrypted TEXT into a C# DTO, the JSON structure is valid and all nested fields (e.g., `demographics.firstName`, `medications[0].name`) are accessible without error.

## Edge Cases

- What happens when the encrypted JSONB payload for `medications` exceeds the maximum TEXT column size? ‚Üí PostgreSQL TEXT columns have no practical size limit (up to 1 GB); however, the service layer enforces a maximum payload size of 64 KB per JSONB field before encryption; payloads exceeding this threshold are rejected with a `PayloadTooLargeException` at the service boundary, not the database layer.
- What happens when the PHI_ENCRYPTION_KEY environment variable is rotated mid-session? ‚Üí The value converter reads the key from the environment at application startup; mid-session key changes require an application restart and a re-encryption migration; no runtime key-switching is supported in Phase 1 to avoid partial encryption state.
- What happens when a `PatientIntake` row is queried for a `patient_id` that does not exist in the `users` table? ‚Üí The FK constraint on `patient_intakes.patient_id` is set to `RESTRICT`; an INSERT or UPDATE referencing a non-existent patient raises a `DbUpdateException`; the repository layer surfaces this as a `PatientNotFoundException`.
